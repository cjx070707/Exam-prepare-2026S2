## Part 0 — Data Engineering 概览（W1）

> [!info] 问题链
> 数据工程是什么？→ 把原始数据转化为高质量信息供下游使用 → 工程师有不同分工类型 → 整个过程分五个阶段：Lifecycle

### Data Engineering 定义 `★★★`

> [!tip] "Data Engineering" 和 "Data Science" 有什么区别？

**Data Engineering**：将原始数据转化为高质量信息，供分析师和 ML 系统使用。
（Reis & Housley 2022）

**Data Science Hierarchy of Needs**（金字塔，从底到顶）：
```
AI / Deep Learning
Learn & Optimize
Aggregate & Label
Explore / Transform
Move / Store
Collect        ← Data Engineering 在底部，是所有上层的基础
```

<span style="color: #9b59b6">💡 **白话理解**：没有数据工程，Data Science 的高楼无处落脚——数据工程师负责的是"铺管道、建仓库、保质量"，而不是做模型。</span>

<span style="color: #e67e22">*数据工程有完整的生命周期，覆盖从数据产生到最终服务的每个阶段。*</span>

### Data Engineering Lifecycle `★★★`

> [!tip] "将数据从源系统取出并分析展示"——这个过程中，数据经过哪些阶段？

```
Generation → Acquisition/Ingestion → Transformation → Serving
                        ↑↓                                ↑↓
                    Storage（贯穿全程）
```

**底层 Undercurrents（横跨所有阶段）**：Security / DataOps / Data Architecture / Orchestration / Software Engineering

| 阶段 | 职责 |
|------|------|
| **Generation** | 源系统产生数据（DB、日志、IoT） |
| **Acquisition / Ingestion** | 把数据拉进来（Batch / Streaming，Push / Pull）|
| **Transformation** | 清洗、格式转换、聚合 |
| **Serving** | 提供给分析师、ML、仪表盘使用 |
| **Storage** | 贯穿所有阶段，支撑读写 |

<span style="color: #9b59b6">💡 **白话理解**：Lifecycle 是"数据的完整旅程"——产生 → 运输 → 加工 → 上架 → 存储。Undercurrents 是整条流水线的"基础设施"：安全、自动化、质量监控。</span>

<span style="color: #e67e22">*数据工程师分两种类型——知道自己属于哪类，有助于理解不同场景下的工具选择。*</span>

### Type A vs Type B Data Engineer `★★`

> [!tip] 一家公司买了现成的 Snowflake，另一家自研分布式系统——两种工程师有什么不同？

| | **Type A（Abstraction）** | **Type B（Build）** |

<span style="color: #9b59b6">💡 **白话理解**：Type A 是"用乐高积木搭建"，Type B 是"自己设计并制造积木"。大部分数据工程师是 Type A——工具够用就不需要自研。</span>

## Part 1 — 数据从哪来（W1 + W2）

> [!info] 问题链
> 数据在哪里？→ 文件 / 数据库 / 网页 / 流 → 不同来源用不同方式获取 → 数据到手了，大小有没有边界？→ 无界流数据必须切成有界批次才能处理 → 数据怎么"推"或"拉"过来？

### 数据源分类 `★★`

> [!tip] 一个零售公司既有历史 CSV 订单、又有实时传感器数据——从哪取，用什么格式？

| 来源 | 例子 | 常见格式 |
|------|------|----------|
| **File Access** | 企业已有数据集，data.gov.au 下载 | CSV, Excel, XML, JSON, 图片 |
| **Database Access** | 生产系统数据库 | SQL 查询结果 |
| **Web / APIs** | 网页爬取，Web Service API | HTML, JSON, XML |
| **CDC / Logs** | 数据库变更日志，消息队列 | 流数据 |
| **Messages / Streams** | IoT 传感器，Pub/Sub 系统 | 实时流 |

<span style="color: #9b59b6">💡 **白话理解**：数据来源决定获取手段——静态文件直接下载，数据库用 SQL 查，网页用爬虫或 API，流数据要用消息系统（Kafka/MQTT）。</span>

<span style="color: #e67e22">*获取了数据，首先要问：这批数据是有终点的文件，还是永不停歇的流？——这决定后续怎么处理它。*</span>

### Bounded vs Unbounded Data `★★★`

> [!tip] 传感器每秒发一条温度读数，永远不会停——这种数据怎么处理？

| 类型 | 含义 | 例子 |
|------|------|------|
| **Bounded（有界）** | 有固定大小的数据集或批次 | 下载的文件，API 一次返回的结果 |
| **Unbounded（无界）** | 持续产生、没有终点的数据 | 传感器读数流，日志流 |

> 所有 Unbounded 数据在处理前必须转换成 Bounded（划定窗口或批次）

<span style="color: #9b59b6">💡 **白话理解**：Bounded 是"一袋苹果"（有头有尾）；Unbounded 是"自来水管"（不停流）。批处理系统只能处理袋装苹果——流数据必须先切成一段一段（窗口）才能批量计算。</span>

<span style="color: #e67e22">*知道数据在哪还不够——怎么"取"数据？有三种模式，决定了谁主动、谁等待。*</span>

### Ingestion 模式：Push vs Pull vs Poll `★★★`

> [!tip] Kafka 把数据推给消费者，还是消费者主动来拉？网络爬虫属于哪种模式？

| 模式 | 方向 | 描述 | 例子 |
|------|------|------|------|
| **Push** | 源 → 系统 | 源系统主动发送数据给接收方 | Kafka Producer、IoT 设备上报 |
| **Pull** | 系统 → 源 | 接收方主动向源系统请求数据 | API 调用、数据库查询 |
| **Poll** | 系统定期询问源 | 定时检查源系统是否有新数据 | Web 爬虫、定时 ETL 任务 |

> 🔑 **Poll 是 Pull 的周期性变体**：接收方不是一次性拉取，而是**定期轮询**是否有新数据。

<span style="color: #9b59b6">💡 **白话理解**：Push = "快递主动送上门"；Pull = "你去超市取货"；Poll = "你每小时去快递柜查一次有没有新包裹"。Kafka 是 Push；REST API 调用是 Pull；爬虫定时抓是 Poll。</span>

## Part 2 — Unix 命令行工具（W1） `★`

> [!info] 问题链
> 数据到手后，最快的初步探查方式是什么？→ Unix 命令行——一行命令就能看头几行、统计行数、过滤特定行

> **注意**：老师明确说不考 Unix 命令代码，以下为背景了解，重点在理解每个命令的作用。

> [!tip] 拿到一个 100 万行的 CSV，怎么快速看看前 5 行和总行数？

| 命令 | 用途 | 例子 |
|------|------|------|
| `cat` | 输出文件内容 | `cat data.csv` |
| `head` / `tail` | 看前/后 N 行 | `head -5 data.csv` |
| `sort` | 排序 | `sort data.csv` |
| `grep` | 过滤匹配行 | `grep "Sydney" data.csv` |
| `wc` | 统计行/词/字符数 | `wc -l data.csv` |
| `cut` | 按列切分 | `cut -d',' -f1,3 data.csv` |
| `sed` | 流式文本替换 | `sed -e 's/hello/world/g' file` |

**Piping（管道）**：把一个命令的输出直接作为下一个命令的输入

```bash
cat data.csv | sort | head -3
```

<span style="color: #9b59b6">💡 **白话理解**：Unix 中"一切皆文件"——命令输出是流，可以直接通过 `|` 传递，无需中间存储。像流水线：一个命令的输出直接喂给下一个命令。</span>

## Part 3 — 数据质量（W2）

> [!info] 问题链
> 数据拉进来了——但它干净吗？→ 先评估质量六维度 → 发现哪类问题 → 选对应清洗策略

### 数据质量六维度 `★★★`

> [!tip] 数据显示某产品上周销量 10 倍暴增，但你知道那周仓库关门了——这是哪类质量问题？

| 维度 | 含义 | 例子 |
|------|------|------|
| **Believable** | 数据是否可信 | 非高峰期销售数据突然暴增 |
| **Value-Added** | 数据是否有用 | 与转化率无关的流量数据 |
| **Relevant** | 数据是否相关 | 把已停产品数据混入当前分析 |
| **Accurate** | 数据是否准确 | 姓名地址不匹配 |
| **Interpretable** | 数据是否可理解 | 报告中充满未定义的缩写 |

<span style="color: #9b59b6">💡 **白话理解**：数据质量不只是"有没有缺失值"——一条准确但不相关的数据，对分析来说同样是噪声。六维度帮你全面评估数据是否"值得用"。</span>

<span style="color: #e67e22">*评估完质量，下一步是识别具体问题类型，对症下药处理。*</span>

### 常见数据问题类型 `★★★`

> [!tip] 年龄字段里有 0、有 999、有空值——三种情况分别是哪类问题，怎么处理？

| 问题 | 原因 | 处理方式 |
|------|------|----------|
| **Missing（缺失）** | 未填写、传输丢失 | `dropna()` 删除，或 `fillna()` 填充 |
| **Default（占位符）** | 用 0 / 99999 代替缺失 | 识别后替换为 NaN |
| **Incorrect（错误值）** | 传感器故障，输入错误 | 用领域知识检测，删除或修正 |
| **Inconsistent（不一致）** | 格式不统一，大小写混用 | 标准化，如统一小写 |

<span style="color: #9b59b6">💡 **白话理解**：Missing 是"空格子"；Default 是"占位符假装有值"（比如用 99999 表示未知年龄）；Incorrect 是"填了但填错了"；Inconsistent 是"格式乱了"（如 "sydney" / "Sydney" / "SYDNEY" 同一城市三种写法）。</span>

### 缺失值处理 `★★★`

> [!tip] `dropna` 和 `fillna` 分别在什么情况下选？

<span style="color: #9b59b6">💡 **白话理解**：`dropna` = 删掉整行（缺失太多时用）；`fillna` = 用某个值填坑（缺失少、能估计时用）；`replace` = 把占位符（如 99999）换成 NaN，再后续处理。</span>

## Part 5 — Web 爬虫与 API（W4）

> [!info] 问题链
> 数据在网页上——先检查是否允许爬取（robots.txt）→ 理解静态 vs 动态页面 → 用 HTTP 取回 HTML → 解析 HTML（CSS Selector / XPath）→ 提取数据 → 或直接调用 API（REST vs SOAP）

### Web Scraping 五步流程 `★★★`

> [!tip] 在开始写爬虫之前，第一步应该做什么？

| 步骤 | 操作 | 关键点 |
|------|------|--------|
| **1. Reconnaissance（侦察）** | 检查 terms-of-service 和 robots.txt | ⚠️ **先检查是否允许爬取** |
| **2. Webpage Retrieval（取回）** | 发 HTTP 请求，取回 HTML | `requests.get(url)` |
| **3. Data Extraction（提取）** | 解析 HTML，定位目标数据 | CSS Selector / XPath |
| **4. Data Cleaning（清洗）** | 去除 HTML 标签，标准化格式 | |
| **5. Data Storage（存储）** | 写入 CSV / DB / JSON | |

**robots.txt / 爬虫礼仪**：
- `robots.txt` 文件声明哪些路径**禁止爬取**（`Disallow: /private/`）
- 违反 terms-of-service 可能触发法律风险
- "Being a good net citizen"：控制爬取频率，不要让服务器过载

<span style="color: #9b59b6">💡 **白话理解**：Reconnaissance 是"开工前看施工许可证"——先查 robots.txt 和服务条款，再动手。跳过这一步，爬到一半被封 IP 甚至吃官司。</span>

<span style="color: #e67e22">*知道允许爬取后，还要分清目标是静态页面还是动态页面——决定用什么工具取回内容。*</span>

### 静态页面 vs 动态页面 `★★`

> [!tip] `requests.get()` 拿到的页面没有数据，但浏览器里看得到——是什么原因？

| | **静态页面** | **动态页面** |
|-|------------|------------|
| **内容来源** | Web Server 直接返回完整 HTML | JavaScript 在浏览器里运行后生成 |
| **requests 能获取？** | ✅ 可以 | ❌ 取到的是执行 JS 前的空壳 |
| **解决方案** | BeautifulSoup | Selenium（真正运行浏览器执行 JS）|

<span style="color: #9b59b6">💡 **白话理解**：静态页面 = "服务器发来的 HTML 就是你看到的全部"；动态页面 = "HTML 是空盒子，JS 代码在你浏览器里填数据"。requests 只能拿盒子，拿不到里面 JS 填进去的内容。</span>

<span style="color: #e67e22">*拿到 HTML 之后，要在里面定位数据——四种内容提取方式，CSS Selector 和 XPath 最重要。*</span>

### HTML 基础 `★★★`

> [!tip] 网页上的表格数据，怎么告诉爬虫"我要这个格子"而不是旁边那个？

- HTML 是**树形结构**（DOM 树）
- 标签有类型（`<div>`, `<table>`, `<h1>`）、class 属性、id 属性
- Web 爬虫的核心：找到正确的 HTML 元素提取数据

<span style="color: #9b59b6">💡 **白话理解**：HTML 像一棵家族树——`<html>` 是祖先，`<body>` 是父节点，`<table>` 是子节点，`<td>` 是孙节点。CSS Selector 就是家族地址："找到 class='data' 的 table 下第二个 tr 里的第一个 td"。</span>

<span style="color: #e67e22">*能读懂 HTML 结构之后，下一步是用 HTTP 发请求拿到网页内容。*</span>

### HTTP 请求 `★★`

> [!tip] `GET` 和 `POST` 的区别是什么？什么时候用哪个？

- **GET**：从服务器取数据，参数拼在 URL 里
- **POST**：向服务器提交数据（如登录），参数在请求体里
- 状态码：`200` = 成功，`404` = 找不到，`403` = 禁止访问

<span style="color: #e67e22">*拿到 HTML 内容后，需要解析它——找到目标数据所在的标签。*</span>

### BeautifulSoup 解析 HTML `★★`

> [!tip] 一个页面有 5 个 `<table>`，怎么只取第二个里面的数据？

```python
from bs4 import BeautifulSoup

content = BeautifulSoup(response.text, 'html5lib')

content.find('h3')                    # 找第一个 h3
content.find_all('table')             # 找所有 table → 返回列表
content.find(id='results')            # 按 id 找（唯一）
content.find_all('table', 'data')     # 按标签 + class 找
content.select('#ship.data')          # 用 CSS selector
```

<span style="color: #9b59b6">💡 **白话理解**：`find` = "找第一个"；`find_all` = "找全部，返回列表"；`select` = "用 CSS 地址精确定位"。`find_all('table')[1]` 就是第二个表格。</span>

<span style="color: #e67e22">*BeautifulSoup 解析后，CSS Selector 和 XPath 是两种主流定位方式——各有优势。*</span>

### 四种内容提取方式 `★★★`

> [!tip] CSS Selector 和 XPath 有什么区别？什么时候用哪个？

| 方式 | 描述 | 优势 | 劣势 |
|------|------|------|------|
| **Text Patterns** | 用正则表达式匹配文本 | 简单，无需解析树 | 脆弱，页面改版即失效 |
| **DOM Navigation** | 手动沿树遍历（`.children`, `.parent`）| 精确 | 代码冗长 |
| **CSS Selectors** | 用 CSS 地址定位元素 | 简洁，浏览器原生支持 | 不支持逆向查找（父节点） |
| **XPath Expressions** | 沿 DOM 树路径导航，支持谓语过滤 | 功能最强，可找父节点 | 语法复杂 |

**XPath 核心语法**：
- `/html/body/table` — 绝对路径
- `//table` — 文档中任何位置的 table
- `//table[@class='data']` — 属性过滤
- `//tr[position()>1]` — 谓语（跳过表头行）

<span style="color: #9b59b6">💡 **白话理解**：CSS Selector 像"街道门牌号"（简洁但只能从上往下找）；XPath 像"坐标系导航"（可以向上找父节点、用条件过滤）。爬虫中 CSS Selector 够用；数据提取有复杂逻辑时用 XPath。</span>

### CSS Selector 规则 `★★★`

> [!tip] 如何用一行 CSS Selector 选到 class 为 "price" 的 span 里的文字？

| 选择器 | 含义 | 例子 |
|--------|------|------|
| `tag` | 按标签类型 | `table` |
| `tag.class` | 按标签 + class | `table.data` |
| `#id` | 按 id | `#results` |
| `parent child` | 后代选择器 | `div.container span` |
| `tag:nth-child(n)` | 第 n 个子元素 | `li:first-child` |

<span style="color: #9b59b6">💡 **白话理解**：CSS Selector 像 GPS 地址——`table.data td` 意思是"找 class='data' 的 table 里的所有 td 格子"。Quiz 2 考过 HTML 代码阅读，这是重点。</span>

<span style="color: #e67e22">*网页爬取不稳定（页面结构可能随时改）——如果数据提供方有 API，优先用 API。*</span>

### Web API `★★`

> [!tip] API 和直接爬网页相比，有什么优势？REST 和 SOAP 有什么区别？

- API 返回**结构化数据**（JSON 或 XML），不含 HTML 展示信息
- **更稳定**：网页改版不影响 API 响应格式
- **更适合自动化**：不需要解析 HTML，直接 `.json()` 获得 Python 字典

**API 三种类型**：
| 类型 | 说明 |
|------|------|
| **Public（公开）** | 无需注册，任何人可用 |
| **Official requiring registration** | 需申请 API Key 控制访问频率 |
| **Closed / B2B partner** | 仅对特定合作伙伴开放 |

**RESTful API vs SOAP（XML Web Services）**：

| | **RESTful API** | **SOAP** |
|-|-----------------|---------|
| **协议** | HTTP（GET/PUT/POST/DELETE）| 专用 XML 消息协议 |
| **状态** | **Stateless（无状态）** | 有状态 |
| **响应格式** | HTML / JSON / XML（灵活）| 仅 XML |
| **复杂度** | 简单，现代主流 | 复杂，企业遗留系统 |
| **课程定义** | "Web Service" = RESTful API | 更老的标准 |

<span style="color: #9b59b6">💡 **白话理解**：REST API 是"轻便快递"（用标准 HTTP、JSON，简单直接）；SOAP 是"重型货运"（一大包 XML 消息，规范严格但复杂）。现代 Web API 几乎都是 REST；课程中 "web service" 默认指 RESTful。</span>

### 工具对比 `★★`

> [!tip] 网站用 JavaScript 动态加载内容，requests 拿不到数据，该用什么工具？

| 工具 | 适用场景 |
|------|----------|
| `requests` + `BeautifulSoup` | 单页面/少量页面爬取 |
| `Scrapy` | 多页面自动跟链爬取（spider） |
| `Selenium` | 含 **JavaScript** 的动态网页 |
| `pd.read_html()` | 直接提取 HTML 表格（**仅限 HTML table 标签**，动态内容无效）|
| Google Sheets `ImportHTML()` | 快速一次性导入网页表格 |

<span style="color: #9b59b6">💡 **白话理解**：`requests` 只能拿到服务器返回的原始 HTML，JS 动态生成的内容它看不到；`Selenium` 会真正打开浏览器执行 JS，所以能拿到动态内容。</span>

## 高频考点

- `★★★` **Data Engineering Lifecycle**：Generation → Ingestion → Transformation → Serving；Storage 贯穿全程；Undercurrents = Security/DataOps/Architecture/Orchestration/SE
- `★★★` **Bounded vs Unbounded**：有界 vs 无界；Unbounded 必须切成窗口后才能批处理
- `★★★` **Push vs Pull vs Poll**：Push=源主动推；Pull=接收方主动取；Poll=定期轮询（爬虫是 Poll）
- `★★★` **Web Scraping 五步**：Reconnaissance（查 robots.txt）→ Retrieval → Extraction → Cleaning → Storage
- `★★★` **四种内容提取方式**：Text Patterns / DOM Navigation / CSS Selectors / XPath；XPath 最强（支持父节点查找）
- `★★★` **数据质量四类问题**：Missing / Default（占位符）/ Incorrect / Inconsistent；Inconsistent remedy = reconcile
- `★★★` **HTML DOM 结构**：树形，标签 + class + id；CSS Selector 定位元素；已在 Quiz 2 考过代码阅读
- `★★★` **CSS Selector 语法**：`tag.class` / `#id` / 后代选择器
- `★★` **Type A vs Type B**：A = 用现成工具；B = 自研底层系统
- `★★` **静态 vs 动态页面**：动态页面需 Selenium；requests 只能拿静态 HTML
- `★★` **RESTful vs SOAP**：REST = stateless + HTTP + JSON/XML；SOAP = XML 消息、复杂；课程中 web service = RESTful
- `★★` **Web API 三种类型**：Public / Registration required / Closed B2B
- `★★` **数据质量六维度**：Believable / Value-Added / Relevant / Accurate / Interpretable
- `★★` **BeautifulSoup 方法**：`find` / `find_all` / `select`（CSS 精确定位）
- `★★` **工具选型**：`Selenium` 应对 JS 动态页面；`Scrapy` 多页面跟链；`pd.read_html()` 仅限 HTML table
- `★★` **Pandas 缺失值处理**：`dropna` / `fillna` / `replace` / `astype`（转 int 前先处理 NaN）
