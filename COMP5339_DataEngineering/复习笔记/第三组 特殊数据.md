## Part 1 — 空间数据（W6）

### 为什么需要专门处理 `★`

普通数据库支持点查询和范围查询，但空间查询（"找出距我 10km 内的餐厅"）需要：
- 理解地理坐标的几何含义
- 高效的空间索引
- 专用的空间操作符

### 空间数据类型 `★★`

| 类型 | 含义 | 例子 |
|------|------|------|
| **Point** | 单个位置坐标 (x, y) | GPS 坐标，地址 |
| **LineString** | 有序点的连线 | 道路，河流 |
| **Polygon** | 封闭区域 | 行政区划，建筑轮廓 |
| **Geometry** | 以上类型的统称 | — |

### 坐标系（SRID） `★★`

- **SRID（Spatial Reference ID）**：定义坐标的参考系
- **WGS84（SRID 4326）**：GPS 使用的标准经纬度坐标系
- 进行空间运算时，所有数据必须使用**相同的 SRID**，否则结果错误

### 空间查询类型 `★★★`

| 查询类型 | 含义 | SQL 函数（PostGIS） |
|----------|------|---------------------|
| **Distance** | 两点间距离 | `ST_Distance(a, b)` |
| **Contains** | A 是否包含 B | `ST_Contains(a, b)` |
| **Intersects** | A 和 B 是否相交 | `ST_Intersects(a, b)` |
| **Within** | A 是否在 B 内 | `ST_Within(a, b)` |
| **Spatial Join** | 按空间关系连接两张表 | `WHERE ST_Contains(region, point)` |

### 空间索引 `★★★`

- 普通 B-Tree 索引无法处理空间查询（不支持多维比较）
- **GiST（Generalised Search Tree）**：PostgreSQL 的通用索引，支持空间数据
- **R-Tree**：另一种常用空间索引，按边界矩形分层组织空间对象

```sql
CREATE INDEX spatial_idx ON Locations USING GIST (geom);
```

### 空间数据格式 `★★`

| 格式 | 特点 | 用途 |
|------|------|------|
| **GeoJSON** | JSON 格式的空间数据，易与 Web API 配合 | Web 地图，API 数据交换 |
| **KML** | XML 格式，Google Earth 使用 | 地理数据可视化 |
| **WKT** | Well-Known Text，如 `POINT(151.2 -33.8)` | 数据库存储 |

### 工作流程 `★`

```
获取含空间属性的数据（CSV with lat/lon，GeoJSON API）
→ 存入数据库（PostGIS 扩展）
→ 确保坐标系（SRID）一致
→ 建空间索引（GiST）
→ 执行空间查询（距离、包含、空间 Join）
→ 导出为 GeoJSON/KML 可视化
```

## Part 2 — 时序数据（W7）

> 详细内容见 `Quiz2_Revision_W7-W11.md` — Week 7 部分

> [!info] 问题链
> 普通数据库只存"最新状态"，历史版本永久丢失 → Valid Time 记录事实在现实中的时间区间 → Transaction Time 记录数据操作时间，不可修改（审计用）→ 删除不能用 DELETE，用逻辑删除关闭区间 → 查询分三类：当前 / 某时刻 / 全历史

### 核心概念速查

> [!tip] 员工工资更新了——怎么既记录新工资，又能查出三年前的工资是多少？

**两种时间语义**：

| 时间 | 含义 | 特点 |
|------|------|------|
| **Valid Time** | 事实在现实中成立的时间 | 可向前/向后修改 |
| **Transaction Time** | 数据存入数据库的时间 | 只能向前，支持审计回溯 |

**Bitemporal table** = 同时包含两种时间列的表

**表示"现在"**：用 `9999-12-31`（max-timestamp），PostgreSQL 用 `'infinity'`

**时态查询三类**：Current（当前）/ Sequenced（某时刻）/ Nonsequenced（任意时刻）

<span style="color: #9b59b6">💡 **白话理解**：Valid Time 是"事实在现实中的有效期"，Transaction Time 是"这条记录什么时候存进数据库的"。Bitemporal 表两者都有，可以回答"某时刻数据库里，关于现实某时间段的状态是什么"。</span>

> 📌 **新增**

<span style="color: #e67e22">*有了时间区间，删数据就不能直接 DELETE——物理删除会让历史永久消失。*</span>

### 逻辑删除（Logical Delete） `★★★`

> [!tip] 保险合同到期了，用 DELETE 删掉会有什么问题？

时态数据库**不做物理删除（DELETE）**，而是通过设置 `valid_end` 来"关闭"记录：

```sql
-- ❌ 错误做法：物理删除，历史永久丢失
DELETE FROM policies WHERE policy_id = 42;

-- ✅ 正确做法：逻辑删除，关闭区间
UPDATE policies
SET valid_end = CURRENT_DATE
WHERE policy_id = 42;
```

关闭后该记录不再出现在当前查询中，但历史数据完整保留，支持审计回溯。

<span style="color: #9b59b6">💡 **白话理解**：不用 DELETE，而是把 valid_end 设成今天——记录还在数据库里，只是"关闭"了，历史查询还能查到。</span>

> 📌 **新增**

<span style="color: #e67e22">*时态语义解决了"历史怎么存"，但更简单的情况：只追踪"当前状态从什么时候开始"——Semi-Temporal 表。*</span>

### Semi-Temporal Table vs Valid-Time Table `★★`

> [!tip] 只有一个 `since` 列的表，能回答"三年前员工在哪个城市"吗？

| | Semi-Temporal Table | Valid-Time Table |
|--|--------------------|--------------------|
| **列** | 只有 `since` | `start_date` + `end_date` |
| **语义** | 当前事实从 since 起持续到现在 | 事实在 `[start, end)` 区间成立 |
| **能回答** | "现在是什么？从什么时候起？" | "某时刻是什么？历史如何变化？" |
| **不能回答** | "之前是什么" / "何时改变" | — |
| **优点** | 简单，低存储 | 完整历史，可审计 |

> **Semi-temporal 是 valid-time 的简化版**，牺牲了完整历史换取简洁性。

<span style="color: #9b59b6">💡 **白话理解**：Semi-Temporal 只有一个 since 时间戳，只知道当前这条记录从什么时候开始，不能追溯历史。Valid-Time 有 start+end 两列，能还原任意时刻的状态。</span>

<span style="color: #e67e22">*逻辑删除保住了历史，但查询时怎么只拿到"某时刻有效"的记录？三类查询对应三种不同的 WHERE 写法。*</span>

### 时态 SQL 查询模板 `★★★`

> [!tip] 怎么用 SQL 查"2024 年 1 月 1 日这天这位员工的工资"？

三类查询对应不同的 `WHERE` 写法：

```sql
-- (a) Current Query：查当前有效记录
SELECT emp_id, salary
FROM emp_salary
WHERE valid_end = '9999-12-31';        -- 或 valid_end > CURRENT_DATE

-- (b) Sequenced Query：查某时间点的值（左闭右开区间）
SELECT salary
FROM emp_salary
WHERE emp_id = 101
  AND valid_start <= '2024-01-01'
  AND valid_end   >  '2024-01-01';     -- 右端用 >，避免边界重复计数

-- (c) Nonsequenced Query：全历史查询（忽略时态语义，当普通表用）
SELECT DISTINCT emp_id
FROM emp_salary
WHERE salary > 85000;
```

**关键细节**：Sequenced Query 用**左闭右开** `[valid_start, valid_end)` 区间，防止同一天边界被两条记录都命中。

<span style="color: #9b59b6">💡 **白话理解**：Current 查 valid_end = 9999-12-31（当前有效）；Sequenced 用左闭右开区间查某时刻；Nonsequenced 无视时间条件，当普通表查全历史。</span>

<span style="color: #e67e22">*时态查询逻辑解决了，但高频时序数据（传感器每秒一条）怎么存才高效？*</span>

### 时序存储三方案对比 `★★`

> [!tip] 传感器每秒一条读数，存一年就是 3000 万行——三种存法各有什么取舍？

| 方案 | 结构 | 优点 | 缺点 |
|------|------|------|------|
| **Point-based** | 每行一个时刻 | 标准 SQL，B-Tree 索引，所有 DBMS 支持 | 高频数据行数多 |
| **Sequence-based** | 一行存数组（违反 1NF） | 紧凑，行数少，批量分析友好 | 查询复杂，需 UNNEST，GIN 索引，不跨 DBMS |
| **Dedicated TSDB** | TimescaleDB / InfluxDB | 高性能，自动分区，内置 downsampling + retention | 额外系统维护，学习曲线 |

**Sequence-based 关键操作** `★★`：
- `UNNEST(array_col)` — 数组展开为多行（用于查询）
- `array_agg(value)` — 多行聚合为数组（用于写入）
- 索引：**GIN**（Generalised Inverted Index），不是 GiST

**时序数据五大工程挑战** `★★`：

| 挑战 | 含义 |
|------|------|
| **Missing data** | 传感器故障，数据缺失 |
| **Irregular intervals** | 事件驱动，间隔不等 |
| **Large volume** | 高频数据量极大，需 downsampling |
| **Time zones** | 跨时区时间对齐复杂 |
| **Late-arriving data** | 网络延迟，数据乱序到达 |

<span style="color: #9b59b6">💡 **白话理解**：Point-based 每时刻一行（B-Tree 最简单）；Sequence-based 一行存整个数组（紧凑但违反 1NF，UNNEST 展开，GIN 索引）；TSDB 专门为时序优化（内置 downsampling/retention）。</span>

## Part 3 — 非结构化数据（W8）

> 详细内容见 `Quiz2_Revision_W7-W11.md` — Week 8 部分

> [!info] 问题链
> 文本/图像无法直接用 SQL 查询 → 需要特征提取转成数值向量 → 关键词搜索用倒排索引（稀疏向量）→ 遇到词汇表外的新词（OOV）用 Subword 分词解决 → 语义相似搜索用稠密向量（BERT）+ 向量数据库 ANN

### 核心概念速查

> [!tip] 一篇文章怎么让计算机"读懂"并检索它？

**处理流程**：
```
原始数据（文本/图片）→ 预处理 → 特征提取 → 数值向量 → ML / 相似度搜索
```

**文本特征提取**：

| 方法 | 类型 | 特点 |
|------|------|------|
| **TF-IDF** | Bag of Words | 稀疏，无语义，计算简单 |
| **Word2Vec** | 词向量 | 稠密，有语义相似性 |
| **BERT** | 上下文嵌入 | 同词不同语境有不同向量（bidirectional） |

**图像类型**：RGB（3通道 24bit）/ Grayscale（单通道）/ Binary（黑白 mask）

**特征提取两种路径**：
- White box：手工设计（基于图像梯度，如 skimage）
- Black box：神经网络自动学习（CNN、LLM）

**元数据工具**：`exiftool`（读取 EXIF/XMP 元数据，支持批量、脚本化）

<span style="color: #9b59b6">💡 **白话理解**：文本/图像要先变成数字向量——TF-IDF 做稀疏词袋向量，BERT 做稠密语义向量，图像用 CNN 提取特征向量，然后才能做 ML 或相似度搜索。</span>

> 📌 **新增**

<span style="color: #e67e22">*有了向量，搜索时怎么快速找到包含某关键词的文档？全文扫描太慢——倒排索引解决这个问题。*</span>

### Inverted Index（倒排索引） `★★★`

> [!tip] 搜索"data pipeline"，为什么不用挨个扫描所有文档？

**结构**：`词 → [文档列表（含位置/频率）]`

```
"data"       → [(Doc1, pos:0), (Doc2, pos:0)]
"pipeline"   → [(Doc1, pos:1), (Doc3, pos:0)]
"science"    → [(Doc2, pos:1)]
```

**查询加速原理**：查询 "data pipeline" 时：
1. 查索引找 "data" → `{Doc1, Doc2}`
2. 查索引找 "pipeline" → `{Doc1, Doc3}`
3. 求交集 → `{Doc1}`

无需扫描任何文档内容，时间复杂度从 **O(N × 文档长度)** 降为 **O(1) 索引查找 + O(k) 集合运算**。Elasticsearch、PostgreSQL `tsvector` 均基于此原理。

<span style="color: #9b59b6">💡 **白话理解**：倒排索引就像书的"索引页"——查"data"在哪些文档，再查"pipeline"在哪些文档，取交集，不用翻任何文档内容。</span>

> 📌 **新增**

<span style="color: #e67e22">*倒排索引解决了搜索效率，但"词"本身怎么切？遇到词汇表里没有的新词（OOV）怎么办？*</span>

### Tokenization & Subword 分词（BPE / WordPiece） `★★`

> [!tip] GPT 遇到"ChatGPT"这个训练时没见过的词，怎么处理？

**词级分词的问题**：词汇表固定，训练集未见过的词（OOV）只能替换为 `[UNK]`，丢失所有信息。

**Subword Tokenization 的解决方案**：将词分割为高频子词单元，即使面对未见过的词（如新产品名、专有名词），也能通过组合已知子词理解部分含义。

| 算法 | 使用者 | 特点 |
|------|--------|------|
| **BPE**（Byte-Pair Encoding） | GPT 系列 | 合并最高频字节对 |
| **WordPiece** | BERT | `##` 前缀表示非词首部分（non-word-initial） |

**NLP 管道顺序**：`原始文本 → Tokens → Token IDs → Embeddings → Model`

<span style="color: #9b59b6">💡 **白话理解**：词级分词遇到新词直接变 [UNK]，什么都不知道。Subword 把词拆成小块（BPE 合并高频字节对，WordPiece 用 ## 标记非词首），遇到新词也能拼出大概含义。</span>

> 📌 **新增**

<span style="color: #e67e22">*分词解决了，TF-IDF 和 BERT 产生了完全不同的向量——该用哪种取决于你要做什么搜索。*</span>

### Sparse Vector vs Dense Vector `★★★`

> [!tip] 搜索"语义相似的新闻" vs 搜索"包含某个词的新闻"——哪个需要语义向量？

| | Sparse Vector（TF-IDF） | Dense Vector（BERT） |
|-|------------------------|---------------------|
| **维度** | 词汇表大小（数万维） | 固定低维（如 768 维） |
| **非零值** | 极少（大部分为 0） | 全部非零 |
| **语义** | 无（词袋模型） | 有（上下文语义） |
| **存储** | 稀疏格式（只存非零项），节省空间 | 固定 768 × float32 ≈ 3KB/向量 |
| **大规模搜索** | 倒排索引 | 向量数据库（Faiss、pgvector）做 ANN 搜索 |

**核心结论**：关键词精确匹配用 Sparse（Inverted Index + TF-IDF）；语义相似度搜索用 Dense（BERT Embedding + Cosine Similarity）。

<span style="color: #9b59b6">💡 **白话理解**：稀疏向量（TF-IDF）= 高维、大部分为零、只管词频，适合关键词精确匹配（用倒排索引）；稠密向量（BERT）= 低维全非零、有语义，适合"找相似内容"（向量数据库 ANN 搜索）。</span>
