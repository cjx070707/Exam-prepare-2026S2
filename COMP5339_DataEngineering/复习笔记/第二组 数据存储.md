## Part 1 — 关系型数据库（W3）

> [!info] 问题链
> 数据要存起来——最经典的方式是关系型数据库（表）→ 表靠主键唯一标识每行 → 表之间靠外键关联 → 但关系型数据库擅长的是事务（OLTP），分析查询（OLAP）的需求不同

### 核心概念 `★★★`

> [!tip] 一张 Enrolment 表记录了哪个学生选了哪门课——怎么保证一行记录唯一，且外键引用合法？

- **Relation（关系）**：一张表，是一组 tuple 的集合
- **Schema**：描述表的列名和类型
- **Primary Key**：唯一标识每一行（可以是单列或多列组成的 Composite PK）
- **Foreign Key**：引用另一张表的主键，建立"逻辑指针"

```sql
CREATE TABLE Enrolment (
    student_id INT NOT NULL,
    course_id  INT NOT NULL,
    enrolled_at DATE NOT NULL,
    marks INT,
    PRIMARY KEY (student_id, course_id),
    FOREIGN KEY (student_id) REFERENCES Student(student_id),
    FOREIGN KEY (course_id)  REFERENCES Course(course_id),
    CHECK (marks IS NULL OR marks BETWEEN 0 AND 100)
);
```

<span style="color: #9b59b6">💡 **白话理解**：Primary Key 是"身份证"（每行唯一）；Foreign Key 是"指针"（指向另一张表的某行）；CHECK 是"守门员"（值不合法就拒绝写入）。</span>

<span style="color: #e67e22">*关系型数据库擅长事务处理——但分析查询（聚合、大范围扫描）的需求完全不同，两者要区分对待。*</span>

### Push Compute vs Pull Data `★★`

> [!tip] 有 1 亿条记录，在 Python 里过滤 vs 让数据库过滤——哪个更快？

**Push compute into DBMS**（把计算推给数据库）vs **Pull all data**（拉到 Python 处理）：
- **推给 DBMS**：利用索引和 join 优化，只传必要数据，扩展性强
- **拉到 Python**：简单灵活，但数据必须能放进内存（小数据集才可行）

<span style="color: #9b59b6">💡 **白话理解**：让数据库做 WHERE、GROUP BY 过滤，只返回结果给 Python——比把 1 亿行全拉过来再在内存里过滤快得多。</span>

## Part 2 — OLTP vs OLAP（W3）

> [!info] 问题链
> 同一份业务数据，订单系统需要"快速写入单条"，分析团队需要"扫描全量统计"——同一个数据库能同时应付吗？→ 不能，需要分开：OLTP 处理事务，OLAP 处理分析

### OLTP vs OLAP 对比 `★★★`

> [!tip] 为什么不把分析查询直接跑在生产数据库上？

| 维度 | OLTP（在线事务处理） | OLAP（在线分析处理） |
|------|---------------------|---------------------|
| **主要读模式** | 点查询（按 key 取少量记录） | 聚合扫描（COUNT/SUM/AVG 大量记录） |
| **主要写模式** | Insert / Update / Delete 单条 | 批量导入（ETL/ELT）或事件流 |
| **用户** | Web/移动应用的终端用户 | 内部数据分析师 |
| **查询特点** | 固定预定义查询 | 分析师可以任意查询 |
| **数据时态** | 当前状态快照 | 历史事件记录 |
| **数据量** | GB 到 TB | TB 到 PB |
| **例子** | 银行转账、订单录入 | 数据仓库、商业报表 |

<span style="color: #9b59b6">💡 **白话理解**：OLTP 是"收银台"（快速处理每笔交易）；OLAP 是"财务报表"（扫描所有历史交易做统计）。两者的查询模式南辕北辙，分开系统才能都做好。</span>

<span style="color: #e67e22">*分析场景需要单独的存储系统——数据仓库（Data Warehouse）就是为 OLAP 设计的。*</span>

## Part 3 — 数据仓库（W3）

> [!info] 问题链
> 需要支持"按地区、按产品、按时间"多维度分析 → Star Schema 是数据仓库的标准设计 → 数据怎么从 OLTP 进来？→ ETL 或 ELT

### Star Schema（星型模式）`★★★`

> [!tip] 分析师要查"2024年Q1各地区各品类的总销售额"——数据库表应该怎么设计？

数据仓库的核心设计模式：

- **Fact Table（事实表）**：存储可度量的业务事件，包含各维度的外键
  - 例：`Sales(Market_Id, Product_Id, Time_Id, Sales_Amt)`
- **Dimension Tables（维度表）**：描述事实的各个维度
  - 例：`Market(Market_Id, City, State, Region)`
  - 例：`Product(Product_Id, Name, Category, Price)`
  - 例：`Time(Time_Id, Week, Month, Quarter)`

维度可以有层级结构：
```
Product:  category → pname
Location: country → state → city
Time:     year → quarter → month → week → date
```

<span style="color: #9b59b6">💡 **白话理解**：Fact Table 是中心（存"发生了什么"），Dimension Tables 是外围（存"谁、在哪、什么时候"）。画出来像一颗星——所以叫 Star Schema。</span>

<span style="color: #e67e22">*有了 Star Schema，怎么对多维数据进行分析？OLAP 提供了五种操作。*</span>

### OLAP 五种操作 `★★★`

> [!tip] "先看全国总销售额，再下钻到某省某市"——这两步分别叫什么操作？

| 操作 | 含义 | 例子 |
|------|------|------|
| **Roll Up（上卷）** | 沿维度层级**向上汇总** | 城市 → 省份 → 全国 |
| **Drill Down（下钻）** | 沿维度层级**向下展开** | 年度 → 季度 → 月份 |
| **Slice（切片）** | 固定**一个维度**的某个值，查剩余维度 | 只看 2024 年的数据 |
| **Dice（切块）** | 固定**多个维度**的范围，查子立方 | 只看 2024 Q1 + 华东地区 |
| **Pivot（旋转）** | 重新排列维度视角（行/列互换） | 把时间维度从行换到列 |

> ⚠️ **Slice vs Dice**：Slice = 锁定 1 个维度；Dice = 锁定多个维度（切出更小的"块"）

<span style="color: #9b59b6">💡 **白话理解**：把数据想成一个 3D 魔方（时间 × 地区 × 产品）：Roll Up = 把小格合并成大格；Drill Down = 展开大格；Slice = 拿掉一层；Dice = 切出某个小正方体；Pivot = 把魔方侧过来看。</span>

<span style="color: #e67e22">*分析场景中经常有多个 OLTP 系统各自存数据——造成"数据孤岛"问题，数仓正是为解决这个而生。*</span>

### Data Silo（数据孤岛）`★★`

> [!tip] 销售部门有自己的 CRM，运营部门有自己的 ERP——数据分析师怎么跨系统分析？

- **Data Silo**：多个 OLTP 系统独立运行，数据无法互通
- 导致分析师无法跨系统查询；同一指标在不同系统定义不同
- **Data Warehouse 的动机**：把分散在各 OLTP 系统的数据整合到一个中央仓库，统一 schema，支持跨系统分析

<span style="color: #9b59b6">💡 **白话理解**：Data Silo 像"每个部门都有自己的小冰箱"，食材不共享、可能重复买、无法统一做菜。数仓是"公共厨房"——统一存储、统一标准。</span>

<span style="color: #e67e22">*数据仓库设计好了，数据怎么进来？两种方式：先转换（ETL）还是先存再转（ELT）。*</span>

### ETL vs ELT `★★★`

> [!tip] 云数仓（如 BigQuery）和传统数仓在 ETL/ELT 选择上有什么不同？

| | ETL（Extract-Transform-Load） | ELT（Extract-Load-Transform） |
|-|-------------------------------|-------------------------------|
| **顺序** | 先转换再载入 | 先载入再转换 |
| **转换位置** | 数据仓库**外部** | 数据仓库/数据湖**内部** |
| **适合** | 传统数仓，数据已知结构 | 现代云数仓，原始数据先存再按需转换 |

<span style="color: #9b59b6">💡 **白话理解**：ETL = 先洗碗再上桌（进仓库前已经干净）；ELT = 先端上桌再洗（原始数据先进去，在仓库里用 SQL/dbt 转换）。云数仓计算便宜，更倾向 ELT。</span>

**ETL 的三大痛点（为什么要迁移到 ELT）**：
1. **版本控制难**：转换逻辑在 ETL 工具里，不像代码可以 git 管理
2. **BI 工具业务逻辑过重**：分析师被排除在转换流程之外，依赖工程师
3. **灵活性差**：每次需求变更都要改 ETL pipeline，再重新跑

ELT（配合 dbt）的优势：Data as Code、可版本控制、CI/CD 自动化、赋权分析师直接写 SQL

## Part 4 — 数据仓库 vs 数据湖（W3）

> [!info] 问题链
> 数仓解决了结构化分析问题——但图片、日志、JSON 这些非结构化数据放哪？→ Data Lake：先存再想用途 → 风险：容易变成"数据沼泽"

### 三种存储对比 `★★★`

> [!tip] 数仓（Snowflake）和数据湖（S3）都能存数据——什么情况选哪个？

| 维度 | Database（OLTP） | Data Warehouse（OLAP） | Data Lake |
|------|-----------------|----------------------|-----------|
| **数据类型** | 结构化 | 结构化 | 结构化 + 半结构化 + **非结构化** |
| **Schema** | schema-on-write | schema-on-write | **schema-on-read** |
| **敏捷性** | 低（固定 schema） | 低（固定 schema）| 高（任意格式直接存）|
| **处理方式** | 在线事务 | 批量分析 | 按需处理 |
| **成本** | 高（存储+计算）| 中 | **存储便宜，处理按需付费** |
| **用途** | 生产系统事务 | 历史分析、报表 | 灵活存储，未来用途不确定 |
| **技术** | PostgreSQL、MySQL | Snowflake、BigQuery | HDFS、S3 |
| **风险** | — | — | 容易变成"数据沼泽"（data swamp） |

**场景选择指南**：
- 数据结构清晰、分析需求明确 → **Data Warehouse**
- 数据类型混杂、未来用途不确定 → **Data Lake**
- 实践架构：**原始数据进 Lake，精炼后移入 Warehouse**（两者结合）

**Data Lakehouse** `★★`：Data Lake + Metadata & Governance Layer
- 结合 Lake 的灵活存储 + Warehouse 的数据治理（schema、ACID、权限）
- 代表技术：Delta Lake、Apache Iceberg

<span style="color: #9b59b6">💡 **白话理解**：数仓是"整洁的书架"（有固定结构，找东西快）；数据湖是"大仓库"（啥都往里扔，灵活但容易乱，时间一长变沼泽）；Lakehouse = 给大仓库加上分类标签和访问控制，兼得两者优势。</span>

## Part 5 — 半结构化数据与 NoSQL（W5）

> [!info] 问题链
> 不是所有数据都有固定结构——JSON/XML 嵌套、字段可缺失 → 关系型数据库强制 schema，装不下 → NoSQL：文档型（MongoDB）、图型（Neo4j）、宽列型（HBase）各有适用场景

### 半结构化数据特点 `★★`

> [!tip] JSON 文档里的字段数量不固定，有的有 "email"，有的没有——关系型数据库能直接存吗？

- 没有固定 schema（missing / 额外属性均可）
- **嵌套结构**（树形，如 JSON / XML）
- 同一集合中不同对象可有不同类型
- **自描述（Self-describing）**：数据本身携带字段名

<span style="color: #9b59b6">💡 **白话理解**：关系型数据库要求每行有完全相同的列——半结构化数据"每行格子数不一样"，强行放进去要么浪费空间（大量 NULL），要么放不进去。</span>

<span style="color: #e67e22">*JSON 和 XML 是两种最常见的半结构化格式，它们都是树形，但设计哲学不同。*</span>

### XML vs JSON `★★★`

> [!tip] 企业老系统用 XML 交换数据，现代 Web API 都用 JSON——两者能互换吗？区别是什么？

| | XML | JSON |
|-|-----|------|
| **标签** | 用户自定义标签，有属性 | key-value 对，无属性概念 |
| **结构** | 树形 | 树形 |
| **校验** | DTD / XML Schema | JSON Schema |
| **用途** | 企业数据交换，Web Service | Web API，现代数据存储 |

**XML 合规性（重点）**：
- **Well-formed**：满足 XML 语法约束（标签必须匹配、嵌套正确）
- **Valid**：Well-formed + 符合 DTD / Schema 定义的结构（字段存在、顺序正确）

**DTD 语法速记**（常考）：
```
(name, age)   → 严格顺序，两个都必须有
(name | age)  → 二选一
(name*)       → 零或多个
(name+)       → 一或多个（至少一个）
(name?)       → 可选（零或一个）
```

<span style="color: #9b59b6">💡 **白话理解**：Well-formed = "括号匹配"（语法没错）；Valid = "内容也对"（符合定义的结构规则）。一个 XML 文件可以是 Well-formed 但 Not Valid（语法 OK 但结构不符合 DTD）。</span>

<span style="color: #e67e22">*半结构化数据需要专门的数据库来高效存取——MongoDB 是最常用的文档型 NoSQL。*</span>

### MongoDB（文档型 NoSQL）`★★★`

> [!tip] MongoDB 的 `find({ age: { $gt: 20 } })` 对应 SQL 里的什么查询？

- 数据模型：**Collection**（集合）中存 **Document**（文档），即嵌套 key-value（JSON 格式）
- **Schema-late（= schema-on-read）**：不需要提前定义结构，字段可灵活变化
- 每个文档都有 **`_id` 字段**（用户定义或自动生成的 ObjectID），作为文档唯一标识

**MongoDB 术语对照**：

| RDBMS | MongoDB |
|-------|---------|
| Table | Collection |
| Row | Document |
| Column | Field |
| 1NF（归一化）| Denormalized（允许嵌套/重复）|

**关系建模：两种方式** `★★★`

| 方式 | 做法 | 适合 |
|------|------|------|
| **Embedded sub-document（嵌入）** | 把相关数据直接嵌套在文档内 | 数据一起读、关系稳定（如订单+商品列表）|
| **Reference（引用）** | 存另一个文档的 `_id`，需要时 join | 数据独立更新、共享使用（如用户+评论）|

> ⚠️ 嵌入 = 性能好但冗余；引用 = 灵活但需要额外查询

**MongoDB 分布式架构** `★★★`：
- **Auto Sharding**：按 range 自动分区，数据分散到多个 shard
- **Single-Leader Replication**：Primary 接收写，Follower 可分担读
- ⚠️ 读 Follower 时只有 **Eventual Consistency**（副本未完全同步）
- Load balancer 自动路由请求

**基本操作**：

```javascript
// 插入
db.users.insertOne({ name: "sue", age: 26, status: "pending" })

// 查询（类似 SELECT ... WHERE ...）
db.students.find()                              // SELECT *
db.students.find({ name: 'Jane' })              // WHERE name='Jane'
db.students.find({ age: { $gt: 20 } })          // WHERE age > 20
db.students.find({ $or: [{ name: 'Jane' }, { name: 'Joe' }] })

// 投影（选择返回哪些字段）
db.users.find({ age: { $gt: 26 } }, { name: 1, address: 1 })

// 排序
db.courses.find().sort({ name: -1 })            // ORDER BY name DESC
```

**引号规则**：
- MongoDB Shell（JavaScript 接口）：key 不需要引号
- mongoimport / JSON 文件：key **必须**用双引号（标准 JSON）

<span style="color: #9b59b6">💡 **白话理解**：MongoDB 的查询就是"用 JSON 描述 WHERE 条件"——`{ age: { $gt: 20 } }` 相当于 `WHERE age > 20`，`$or` 相当于 `OR`。</span>

<span style="color: #e67e22">*文档型数据库适合嵌套 JSON——但社交网络、推荐系统的数据本质是节点和关系，图数据库更合适。*</span>

### 关系型 vs 图数据库 `★★`

> [!tip] "找出 A 的朋友的朋友中有没有 B"——用 SQL 写需要多少层 JOIN？图数据库怎么做？

| 维度 | RDBMS | Graph DB（如 Neo4j） |
|------|-------|----------------------|
| **数据模型** | 表（行和列） | 节点（Node）、边（Edge）、属性 |
| **Schema** | 固定 | 灵活或无 Schema |
| **查询语言** | SQL | Cypher（Neo4j）、Gremlin、SPARQL |
| **性能优势** | ACID 事务，复杂 join | **图遍历**（多跳关系查询） |
| **扩展方式** | 垂直扩展（Scale-Up）| 水平扩展（Scale-Out）|
| **ACID 合规** | ✅ 完整 ACID | ✅ Neo4j 也支持 ACID（⚠️ 非典型 NoSQL）|
| **适用场景** | 传统业务系统，事务处理 | 社交网络，推荐系统，欺诈检测 |

**Graph DB 三大优势**：
1. **Efficient Relationship Handling**：many-to-many 关系直接存为边，多跳遍历快
2. **Flexibility**：schema-less，节点和边可随时增加新属性
3. **Performance**：图遍历（沿边走）比反复 JOIN 快得多

> ⚠️ **MCQ 陷阱**：Neo4j 是图数据库，但它**支持 ACID**——"NoSQL 不支持 ACID"是错误描述

<span style="color: #9b59b6">💡 **白话理解**：SQL 的 JOIN 是"逐层做表连接"，关系越深越慢；图数据库把"谁认识谁"直接存成边，遍历六度关系就是沿着边走——天然高效。Neo4j 的 Property Graph Model 让每个节点和边都可以带属性（不只是 ID）。</span>

## Part 6 — NoSQL 选型总结 `★★`

> [!tip] 时序传感器数据、社交网络关系、电商商品 JSON——分别用哪类 NoSQL？

**NoSQL 起源**：传统 RDBMS 专注 Scale-Up（贵且有上限）；云计算兴起后需要 Scale-Out（廉价普通服务器集群）→ NoSQL 应运而生。"NoSQL" 后来重新定义为 **"Not only SQL"**（因为很多新系统后来也加回了 join 等特性）。

| NoSQL 类型 | 代表系统 | 适合数据 | 核心优势 |
|-----------|----------|----------|----------|
| **Document Store** | MongoDB, Couchbase | 半结构化 JSON | 灵活 schema，嵌套文档 |
| **Key-Value Store** | Redis, DynamoDB, Cassandra, RocksDB | 简单键值对 | **极快速、高度可扩展**（无复杂查询）|
| **Graph DB** | Neo4j | 关系密集型数据 | 多跳图遍历 |
| **Time Series DB** | InfluxDB, TimescaleDB | 时序数据 | 时间范围查询优化 |
| **Wide Column Store** | HBase, BigTable, SimpleDB | 稀疏大数据 | 按 column family 分组，超大规模 |

> **Key-Value Store 特点**：非常简单——只支持按 key 存/取；没有复杂查询；换来的是极高的速度和扩展性。典型用途：缓存（Redis）、Session 存储。

<span style="color: #9b59b6">💡 **白话理解**：选 NoSQL 的核心逻辑：数据是嵌套 JSON → Document Store；数据是节点+关系 → Graph DB；数据带时间戳、按时间范围查 → TSDB；数据极稀疏且超大规模 → Wide Column。</span>

## 高频考点

- `★★★` **OLTP vs OLAP**：读写模式（点查询 vs 聚合扫描）、用户、数据量、查询特点四个维度
- `★★★` **Star Schema**：Fact Table（中心，可度量事件）+ Dimension Tables（外围，描述维度）
- `★★★` **OLAP 五种操作**：Roll Up（向上汇总）/ Drill Down（向下展开）/ Slice（固定1维）/ Dice（固定多维）/ Pivot（视角旋转）
- `★★★` **ETL vs ELT**：转换位置（外部 vs 内部）；ELT 优势：版本控制、赋权分析师、CI/CD
- `★★★` **schema-on-write vs schema-on-read**：数仓写入前定义 vs 数据湖读取时套模板
- `★★★` **Well-formed vs Valid XML**：语法合规 vs 符合 DTD 结构
- `★★★` **DTD 语法**：`,`（顺序）/ `|`（选一）/ `*`（零或多）/ `+`（一或多）/ `?`（可选）
- `★★★` **MongoDB 关系建模**：Embedded（嵌入，一起读）vs Reference（引用，独立更新）
- `★★★` **MongoDB 分布式**：Auto Sharding + Single-Leader Replication + Eventual Consistency（读 Follower 时）
- `★★★` **NoSQL 五类**：Document / Key-Value（极快，无复杂查询）/ Graph / TSDB / Wide Column
- `★★` **Data Lake vs Warehouse 五维**：数据类型 / Schema / 敏捷性 / 处理方式 / 成本
- `★★` **Data Lakehouse**：Lake + Metadata & Governance；代表：Delta Lake、Iceberg
- `★★` **Data Silo 与 DW 动机**：多 OLTP 系统孤岛 → 无法跨系统分析 → 数仓统一
- `★★` **ETL 三大痛点**：版本控制难 / BI 工具逻辑过重 / 分析师被排除
- `★★` **NoSQL 起源**："Not only SQL"；Scale-Up（RDBMS）→ Scale-Out（NoSQL）
- `★★` **Neo4j 特性**：Property Graph Model + **ACID 合规**（⚠️ 图数据库反而支持 ACID）
- `★★` **MongoDB 术语对照**：Table=Collection / Row=Document / Column=Field / 1NF=Denormalized
