# 第7周测试 — Temporal Data

---

**Question 1.** A fintech company maintains an employee compensation database. HR occasionally updates salary records retroactively — for example, a raise approved in March is backdated to January. The compliance team also requires a full audit trail: *"What did the system show about Alice's salary on 15 February, even if that record was later corrected?"* Which data modeling approach should the engineering team implement?

- [ ] A. A standard table with a single `updated_at` timestamp column
- [ ] B. A Valid Time table that only tracks when each salary is effective in the real world
- [x] C. A Bitemporal table with both Valid Time (business reality) and Transaction Time (audit trail)
- [ ] D. A Sequence-based table where all salary changes are stored as an array per employee row

> [!note]- Answer
> **C. Bitemporal Table（双时态表）。**
>
> 本题包含两个独立的时间维度：
> - **Valid Time（有效时间）**：薪资在现实中的生效时段，HR 可追溯修改
> - **Transaction Time（事务时间）**：数据进入数据库的时间，Append-only，用于审计回溯
>
> 只有 Bitemporal 表同时追踪两者。A 只有单个时间戳，无法区分两个维度；B 只有 Valid Time，无法回答"2 月 15 日系统里记录的是什么"；D 是存储格式，不是时态建模方案。

**Question 2.** A data engineer is querying an employee salary table with columns `emp_id, salary, valid_start, valid_end`. She notices that many rows have `valid_end = '9999-12-31'`. A junior colleague suggests these rows contain invalid dates and should be cleaned up. What does `valid_end = '9999-12-31'` actually represent?

- [ ] A. The record has expired and should be archived
- [x] B. The record is currently active — it represents an open-ended interval ("still valid as of today")
- [ ] C. The record was entered in error and requires manual correction
- [ ] D. The database auto-filled the field after reaching its maximum date limit

> [!note]- Answer
> **B. Open-ended Interval（开放区间）惯例。**
>
> `9999-12-31`（或 `infinity`）是时态数据库中表示"至今仍有效"的标准惯例——该记录的结束时间尚未确定，即当前记录处于活跃状态。查询当前有效记录的标准写法：
> ```sql
> WHERE valid_end = '9999-12-31'
> -- 或
> WHERE valid_end > CURRENT_DATE
> ```
> 这些行是最新的业务数据，绝不应被删除或"修复"。

**Question 3.** A healthcare system stores patient insurance policy records with `valid_start` and `valid_end` columns. Policy 42 has expired and the team needs to mark it as inactive. A developer proposes running `DELETE FROM policies WHERE policy_id = 42`. A senior engineer immediately objects. What is the correct approach in a temporal database?

- [ ] A. Run the DELETE — expired records waste storage space
- [ ] B. Set all business columns to NULL to indicate the record is no longer active
- [x] C. Set `valid_end = CURRENT_DATE` to close the interval, preserving the full history
- [ ] D. Copy the row to an archive table, then DELETE from the main table

> [!note]- Answer
> **C. Logical Delete（逻辑删除）。**
>
> 时态数据库的核心原则是**不物理删除历史记录**，以支持完整的历史查询和审计。正确做法是将有效区间"闭合"：
> ```sql
> UPDATE policies
> SET valid_end = CURRENT_DATE
> WHERE policy_id = 42;
> ```
> 这使该记录不再出现在当前查询中（`valid_end` 不再是 `9999-12-31`），但历史数据完整保留。D 的归档方案破坏了时态查询的统一性；A 和 B 均导致历史数据永久丢失。

**Question 4.** A hospital IoT system monitors patient vital signs from 200 bedside devices, each reporting every 2 seconds — generating approximately 8.6 million rows per day. The data team is debating storage strategy. Which approach is most appropriate?

- [ ] A. Point-based storage in a standard PostgreSQL table with a B-Tree index on timestamp
- [ ] B. Sequence-based storage, compressing each patient's daily readings into a single array row
- [x] C. A Dedicated Time-Series Database such as TimescaleDB or InfluxDB
- [ ] D. A document store such as MongoDB with one document per device per day

> [!note]- Answer
> **C. Dedicated TSDB（专用时序数据库）。**
>
> 三种方案对比：
> - **A. Point-based**：标准关系表每天 860 万行，B-Tree 索引随数据量膨胀，高频写入吞吐成为瓶颈
> - **B. Sequence-based**：实时流式写入困难（需先缓冲再批量更新数组），查询逻辑非标准 SQL，可移植性差
> - **C. Dedicated TSDB**：专为高频时序写入设计——列式压缩（时间戳 delta 编码，压缩率极高）、高并发写入、内置时间函数（`time_bucket`、降采样、移动平均）；是 IoT / 监控场景的首选方案
> - **D. Document Store**：MongoDB 文档模型适合半结构化嵌套数据，不针对时序优化——无时间压缩、无内置时间聚合函数；每日一文档的方案会导致文档体积随读数累积膨胀，范围查询和聚合效率远低于专用 TSDB

**Scenario 1:** A retail company tracks employee salaries in a temporal table `emp_salary(emp_id, salary, valid_start, valid_end)`. The current data is:

| emp_id | salary | valid_start | valid_end  |
|--------|--------|-------------|------------|
| 101    | 80,000 | 2023-01-01  | 2024-06-30 |
| 101    | 90,000 | 2024-07-01  | 9999-12-31 |
| 102    | 75,000 | 2023-01-01  | 9999-12-31 |
| 103    | 68,000 | 2022-06-01  | 2023-05-31 |

**Question 5:** Write SQL queries for the following three temporal query types. **(6 Marks)**

**(a) Current Query（当前查询）:** Return the `emp_id` and `salary` of all employees with a currently active salary record. **(2 Marks)**

**(b) Sequenced Query（时序查询）:** Return employee 101's salary as it stood on `2024-01-01`. **(2 Marks)**

**(c) Nonsequenced Query（非时序查询）:** Return the `emp_id` of any employee who has ever earned more than $85,000, regardless of time period. **(2 Marks)**

> [!note]- Answer
> **(a) Current Query — 查询当前有效记录**
> ```sql
> SELECT emp_id, salary
> FROM emp_salary
> WHERE valid_end = '9999-12-31';
> -- 或: WHERE valid_end > CURRENT_DATE
> -- 结果：emp_id 101 (90,000)，emp_id 102 (75,000)
> ```
>
> **(b) Sequenced Query — 时间点查询（2024-01-01）**
> ```sql
> SELECT salary
> FROM emp_salary
> WHERE emp_id = 101
>   AND valid_start <= '2024-01-01'
>   AND valid_end > '2024-01-01';
> -- 结果：80,000（该日期落在 2023-01-01 至 2024-06-30 的左闭右开区间内）
> ```
> 注意：区间条件为左闭右开 `[valid_start, valid_end)`，避免在边界处重复计数。
>
> **(c) Nonsequenced Query — 忽略时态语义的全历史查询**
> ```sql
> SELECT DISTINCT emp_id
> FROM emp_salary
> WHERE salary > 85000;
> -- 结果：emp_id 101（曾有 90,000 的记录）
> ```
> Nonsequenced Query 不考虑 `valid_start` / `valid_end`，直接对所有历史记录做统计，等同于把时态表当作普通表处理。

**Question 7.** A company stores employee locations in a table `Employees(id, name, city, since)` — recording where each employee currently works and since when. A manager asks: *"What city was Alice in before she moved to Sydney?"* The engineer cannot answer this question from the table. What type of table is this, and why can it not answer the question?

- [ ] A. Bitemporal table — it is missing a Transaction Time column
- [ ] B. Valid-time table — it lacks a `valid_end` column to close intervals
- [x] C. Semi-temporal table — it only records when the current fact became true, not prior states
- [ ] D. Sequence-based table — history is stored as arrays, not individual rows

> [!note]- Answer
> **C. Semi-temporal table.**
>
> `Employees(id, name, city, since)` 是 **Semi-temporal table**：只有一个 `since` 时间戳，表示当前事实从 `since` 起持续到现在（语义：`[since, NOW)`）。当员工城市变更时，旧记录被新记录覆盖，之前的历史**永久丢失**。  
>
> 对比：
> | 表类型 | 结构 | 能查历史？ |
> |--------|------|-----------|
> | Semi-temporal | 只有 `since` | ❌ 只知当前 |
> | Valid-time | `start_date` + `end_date` | ✅ 完整历史 |
> | Bitemporal | Valid Time + Transaction Time | ✅✅ 历史 + 审计 |
>
> 要回答"之前在哪"，必须改为 Valid-time table（每次变更新增一行，用 `end_date` 关闭旧记录）。

**Question 8.** A data engineer is designing a real-time pipeline for sensor data from a global fleet of delivery trucks. She lists the following challenges: sensors occasionally go offline (no reading for hours), GPS pings may arrive at the processing server 3–5 minutes after the actual event, readings are captured at irregular intervals depending on vehicle speed, the fleet spans 12 time zones, and daily volumes reach 50 GB. Which set of terms correctly names the five engineering challenges she is facing?

- [ ] A. Schema drift / Replication lag / Partitioning skew / Locale conflict / OOM errors
- [x] B. Missing data / Late-arriving data / Irregular intervals / Time zones / Large volume
- [ ] C. NULL propagation / Transaction latency / Index fragmentation / UTC offset / Disk I/O
- [ ] D. Data skew / Network jitter / Clock drift / Format mismatch / Retention overflow

> [!note]- Answer
> **B.** 时序数据的五大工程挑战：
>
> | 挑战 | 场景中的体现 |
> |------|------------|
> | **Missing data** | 传感器离线，数小时无读数 |
> | **Late-arriving data** | GPS ping 延迟 3–5 分钟到达（乱序） |
> | **Irregular intervals** | 读数频率随车速变化，间隔不等 |
> | **Time zones** | 跨 12 个时区，时间对齐复杂 |
> | **Large volume** | 每日 50 GB，需要 downsampling / retention 策略 |
>
> 这五个挑战是区别时序系统与普通 OLTP/OLAP 系统的核心原因，也是选用专用 TSDB（TimescaleDB / InfluxDB）而非标准关系库的动机。

**Question 9.** A data engineer stores one year of temperature readings for 500 weather stations in a PostgreSQL table using Sequence-based representation — each station has a single row with an array of all readings. A colleague points out that this design "violates 1NF." What does this mean, and what SQL function is needed to query individual readings from the array?

- [ ] A. It violates 1NF because arrays cannot be indexed; use a B-Tree index to fix it
- [x] B. It violates 1NF because each cell contains a multi-valued array (non-atomic); use `UNNEST()` to expand arrays into rows for querying
- [ ] C. It violates 1NF because the table has no primary key; add a `station_id` column
- [ ] D. It violates 1NF because timestamps are stored implicitly by position; use `DATERANGE` to fix it

> [!note]- Answer
> **B.**
>
> **1NF（First Normal Form）** 要求每列的每个值必须是**原子值（atomic）**——不可再分。Sequence-based 存储将多个读数打包进一个数组列（如 `FLOAT[]`），违反了这一规则。
>
> **关键操作对比**：
> | 操作 | 函数 | 方向 |
> |------|------|------|
> | 查询时展开数组 | `UNNEST(array_col)` | 数组 → 多行 |
> | 写入时聚合多行 | `array_agg(value)` | 多行 → 数组 |
> | 数组列的索引类型 | `USING GIN` | 非 B-Tree，非 GiST |
>
> Sequence-based 的代价：查询复杂（必须先 UNNEST）、不跨 DBMS 移植、GIN 索引开销高。优势：行数少，批量分析紧凑。

**Scenario 2:** A logistics company stores shipment status updates in a temporal table. Each status change creates a new row with `valid_start` and `valid_end`. A data engineer notices that field agents sometimes submit status updates hours or days after the actual event occurs. She proposes adding `txn_start` and `txn_end` columns to track when each record was entered into the system.

**Question 6:** **(4 Marks)**

**(a)** In this logistics context, what is the difference between Valid Time and Transaction Time? Give a concrete example using a shipment status update. **(2 Marks)**

**(b)** A status record shows `valid_start = 2024-03-10`, `valid_end = 9999-12-31`, `txn_start = 2024-03-12`. What does this tell us about when the event occurred versus when it was recorded in the system? Why does this distinction matter? **(2 Marks)**

> [!note]- Answer
> **(a) Valid Time vs Transaction Time（物流场景）**
>
> - **Valid Time（有效时间）**：该状态在现实中成立的时段。例：货物于 2024-03-10 实际抵达仓库，这是现实中的事件时间，可以被修正（如发现记录错误）。
> - **Transaction Time（事务时间）**：该记录被写入数据库的时间。例：司机于 2024-03-12 才用 App 上传抵达状态，这是数据进入系统的时间，Append-only，不可修改，用于审计。
>
> 两者的差值（2 天）揭示了字段上报的延迟，是物流运营分析的重要指标。
>
> **(b) 解读**
>
> - `valid_start = 2024-03-10`：货物在现实中于 **3 月 10 日**发生状态变更（如抵达仓库）
> - `valid_end = 9999-12-31`：该状态**至今仍有效**（open-ended，尚未被新状态覆盖）
> - `txn_start = 2024-03-12`：该记录于 **3 月 12 日**才被录入数据库（存在 2 天上报延迟）
>
> 若在 3 月 11 日查询系统，系统将**不显示**此记录（Transaction Time 尚未开始）；但从业务现实（Valid Time）看，货物在 3 月 10 日就已抵达。这正是 Bitemporal 建模的核心价值——能分别回答"现实发生了什么"与"系统当时认为发生了什么"两个不同的问题。
