## 选择题

**1.** Batch Processing 和 Stream Processing 的根本区别是？

- A. Batch 处理速度更快，Stream 处理精度更高
- B. Batch 处理有界的存量数据（触发式），Stream 处理无界的持续数据（持续查询）
- C. Batch 只能用于历史分析，Stream 只能用于预测
- D. Batch 需要 Kafka，Stream 需要 Spark

<details><summary>答案</summary>

**答案：B**

解析：**Batch Processing**：数据先存储，触发时一次性处理（如每日凌晨跑 ETL），适合离线分析。**Stream Processing**：数据持续产生，实时处理每条或每批事件，适合实时监控、欺诈检测、推荐系统。核心区别：数据是否有界（Bounded vs Unbounded）、处理是否持续（One-off vs Continuous）。

</details>

**2.** Apache Kafka 中，"Consumer Group"机制的作用是？

- A. 确保同一消息只被一个消费者处理（组内竞争消费，实现负载均衡）
- B. 确保同一消息被组内所有消费者处理
- C. 限制 Producer 的发送速率
- D. 定义 Topic 的分区数量

<details><summary>答案</summary>

**答案：A**

解析：同一 **Consumer Group** 内，每个 Partition 只分配给一个消费者（组内竞争消费），实现负载均衡和水平扩展。不同 Consumer Group 之间相互独立——同一消息会被每个 Group 各消费一次（广播语义）。这两种机制的组合非常灵活。

</details>

**3.** Kafka 的 Replication Factor 设为 3，最多能容忍几台 Broker 同时故障？

- A. 0 台
- B. 1 台
- C. 2 台
- D. 3 台

<details><summary>答案</summary>

**答案：C**

解析：Replication Factor = N 时，每个 Partition 有 1 个 Leader 和 N-1 个 Follower 副本。最多容忍 **N-1 台**故障（只要 Leader 还在或能选出新 Leader）。RF=3 → 最多容忍 2 台故障，至少需要 1 台存活。这是生产环境推荐将 RF 设为 3 的原因。

</details>

**4.** Tumbling Window（翻滚窗口）和 Sliding Window（滑动窗口）的区别是？

- A. Tumbling Window 可以重叠，Sliding Window 不可以
- B. Tumbling Window 不重叠（每次窗口结束后才开始下一个），Sliding Window 可以重叠（按步长滑动）
- C. 两者都不重叠，只是窗口大小不同
- D. 两者完全相同，叫法不同而已

<details><summary>答案</summary>

**答案：B**

解析：三种窗口类型：
- **Tumbling Window**：固定大小，不重叠（如每5分钟一个窗口）；同一事件只属于一个窗口
- **Sliding Window**：固定大小但按步长滑动，可重叠（如每1分钟滑动、覆盖5分钟）；同一事件可属于多个窗口
- **Session Window**：由活动间隔定义（无活动超过N秒则关闭窗口），大小动态变化

</details>

**5.** 在流处理中，"Exactly-once semantics"（精确一次语义）解决了什么问题？

- A. 保证消息按严格顺序处理
- B. 保证每条消息被处理且只被处理一次（既不丢失也不重复）
- C. 保证处理延迟低于1毫秒
- D. 保证所有消费者同时收到消息

<details><summary>答案</summary>

**答案：B**

解析：流处理的三种语义保证：
- **At-most-once**：消息最多处理一次（可能丢失，但不重复）
- **At-least-once**：消息至少处理一次（不丢失，但可能重复）
- **Exactly-once**：消息恰好处理一次（不丢失不重复），实现最复杂，需要幂等性写入 + 分布式事务支持（Flink、Kafka Streams 支持）

</details>

**6.** Event Time、Processing Time 和 Ingestion Time 三者的区别是？

- A. Event Time 是处理时间；Processing Time 是事件发生时间；Ingestion Time 是存储时间
- B. Event Time 是事件真实发生时间；Processing Time 是处理引擎处理该事件的时间；Ingestion Time 是数据进入消息系统的时间
- C. 三者在分布式系统中完全相同，可以互换
- D. 只有 Processing Time 是精确的，其他两者不可信

<details><summary>答案</summary>

**答案：B**

解析：
- **Event Time**：事件实际发生的时刻（最准确，但可能因网络延迟乱序到达）
- **Ingestion Time**：数据进入 Kafka/消息队列的时间（折中方案）
- **Processing Time**：流处理引擎处理该记录时的系统时钟（最容易获取，但受处理延迟影响，不反映真实顺序）
- 推荐使用 **Event Time** + **Watermark** 处理乱序数据

</details>

**7.** Watermark 在流处理中的作用是？

- A. 标记数据质量差的记录，用于过滤
- B. 处理器声明"某时间点之前的所有事件已到达"，用于触发基于 Event Time 的窗口计算，处理乱序数据
- C. 限制消息队列的最大容量
- D. 标记批处理任务的开始和结束

<details><summary>答案</summary>

**答案：B**

解析：由于网络延迟，Event Time 为 T 的事件可能在 T+几秒 才到达处理器。**Watermark(T)** 表示处理器认为"T 之前的所有事件已经到达"，从而安全地触发 T 之前的窗口计算。晚于 Watermark 到达的事件是 **late data**，可选择丢弃或进入 side output。Watermark 是 Event Time 处理乱序的关键机制。

</details>

**8.** 以下哪种场景最适合使用 Session Window（会话窗口）？

- A. 每5分钟统计一次网站的总访问量
- B. 统计过去30分钟内每分钟的订单数（实时滑动统计）
- C. 统计用户每次浏览会话（session）的总页面浏览量，会话之间间隔超过30分钟即为新会话
- D. 按固定时间段统计每日销售额

<details><summary>答案</summary>

**答案：C**

解析：Session Window 按**活动间隔**（inactivity gap）划分窗口，不固定大小。用户行为分析是典型场景：用户浏览一段时间后停止超过30分钟（gap），则前一段活动构成一个 session。A 是 Tumbling Window；B 是 Sliding Window；D 是 Tumbling Window（天级别）。

</details>

## 简答题

**9.** 解释 Pub/Sub 模式的工作原理，说明 Publisher、Broker、Topic、Subscriber 各自的角色，以及它与传统点对点消息队列的区别。

<details><summary>答案</summary>

**Pub/Sub 工作原理**：
- **Publisher（发布者）**：生产消息，将消息发送到 Broker 上的特定 Topic，不关心谁接收
- **Broker（中间件）**：接收、持久化并分发消息；解耦 Publisher 和 Subscriber（Kafka 就是 Broker）
- **Topic（主题）**：消息的逻辑分类，Publisher 向 Topic 发布，Subscriber 订阅 Topic
- **Subscriber（订阅者）**：订阅 Topic，接收所有发往该 Topic 的消息

**与点对点（P2P）的区别**：P2P 中消息被一个消费者消费后即删除（Queue 语义）；Pub/Sub 中同一消息可被多个 Subscriber 各自消费（Broadcast 语义）。Kafka 通过 Consumer Group 实现两种语义的融合。

</details>

**10.** 你在设计一个实时交易欺诈检测系统：每笔交易包含 `tx_id, user_id, amount, timestamp`，需要检测用户在**过去1小时内**总交易额是否超过 10,000 元。

- 这个查询是 Stateless 还是 Stateful？
- 应该使用哪种窗口？
- 应该基于 Event Time 还是 Processing Time？说明理由。

<details><summary>答案</summary>

**Stateful（有状态）**：计算用户过去1小时总额需要跨多条交易记录维护中间状态（累积金额），不能逐条独立判断。

**窗口类型**：**Sliding Window**（滑动窗口）——"过去1小时"意味着窗口随时间连续滑动（如每1分钟更新一次、覆盖60分钟），而非固定的不重叠区间。

**Event Time**：交易的发生时间（timestamp）才是业务真实时间。使用 Processing Time 会导致因网络延迟到达的交易被错误归入另一个时间窗口，导致漏检或误检。Event Time + Watermark 能正确处理乱序到达的交易记录。

</details>

**11.** 一个 Kafka Topic 有 4 个 Partition，有 GroupA（2个消费者）和 GroupB（3个消费者）两个 Consumer Group。回答以下问题：
1. GroupA 和 GroupB 消费是否互相影响？
2. GroupA 中每个消费者消费几个 Partition？GroupB 中呢？
3. GroupA 中一个消费者宕机后会发生什么？

<details><summary>答案</summary>

1. **不影响**。两个 Consumer Group 各自维护独立的 offset 指针，同一条消息会分别传递给 GroupA 和 GroupB，互不干扰。

2. **GroupA**：4个 Partition ÷ 2个消费者 = **每人2个 Partition**。
   **GroupB**：4个 Partition ÷ 3个消费者 = **2个消费者各消费2个，1个消费者消费0个**（Partition 数 < 消费者数时，多余的消费者空闲）。

3. 触发 **Rebalance（再平衡）**：Kafka 协调器（Group Coordinator）检测到消费者宕机，将其负责的 Partition 重新分配给存活的消费者（现在那个消费者接管全部4个 Partition）。从上次提交的 offset 继续消费，**不丢失数据**。

</details>
