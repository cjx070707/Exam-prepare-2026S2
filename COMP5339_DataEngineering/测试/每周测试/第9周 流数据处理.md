# 第9周测试 — Stream Processing

---

**Question 1.** An e-commerce platform currently runs a nightly ETL job that processes all orders from the past 24 hours to update inventory levels. The business team requests a new feature: **detect and block fraudulent orders within 2 seconds of placement**. Which architectural shift is required?

- [ ] A. Switch from SQL to NoSQL — the current database cannot support real-time queries
- [ ] B. Switch from Batch Processing to Stream Processing — fraud detection requires continuous, low-latency processing of unbounded data, not one-off batch jobs
- [ ] C. Add more CPU cores to the existing batch job to run it every minute instead of nightly
- [ ] D. Use a Dedicated TSDB instead of a relational database to store order events

> [!note]- Answer
> **B. Stream Processing（流数据处理）是必要的架构转变。**
>
> 核心区别：
> - **Batch Processing（批处理）**：数据有界（bounded），触发式执行（如每天一次），适合离线分析。2 秒内检测欺诈用批处理根本无法实现。
> - **Stream Processing（流处理）**：数据无界（unbounded），持续查询（continuous query），每条事件实时处理，延迟可达毫秒级。
>
> 仅缩短批处理间隔（如每分钟跑一次）仍是批处理思路，延迟远不够；流处理是架构上的根本不同，需要引入 Kafka + Flink / Spark Streaming 等流处理栈。

**Question 2.** A Kafka topic `orders` has 6 partitions. **Group A** has 4 consumers, **Group B** has 2 consumers. A new analytics team creates **Group C** with 8 consumers to consume the same topic independently. Which statement is correct?

- [ ] A. Group C cannot consume from `orders` because Group A and B are already using it
- [ ] B. Group C will share the same offset pointer as Group A, receiving only messages Group A hasn't processed yet
- [ ] C. Group C maintains its own independent offset and will receive all messages; however, since 8 consumers > 6 partitions, 2 consumers in Group C will be idle
- [ ] D. Adding Group C will cause Group A to receive duplicate messages

> [!note]- Answer
> **C. Consumer Group 独立消费 + Partition 分配规则。**
>
> 两个关键机制：
> 1. **Consumer Group 隔离**：每个 Group 维护独立的 offset 指针，同一消息会被每个 Group 各自消费一次（广播语义）。Group C 独立消费不影响 Group A / B。
> 2. **Partition 分配规则**：组内每个 Partition 只分配给一个消费者（竞争消费，负载均衡）。当消费者数 > Partition 数时，多余的消费者空闲（idle）。6 partitions ÷ 8 consumers → 6 个消费者各负责 1 个 Partition，2 个消费者空闲。

**Question 3.** A stock trading platform streams market tick data. The risk team needs two different aggregations:
- **Metric A**: "Total trading volume in each fixed 5-minute interval (00:00–00:05, 00:05–00:10, …)"
- **Metric B**: "Average price over the last 5 minutes, updated every 30 seconds"

Which window types should be used for Metric A and Metric B respectively?

- [ ] A. Session Window for Metric A; Tumbling Window for Metric B
- [x] B. Tumbling Window for Metric A; Sliding Window for Metric B
- [ ] C. Tumbling Window for both — they both cover a 5-minute period
- [ ] D. Sliding Window for Metric A; Session Window for Metric B

> [!note]- Answer
> **B. Tumbling Window for Metric A; Sliding Window for Metric B。**
>
> - **Metric A → Tumbling Window（翻滚窗口）**：固定大小、不重叠——每5分钟是一个独立窗口，同一事件只属于一个窗口。适合按固定时段汇总的报告（如每5分钟成交量）。
>
> - **Metric B → Sliding Window（滑动窗口）**：固定大小但按步长连续滑动——窗口覆盖5分钟，每30秒更新一次（size=5min, step=30s）。同一事件会出现在多个窗口中。适合"过去N分钟"的滚动统计。
>
> **Session Window** 适合用户行为分析（由活动间隔动态定义窗口边界），不适用于固定时间长度的聚合。

**Question 4.** A ride-hailing app processes GPS location events from drivers. Due to intermittent connectivity, a driver's event timestamped `14:30:00` may arrive at the processing engine at `14:31:45` — 105 seconds late. The team uses **Event Time** with a **Watermark tolerance of 2 minutes**. What happens to this event?

- [x] A. The event will be processed correctly in the `14:30` window — the 2-minute Watermark allows events up to 2 minutes late
- [ ] B. The event will be dropped — Event Time processing requires all events to arrive in order
- [ ] C. The event will be processed using Processing Time (14:31:45) instead, causing it to land in the wrong window
- [ ] D. The Watermark has no effect on this event; it only applies to aggregation functions

> [!note]- Answer
> **A. Watermark（水位线）处理乱序事件的机制。**
>
> - **Event Time（事件时间）**：事件实际发生的时间戳（14:30:00），是业务的真实时间。
> - **Watermark(T)**：处理引擎声明"T 之前的所有事件已经到达"，触发 T 之前的窗口计算。设置 2 分钟容忍意味着引擎等待最多 2 分钟的延迟事件。
>
> 该事件延迟 105 秒（< 2 分钟），在 Watermark 容忍范围内，被正确归入 14:30 窗口处理。若延迟超过 2 分钟，则成为 **late data**（可选择丢弃或送入 side output）。
>
> **Processing Time** 使用处理引擎的系统时钟，不反映真实事件顺序，不适合对时序准确性要求高的场景。

**Question 5.** A smart city platform manages two independent data pipelines:
- **Pipeline A**: 80,000 environmental sensors (temperature, humidity, air quality) transmit readings every 15 seconds over low-bandwidth cellular connections; each device has limited battery and CPU
- **Pipeline B**: A ride-hailing app generates 3 million booking, GPS, and payment events per second; the data science team needs to replay 60 days of events for weekly model retraining and retain all messages for compliance audits

Which messaging system architecture best fits each pipeline?

- [ ] A. Kafka for both — Kafka's high throughput and persistence make it the optimal choice in all streaming scenarios
- [ ] B. MQTT for Pipeline A; Kafka for Pipeline B — MQTT's lightweight protocol suits constrained IoT devices; Kafka's persistent, replayable log handles high-throughput event streaming and long-term retention
- [ ] C. MQTT for both — it supports QoS 2 (exactly-once), providing the strongest delivery guarantee for both use cases
- [ ] D. Kafka for Pipeline A; MQTT for Pipeline B — Kafka's Replication Factor ensures sensor data is never lost

> [!note]- Answer
> **B. MQTT for IoT sensors（Pipeline A）; Kafka for high-throughput event streaming（Pipeline B）。**
>
> - **A 错误**：Kafka 客户端库较重，需要稳定网络连接和较高计算资源，不适合电池供电、低带宽的 IoT 传感器设备。
> - **B 正确**：
>   - **MQTT（Pipeline A）**：轻量级 Pub/Sub 协议，专为资源受限设备设计——低带宽、断线重连、QoS 灵活（0/1/2）。传感器读数无需长期保留，MQTT 完全足够。
>   - **Kafka（Pipeline B）**：持久化存储到磁盘、支持消息回放（60天历史重放）、高吞吐分区架构、可配置长期 retention——是高吞吐 + 历史重放 + 合规保留的标准选择。
> - **C 错误**：MQTT 不支持大规模高吞吐，无法高效保留海量历史数据用于模型重训。
> - **D 错误**：角色完全颠倒。Kafka 不适合低功耗传感器；MQTT 无法处理每秒百万级事件的持久化流。

**Scenario 1:** Your team runs a Kafka cluster with the following configuration:
- Topic `transactions` with **4 partitions**, Replication Factor = **3**, deployed on **5 brokers**
- **Group Alpha**: 3 consumers (real-time fraud detection)
- **Group Beta**: 1 consumer (audit log writer)

**Question 6:** **(6 Marks)**

**(a)** How many partition replicas exist in total for topic `transactions`? How many broker failures can the cluster tolerate while remaining operational? **(2 Marks)**

**(b)** Describe how partitions are distributed between Group Alpha's 3 consumers. What happens to Group Alpha if one consumer crashes? **(2 Marks)**

**(c)** The fraud detection system must ensure each transaction is processed **exactly once** — neither missed nor double-counted. Which delivery semantic does this require, and why is it the hardest to implement? **(2 Marks)**

> [!note]- Answer
> **(a) Replicas 和容错能力**
>
> - 总 Replica 数 = 4 partitions × Replication Factor 3 = **12 个副本**（分布在5个 Broker 上）
> - 容错能力：Replication Factor = N → 最多容忍 **N-1 = 2 台** Broker 同时故障（只要至少1台存活且能选出 Leader）
>
> **(b) Partition 分配 + Rebalance**
>
> - **Group Alpha（3 consumers, 4 partitions）**：约为 1 个消费者负责 2 个 Partition，另 2 个消费者各负责 1 个（4÷3 分配）。
> - **一个消费者宕机**：触发 **Rebalance（再平衡）**——Kafka Group Coordinator 检测到消费者失联，将其负责的 Partition 重新分配给剩余 2 个消费者，从上次提交的 offset 继续消费，不丢失数据。
>
> **(c) Exactly-once Semantics（精确一次语义）**
>
> 欺诈检测需要 **Exactly-once**：每条交易恰好处理一次，既不丢失（at-most-once 的缺陷），也不重复计算（at-least-once 的缺陷）。
>
> 实现难度最高的原因：需要同时满足：
> 1. **幂等写入（Idempotent Producer）**：重试不产生重复消息
> 2. **原子性提交**：消费 offset 的更新与业务处理结果的写出必须原子完成（分布式事务）
>
> Flink 和 Kafka Streams 通过两阶段提交（2PC）支持 Exactly-once，但引入了额外延迟和复杂度。

**Scenario 2:** A food delivery platform wants to detect when a delivery partner has been stationary for more than 10 minutes during an active order. Events are GPS pings from each partner's app, arriving every 30 seconds with an Event Time timestamp. The GPS app sometimes loses signal and sends delayed pings — a ping from 14:20 may arrive at 14:23.

**Question 7:** **(4 Marks)**

**(a)** Should this pipeline use **Stateless** or **Stateful** processing? Which window type is most appropriate? Justify both answers. **(2 Marks)**

**(b)** Should the team use **Event Time** or **Processing Time**? What mechanism handles the delayed GPS pings? **(2 Marks)**

> [!note]- Answer
> **(a) Stateful + Session Window**
>
> - **Stateful（有状态）**：判断"是否静止超过10分钟"需要记录该配送员之前的 GPS 位置和时间——跨多条 ping 维护中间状态，不能逐条独立判断（Stateless）。
>
> - **Session Window（会话窗口）**：检测"超过10分钟无移动"正是 Session Window 的核心语义——当两次相邻 ping 的间隔（inactivity gap）超过阈值时，窗口关闭并触发告警。相比 Tumbling / Sliding，Session Window 的边界由活动间隔动态决定，完全符合此场景。
>
> **(b) Event Time + Watermark**
>
> 应使用 **Event Time**：GPS ping 的时间戳才是配送员真实的位置时间，用 Processing Time 会因网络延迟把14:20的事件误判为14:23发生，导致漏检或误检。
>
> **Watermark** 处理延迟到达：设置适当的 Watermark 容忍（如3分钟），允许延迟不超过3分钟的 ping 被正确归入其 Event Time 对应的窗口；超过容忍时间的 ping 视为 late data 处理。

**Question 8.** A real-time payment system processes 50,000 transactions per second and needs to enrich each transaction with the merchant's category (e.g., "food", "electronics"). Merchant data is stored in a reference database and updated nightly. Which approach is most appropriate?

- [ ] A. Query the production MySQL database for each transaction event
- [ ] B. Load merchant data into in-memory storage (e.g., Redis) and look it up per event
- [ ] C. Use a Stream-Stream Join — join the transaction stream with a live merchant update stream
- [ ] D. Collect transactions in a 1-hour Tumbling Window, then batch-join with the merchant table

> [!note]- Answer
> **B. Enrichment（Stream-Table Join）— in-memory lookup via Redis。**
>
> - **A 错误**：50,000 TPS 直接查生产数据库会严重拖慢流处理，同时影响在线业务。
> - **B 正确**：将静态参考数据（商户类别）预加载到 Redis，每条事件毫秒级 lookup。商户数据每晚更新，无需实时流——更新时刷新缓存即可。这是 **Enrichment 模式（Stream-Table Join）** 的标准实现：避免直接查生产库的最佳实践。
> - **C 错误**：Stream-Stream Join 用于两条都实时变化的流（如点击流 join 曝光流）；商户数据每晚才变，不需要流 join。
> - **D 错误**：1小时批次延迟不可接受，不适合实时支付。
>
> **三种 Stream Join 模式**：Conventional（流→表→join）/ Enrichment（流 + 静态参考数据，Redis lookup，最常见）/ Stream-Stream（两流窗口内配对）。
