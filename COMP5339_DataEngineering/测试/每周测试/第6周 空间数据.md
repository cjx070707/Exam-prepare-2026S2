# 第6周测试 — 空间数据

## 选择题

**1.** 一家城市规划公司正在比较两种空间数据建模方式：一种用离散的建筑物、道路、公园对象表示城市；另一种用连续的温度分布图层表示热岛效应。这两种分别对应哪种空间数据模型？

- A. Field-based 和 Object-based
- B. Object-based 和 Field-based
- C. 两种都是 Object-based，只是粒度不同
- D. 两种都是 Field-based，区别在于栅格分辨率

> [!tip]- 答案
> **答案：B**
> 解析：**Object-based 模型**将空间表示为离散、有边界的对象（建筑物、道路、公园），每个对象有独立属性。**Field-based 模型**将空间表示为连续的值域（如温度、高程、降雨量），整个研究区域每一点都有值。两者本质区别：前者是"有什么对象在这里"，后者是"这里的值是多少"。

**2.** 一个 GIS 团队在澳大利亚境内做土地管理分析，需要在法律层面匹配官方地籍数据。他们应选用哪个坐标参考系，原因是什么？

- A. WGS84（SRID 4326），因为它是 GPS 的全球标准，精度最高
- B. Mercator 投影，因为它保持面积不变，适合土地面积计算
- C. GDA94，因为它是澳大利亚官方测量坐标系，与本地地籍数据一致
- D. Geocentric 坐标系，因为它以地心为原点，误差更小

> [!tip]- 答案
> **答案：C**
> 解析：**GDA94**（Geocentric Datum of Australia 1994）是澳大利亚官方测量和地籍的标准坐标系，政府土地数据均以此为基准。WGS84 是 GPS 全球标准，与 GDA94 差约1米（在很多应用中可互换，但法定测量场景必须用 GDA94）。Mercator 投影不保面积而保角（等角投影），大范围面积计算会失真。

**3.** 某全球物流公司需要计算跨越多个大洲的货船航线长度（数千公里）。工程师在 PostGIS 中存储航线时，应选 `geometry` 还是 `geography` 类型，为什么？

- A. `geometry`，因为它运算速度更快，对长距离路线更合适
- B. `geography`，因为它基于球面模型，在大范围距离计算上精度更高
- C. 两者没有区别，PostGIS 会自动选择合适模型
- D. `geometry`，因为 `geography` 类型不支持 LineString

> [!tip]- 答案
> **答案：B**
> 解析：**`geometry`** 基于平面模型（笛卡尔坐标），距离单位取决于 SRID（如度），在小范围或已投影坐标系下计算快、够用。**`geography`** 基于球面（椭球体）模型，距离单位始终为米，适合跨大洲的大范围计算，不会因 Mercator 等投影的变形导致误差。核心原则：**范围越大 → 用 `geography`**。

**4.** 九交叉模型（DE-9IM）将两个几何体的空间关系分解为 Interior/Boundary/Exterior 三个部分的两两相交，共9种组合。下列哪种关系描述了"两个多边形共享边界线段但内部不重叠"？

- A. Contains
- B. Disjoint
- C. Touches
- D. Overlaps

> [!tip]- 答案
> **答案：C**
> 解析：**Touches**（接触）：两个几何体的边界（Boundary）有交集，但内部（Interior）不相交。典型场景：两块相邻的土地地块，共享边界线但不重叠。**Disjoint**：完全不接触。**Overlaps**：内部有部分重叠（面积交集）。**Contains**：一个完全包含另一个的内部。九交叉模型的核心就是通过这9个 I/B/E 组合的 T/F 矩阵区分这8种拓扑关系。

**5.** 工程师为一张存储100万个餐厅位置点的表建立了 GiST 空间索引（R-Tree）。下列对该索引工作方式的描述，哪个最准确？

- A. R-Tree 将所有点按经度排序后存入 B-Tree，再按纬度二次过滤
- B. R-Tree 将空间对象用最小外接矩形（MBR）包围，并按层级嵌套组织，查询时先矩形粗筛再精确计算
- C. R-Tree 把整个地图切成等大的网格格子，每个格子存对应的点
- D. R-Tree 将空间对象哈希到固定桶中，实现 O(1) 查找

> [!tip]- 答案
> **答案：B**
> 解析：**R-Tree** 的核心思路：① 每个空间对象用 **MBR（Minimum Bounding Rectangle）** 近似；② MBR 按层级组织成树，父节点 MBR 包含所有子节点 MBR；③ 查询时从根节点开始，跳过与查询区域不相交的子树（**粗筛剪枝**），只对少量候选对象做精确几何计算（**精筛**）。这将时间复杂度从 O(n) 降至 O(log n)。选项 C 描述的是网格索引，D 是哈希索引，两者都不是 R-Tree。

**6.** 一名数据工程师将来自两个不同部门的空间数据集做 Spatial Join：部门 A 的数据使用 GDA94（SRID 7844），部门 B 使用 WGS84（SRID 4326）。如果不做任何处理直接运算，最可能出现什么问题？

- A. 查询直接报错，PostGIS 拒绝执行 SRID 不一致的操作
- B. 结果看上去正常，但空间关系在错误的坐标框架下计算，产生系统性偏差
- C. PostGIS 自动识别两种坐标系并在后台完成转换，结果正确
- D. 只影响距离计算，不影响 Contains 和 Intersects 等拓扑判断

> [!tip]- 答案
> **答案：B**
> 解析：PostGIS 在 SRID 不一致时**不保证自动转换**——某些版本会报错，某些版本静默地在混用坐标系的情况下计算，结果完全不可信（拓扑关系和距离都会出错）。正确做法是用 `ST_Transform()` 显式统一 SRID 后再运算。**考试核心点**：SRID 不统一是空间数据工程中最常见的"静默错误"来源，结果看似有输出，实则错误。

**7.** 一个移动应用需要将用户标注的地点实时传输给 Web 后端并在地图上展示。下列哪种空间数据格式最适合这个场景？

- A. Shapefile，因为它是 GIS 行业标准，兼容性最好
- B. WKT，因为它语法简洁、文件体积小
- C. KML，因为它包含丰富的样式和标注信息
- D. GeoJSON，因为它基于 JSON，可直接嵌入 REST API 响应并被 Web 地图库解析

> [!tip]- 答案
> **答案：D**
> 解析：**GeoJSON** 是 Web 场景的首选：① 基于 JSON，与现代 REST API 无缝集成；② Leaflet、Mapbox、Google Maps 等主流 Web 地图库原生支持；③ 轻量、人类可读。**Shapefile** 是多文件格式（.shp/.dbf/.prj等），不适合 API 传输。**KML** 是 XML 格式、Google Earth 专用，Web 集成复杂。**WKT** 是数据库内部文本格式，缺乏属性信息承载能力，不适合 API 响应。

**8.** DuckDB 的空间扩展与 PostGIS 相比，最主要的局限性是什么？

- A. DuckDB 不支持 GeoJSON 格式的读写
- B. DuckDB 没有独立的 Geography 类型，所有空间计算默认基于平面模型（EPSG:4326），无法原生做球面距离计算
- C. DuckDB 空间扩展不支持 Point 和 Polygon 之外的几何类型
- D. DuckDB 的空间索引性能比 PostGIS 差10倍以上，不适合生产环境

> [!tip]- 答案
> **答案：B**
> 解析：PostGIS 有 `geometry`（平面）和 `geography`（球面）两种类型，`geography` 可以做精确的球面距离计算，适合大范围全球数据。DuckDB 的空间扩展**只有 `geometry` 类型**，默认坐标系 EPSG:4326，所有计算基于平面模型——这在小范围（城市级别）可接受，但跨洲际的大范围距离计算会有系统性误差。DuckDB 空间扩展的定位是分析型场景的便捷工具，而非替代 PostGIS 的生产级空间数据库。

## 简答题

**9.** 解释 Spatial 和 Geospatial 的区别，并说明 SDBMS 与 GIS 在职责分工上的不同。**(4 Marks)**

> [!tip]- 答案
> **Spatial vs Geospatial（2分）：**
> - **Spatial**（空间数据）：泛指任意坐标空间中的数据，坐标不必对应真实地理位置（如医学影像坐标、工厂平面图坐标）。
> - **Geospatial**（地理空间数据）：特指对应地球表面真实地理位置的数据，使用经纬度或地图投影坐标，与现实地理位置绑定。
>
> **SDBMS vs GIS 分工（2分）：**
> - **SDBMS（Spatial DBMS）**：负责后端存储、空间索引（R-Tree/GiST）、空间查询执行（如 ST_Contains、ST_Distance）——是"存储和检索引擎"，如 PostGIS。
> - **GIS（Geographic Information System）**：负责前端分析与可视化，将数据组织成图层（Layer），提供地图渲染、叠加分析、路径规划等功能——是"分析和展示平台"，如 QGIS、ArcGIS。
> - 两者常配合使用：GIS 从 SDBMS 查询数据，再做空间分析和可视化。

**10.** 一家共享单车公司计划分析哪些区域的单车停放密度超标（每平方公里超过500辆）。请描述完成此类空间分析需要处理的三类工程挑战，并对每项挑战给出具体应对策略。**(6 Marks)**

> [!tip]- 答案
> **挑战一：坐标系不统一（2分）**
> - 问题：单车 GPS 数据（WGS84）与城市区域边界数据（可能是 GDA94 或本地投影）SRID 不一致，直接做 Spatial Join 结果错误。
> - 应对：在入库或查询时统一用 `ST_Transform()` 转换到同一坐标系，并在 pipeline 中做 SRID 校验。
>
> **挑战二：Spatial Join 计算代价高（2分）**
> - 问题：将数百万条停车记录与城市区域多边形做空间包含查询（ST_Contains），若无索引则为 O(n×m) 全表扫描，性能不可接受。
> - 应对：在几何列上建立 GiST 索引（R-Tree），先用 MBR 粗筛大幅减少候选对，再做精确几何计算。
>
> **挑战三：大范围距离/面积计算的精度问题（2分）**
> - 问题：若使用平面 `geometry` 类型计算城市区域面积，Mercator 等投影在高纬度或大范围会严重失真，导致密度计算错误。
> - 应对：对面积计算使用 `geography` 类型（球面模型），或先将数据投影到适合本地的等积投影（如澳大利亚 Albers 等积投影），再计算面积。

**11.** 解释九交叉模型（Nine-Intersection Model）的基本框架，并用它区分 Disjoint、Touches 和 Overlaps 三种拓扑关系。**(5 Marks)**

> [!tip]- 答案
> **九交叉模型框架（2分）：**
> 将每个几何体分解为三个部分：**Interior（内部 I）**、**Boundary（边界 B）**、**Exterior（外部 E）**。对两个几何体 A 和 B，两两组合得到 9 个交集：I(A)∩I(B)、I(A)∩B(B)、…、E(A)∩E(B)，每个交集判断是否为空（T/F），形成 3×3 的 T/F 矩阵。不同矩阵模式对应不同的拓扑关系。
>
> **三种关系的区别（3分）：**
> - **Disjoint（不相交）**：A 和 B 完全分离，I(A)∩I(B)=∅，B(A)∩B(B)=∅，两者任何部分均无接触。
> - **Touches（接触）**：边界有交集（B(A)∩B(B)≠∅ 或 I∩B≠∅），但内部不重叠（I(A)∩I(B)=∅）。典型例子：两块相邻地块共享边界线。
> - **Overlaps（重叠）**：内部有面积交集（I(A)∩I(B)≠∅），但各自也有对方不覆盖的部分——即部分重叠，非完全包含。典型例子：两个有交集的服务覆盖区域。
