## 选择题

**1.** SRID 4326（WGS84）是什么？

- A. 一种空间索引算法
- B. GPS 使用的标准经纬度坐标参考系（地理坐标系）
- C. PostGIS 的版本号
- D. 一种空间数据文件格式

<details><summary>答案</summary>

**答案：B**

解析：SRID（Spatial Reference Identifier）是坐标参考系的编号。**SRID 4326 / WGS84** 是 GPS 系统使用的全球地理坐标系，单位是度（经度/纬度）。距离计算在 `geography` 类型下单位为米；在 `geometry` 类型（SRID 4326）下单位是度，需注意转换。

</details>

**2.** 为什么普通的 B-Tree 索引不适合空间数据查询？

- A. B-Tree 只支持整数类型，无法存储坐标
- B. B-Tree 是一维线性排序结构，无法处理二维空间关系（如"范围内"、"包含"等）
- C. B-Tree 存储效率比 R-Tree 低10倍以上
- D. B-Tree 不支持 SQL 查询

<details><summary>答案</summary>

**答案：B**

解析：B-Tree 基于线性比较（`<`、`>`、`=`），适合1D有序数据。空间查询需要处理二维（甚至多维）的几何关系（如包含、相交、距离）——没有线性顺序可言。**R-Tree**（PostgreSQL 中通过 GiST 索引实现）用最小外接矩形（MBR）层级组织空间对象，支持多维过滤。

</details>

**3.** PostGIS 函数 `ST_Contains(A, B)` 返回 true 的条件是？

- A. A 和 B 相交（有共同区域）
- B. B 完全包含在 A 的内部（B 的所有点都在 A 内）
- C. A 完全包含在 B 的内部
- D. A 和 B 共享边界

<details><summary>答案</summary>

**答案：B**

解析：`ST_Contains(A, B)` 检查 **A 是否包含 B**（即 B 完全在 A 内部）。注意参数顺序：第一个参数是容器，第二个是被包含的几何体。反过来 `ST_Within(B, A)` 等价。`ST_Intersects` 是有交叉；`ST_Touches` 是只共享边界点。

</details>

**4.** 以下 SQL 查询的作用是？

```sql
SELECT s.name, r.region_name
FROM Stations s, Regions r
WHERE ST_Contains(r.geom, s.geom)
  AND r.region_name = 'Greater Sydney';
```

- A. 找出所有与 Greater Sydney 区域相交的气象站
- B. 找出所有位于 Greater Sydney 区域内部的气象站名称
- C. 找出 Greater Sydney 区域内所有区域名称
- D. 计算 Greater Sydney 区域的面积

<details><summary>答案</summary>

**答案：B**

解析：这是空间 JOIN 查询：`ST_Contains(r.geom, s.geom)` 判断气象站点（`s.geom`）是否在区域多边形（`r.geom`）内部；`AND r.region_name = 'Greater Sydney'` 进一步过滤只看 Greater Sydney 区域。结果：所有完全在 Greater Sydney 内的气象站名称。

</details>

**5.** 执行空间查询时，两张表的坐标系（SRID）不一致会导致什么？

- A. 查询直接报错，PostgreSQL 拒绝执行
- B. 查询能执行，但结果错误（空间关系在错误的坐标系下计算）
- C. 只影响距离计算，不影响包含/相交判断
- D. PostGIS 会自动转换坐标系，结果正确

<details><summary>答案</summary>

**答案：B**

解析：PostGIS 在 SRID 不同时**不会自动转换**（某些版本会报错，某些版本静默计算），结果会是基于错误坐标系的几何计算，完全不可信。正确做法：用 `ST_Transform(geom, target_srid)` 显式转换到同一坐标系后再做运算。**考试要点**：两表 SRID 必须一致才能做空间运算。

</details>

**6.** GeoJSON、KML、WKT 三种格式中，最适合 Web API 数据交换的是？

- A. KML，因为它是 Google 的标准
- B. WKT，因为它最简洁
- C. GeoJSON，因为它基于 JSON，易与 Web 技术集成
- D. 三者都一样适合

<details><summary>答案</summary>

**答案：C**

解析：**GeoJSON** 基于 JSON 格式，与现代 Web API（RESTful）天然集成，被 Leaflet、Mapbox 等 Web 地图库直接支持。**KML** 是 XML 格式，主要用于 Google Earth 可视化。**WKT**（Well-Known Text，如 `POINT(151.2 -33.8)`）是文本格式，主要用于数据库存储和调试，不适合 API 传输。

</details>

**7.** 在 PostGIS 中，建立空间索引的正确 SQL 语句是？

- A. `CREATE INDEX ON restaurants USING BTREE(location)`
- B. `CREATE INDEX ON restaurants USING GIST(location)`
- C. `CREATE SPATIAL INDEX ON restaurants(location)`
- D. `ALTER TABLE restaurants ADD SPATIAL INDEX(location)`

<details><summary>答案</summary>

**答案：B**

解析：PostGIS 的空间索引使用 **GiST**（Generalized Search Tree）访问方法，内部实现为 R-Tree。`CREATE INDEX ON table USING GIST(geom_column)` 是标准写法。B-Tree 索引不能加速空间查询，语法 C 和 D 在 PostgreSQL 中不存在。

</details>

**8.** 查询数据库中距离悉尼市中心（`POINT(151.2093 -33.8688)`）10公里以内的所有餐厅，以下哪个查询语法是正确的？

- A. `WHERE ST_Distance(location, 'POINT(151.2093 -33.8688)') < 10`
- B. `WHERE ST_Distance(location::geography, ST_GeomFromText('POINT(151.2093 -33.8688)', 4326)::geography) <= 10000`
- C. `WHERE ST_Contains(location, ST_Point(151.2093, -33.8688))`
- D. `WHERE location WITHIN 10000 OF POINT(151.2093 -33.8688)`

<details><summary>答案</summary>

**答案：B**

解析：`ST_Distance` 在 `geography` 类型下单位为**米**（10公里 = 10000米）。`::geography` 类型转换告诉 PostGIS 按地球曲面计算真实距离。`ST_GeomFromText('POINT(...)', 4326)` 从 WKT 创建几何体并指定 SRID。A 缺少 geography 转换且单位不对（geometry 类型下是度）；C 用错了函数；D 语法不存在。

</details>

## 简答题

**9.** 解释 R-Tree 空间索引的基本原理，它如何加速空间查询？

<details><summary>答案</summary>

**R-Tree 原理**：
1. 将每个空间对象用**最小外接矩形**（MBR，Minimum Bounding Rectangle）包围
2. 按层级将 MBR 组织成树结构：父节点的 MBR 包含所有子节点的 MBR
3. 查询时：先用矩形过滤快速排除不相关候选（**粗筛**），只对少量候选对象做精确空间计算（**精筛**）

**加速原理**：通过层级 MBR 剪枝，大量空间对象在粗筛阶段被排除，避免对每个对象做精确计算，将时间复杂度从 O(n) 降为 O(log n)。

在 PostGIS 中：`CREATE INDEX ON table USING GIST(geom)` 创建的就是基于 R-Tree 的 GiST 索引。

</details>

**10.** 描述从获取含经纬度的 CSV 数据到执行空间查询的完整工作流程（至少5个步骤）。

<details><summary>答案</summary>

1. **获取数据**：从来源获得含经度（longitude）和纬度（latitude）列的 CSV 文件
2. **安装扩展，建表**：`CREATE EXTENSION postgis;` 并创建含 `geometry` 或 `geography` 类型字段的表
3. **导入并转换坐标**：将 CSV 的经纬度列转为 PostGIS geometry 类型存入数据库，确认 SRID（通常为 4326）：
   ```sql
   UPDATE table SET geom = ST_SetSRID(ST_MakePoint(longitude, latitude), 4326);
   ```
4. **建立空间索引**：`CREATE INDEX ON table USING GIST(geom);`
5. **执行空间查询**：使用 PostGIS 函数（ST_Distance、ST_Contains、ST_Intersects 等）
6. **（可选）可视化**：将结果导出为 GeoJSON，在 Leaflet/QGIS 等工具上可视化验证

</details>

**11.** 对比 GeoJSON、KML、WKT 三种空间数据格式，说明各自格式特点和典型用途。

<details><summary>答案</summary>

| 格式 | 语法基础 | 特点 | 典型用途 |
|------|---------|------|---------|
| **GeoJSON** | JSON | 轻量，人类可读，与 Web API 无缝集成 | Web 地图（Leaflet/Mapbox）、REST API 数据交换 |
| **KML** | XML | Google 标准，支持样式/图标/时间轴 | Google Earth 可视化，Google Maps 导入 |
| **WKT** | 纯文本 | 极简，如 `POINT(151.2 -33.8)`、`POLYGON(...)` | 数据库字段存储（PostGIS）、调试和文档 |

**考试要点**：GeoJSON 是 Web API 首选；KML 是 Google Earth 专用；WKT 是数据库内部格式。

</details>
