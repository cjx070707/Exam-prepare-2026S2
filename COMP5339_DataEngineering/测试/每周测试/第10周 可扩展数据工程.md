## 参考答案

1. **C**
2. **B**
3. **C**
4. **B**
5. **D**
6. **CAP 定理**：分布式系统只能同时满足一致性（C）、可用性（A）、分区容错性（P）中的两个。
   - **CP 系统**：牺牲可用性，保证一致性。适用于金融系统、银行转账（不能读到旧数据）。
   - **AP 系统**：牺牲一致性，保证可用性。适用于社交媒体、购物车（允许短暂不一致，但系统不能宕机）。
7.
   - **Lazy + Primary Copy**：写入只发给 Primary，异步同步到副本，性能好；单 Primary 无写冲突，实现简单。缺点是可能短暂读到旧数据。
   - **Eager + Multi-Leader**：所有副本同步写入，强一致但性能差；多 Leader 可能产生写冲突，需要解决机制，维护复杂。
   - 实践中 **Lazy + Primary Copy** 在性能和简洁性上最优，是 MongoDB 等主流系统的默认策略。
8.
   - **Transformation**：对 RDD/DataFrame 的懒惰操作，不立即执行，只记录计划（如 `filter`、`select`、`groupBy`）。
   - **Action**：触发实际计算并返回结果的操作（如 `count`、`collect`、`show`、`write`）。
   - **好处**：Spark 可以对整个计算计划做全局优化（如谓词下推、合并操作），避免不必要的中间结果，提升效率。
9.
   - 查询1（点查询 by user_id）→ **Hash Partitioning on user_id**：哈希分区将相同 user_id 的数据路由到固定节点，点查询只需访问一个分区，O(1) 定位。
   - 查询2（时间范围查询）→ **Range Partitioning on event_time**：时间范围分区使同一时间段的数据物理相邻，范围查询只扫描相关分区，避免全表扫描。
10. 问题：`print(result)` 只打印 DataFrame 的结构描述，不触发计算，不会输出数据内容。原因：Spark Lazy Evaluation，`filter` 和 `select` 是 Transformation，不执行实际计算。
    修改：
    ```python
    result.show()     # 触发计算并显示前20行
    # 或
    result.collect()  # 触发计算并返回所有行到 Driver
    # 或
    result.write.parquet("s3://output/")  # 触发计算并写出结果
    ```
