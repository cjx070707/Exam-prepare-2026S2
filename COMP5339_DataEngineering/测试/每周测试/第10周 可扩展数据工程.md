## 选择题

**1.** HDFS 中，NameNode 和 DataNode 各自的角色是？

- A. NameNode 存储实际数据块；DataNode 存储元数据
- B. NameNode 管理命名空间和元数据（文件→块位置映射）；DataNode 存储实际数据块
- C. 两者都存储数据块，NameNode 额外负责负载均衡
- D. NameNode 执行 MapReduce 任务；DataNode 存储结果

<details><summary>答案</summary>

**答案：B**

解析：HDFS 架构分离了元数据和数据存储：**NameNode**（单点）：存储文件系统命名空间（文件路径、权限）以及文件块到 DataNode 的映射，不存实际数据；**DataNode**（多个）：存储实际数据块（默认 64MB），并向 NameNode 发送心跳和块报告。客户端先查 NameNode 获取块位置，再直接从 DataNode 读取数据。

</details>

**2.** Column Store（列式存储）相比 Row Store（行式存储）在 OLAP 场景下性能更好的主要原因是？

- A. Column Store 支持更复杂的 SQL 语法
- B. Column Store 只读取查询涉及的列，减少 I/O，且同类型数据压缩率更高
- C. Column Store 写入速度比行式存储快10倍以上
- D. Column Store 支持完整的 ACID 事务

<details><summary>答案</summary>

**答案：B**

解析：OLAP 查询通常只访问少数几列（如 `SELECT SUM(sales) GROUP BY region`）。**Row Store** 必须读取整行（包含不需要的字段）；**Column Store** 只读取相关列，I/O 大幅减少。另外，同一列的数据类型相同（如全是整数/日期），压缩率极高（RLE、字典编码等）。代价：写入需要更新多个列文件，写入性能较差。

</details>

**3.** Hash Partitioning 和 Range Partitioning 各自最适合的查询类型是？

- A. Hash 适合范围查询；Range 适合点查询
- B. Hash 适合点查询（等值查找）；Range 适合范围查询（BETWEEN、ORDER BY）
- C. 两者都适合所有查询类型
- D. Hash 适合聚合查询；Range 适合全表扫描

<details><summary>答案</summary>

**答案：B**

解析：**Hash Partitioning**：将数据按哈希值分配到固定分区，等值查询（`WHERE user_id = 'abc'`）只需访问一个分区，效率高；但范围查询需要扫描所有分区。**Range Partitioning**：按值范围分区（如按日期），范围查询只访问相关分区；等值查询效率也不错；还支持 ORDER BY 利用分区顺序优化。

</details>

**4.** Apache Spark 中，以下哪个操作是 Action（会触发实际计算）？

- A. `df.filter(df.age > 20)`
- B. `df.select("name", "age")`
- C. `df.groupBy("region")`
- D. `df.count()`

<details><summary>答案</summary>

**答案：D**

解析：Spark 中操作分两类：**Transformation**（懒惰，只记录计划，不执行）：filter、select、groupBy、map、join 等；**Action**（触发执行，返回结果）：count()、show()、collect()、write()、save() 等。Spark 的 Lazy Evaluation 让它能对整个计划做全局优化再执行。

</details>

**5.** MapReduce 的 Map 和 Reduce 阶段分别做什么？

- A. Map 汇总结果；Reduce 过滤数据
- B. Map 对每条输入记录产生(key, value)对（过滤/变换）；Reduce 按 key 聚合所有对应的 value（汇总）
- C. Map 将数据写入磁盘；Reduce 从磁盘读取数据
- D. Map 并行读取；Reduce 串行写入

<details><summary>答案</summary>

**答案：B**

解析：**Map**：对每条输入记录独立处理，输出 `(key, value)` 对（如 WordCount 中：每个单词 → `(word, 1)`）。框架自动按 key 分组（Shuffle & Sort）。**Reduce**：对同一 key 的所有 value 进行聚合（如 WordCount 中：对每个 word，将所有 1 求和得词频）。MapReduce 隐藏了并行化、容错、数据分发的细节。

</details>

**6.** 以下 PySpark 代码有什么问题？

```python
df = spark.read.parquet("s3://data/logs/")
result = df.filter(df.status == "error").select("user_id", "message")
print(result)
```

- A. `parquet` 格式不支持 `filter` 操作
- B. `print(result)` 只打印 DataFrame 的结构描述，不会输出实际数据（缺少 Action）
- C. `select` 必须在 `filter` 之前调用
- D. S3 路径格式错误

<details><summary>答案</summary>

**答案：B**

解析：`filter` 和 `select` 都是 Transformation（懒惰），不执行实际计算。`print(result)` 打印的是 DataFrame 对象的字符串描述（schema 信息），不是数据内容。要看实际数据，需要调用 Action：
```python
result.show()      # 显示前20行
result.collect()   # 返回所有行到 Driver
result.write.parquet("s3://output/")  # 写出
```

</details>

**7.** Data Replication（数据复制）中，Synchronous（同步复制）和 Asynchronous（异步复制）的主要区别和权衡是？

- A. 同步复制更快；异步复制更安全
- B. 同步复制：写操作等待所有副本确认（强一致，高延迟）；异步复制：写操作只等 Primary 确认（低延迟，可能短暂不一致）
- C. 两者在现代系统中效果完全相同
- D. 同步复制只用于备份；异步复制用于生产

<details><summary>答案</summary>

**答案：B**

解析：**Synchronous（Eager Replication）**：写操作必须等所有副本都写成功才返回 ACK，保证强一致性，但延迟高（受最慢副本影响）。**Asynchronous（Lazy Replication）**：写入 Primary 后立即返回，副本异步更新，延迟低但存在短暂数据不一致窗口（Primary 宕机时可能丢失未同步的数据）。生产中常用 Lazy + Primary Copy（主从异步复制）作为默认策略。

</details>

**8.** Apache Spark 相比 Hadoop MapReduce 性能更好的核心原因是？

- A. Spark 用 Java 编写，比 MapReduce 的 Python 更快
- B. Spark 基于内存计算（RDD 可缓存在内存中），避免了 MapReduce 每个阶段都要写磁盘的开销
- C. Spark 只能运行在更好的硬件上
- D. Spark 自动优化了网络带宽

<details><summary>答案</summary>

**答案：B**

解析：Hadoop MapReduce 每个 Job 完成后将中间结果写入 HDFS（磁盘），下一个 Job 再从磁盘读取，I/O 开销巨大。**Spark RDD** 可以缓存（cache）在内存中，迭代计算（如机器学习）无需反复读写磁盘，性能通常比 MapReduce 快 10-100 倍。另外，Spark 的 DAG 优化器（Catalyst）能对整个计算图做全局优化。

</details>

## 简答题

**9.** 你有一张10亿条记录的用户行为日志表，需要支持以下两种查询：
- 查询1：`WHERE user_id = 'abc123'`（点查询）
- 查询2：`WHERE event_time BETWEEN '2024-01-01' AND '2024-01-31'`（时间范围查询）

为每种查询推荐最合适的分区策略并说明理由。

<details><summary>答案</summary>

**查询1（等值点查询 by user_id）→ Hash Partitioning on `user_id`**：
哈希函数将相同 user_id 的数据路由到固定分区，查询时只需访问该用户对应的1个分区，无需扫描其他分区，效率极高（O(1) 分区定位）。

**查询2（时间范围查询）→ Range Partitioning on `event_time`**：
按时间范围分区（如按月分区）使同一时间段数据物理相邻，范围查询只需扫描目标时间范围内的分区，大幅减少扫描量。Hash 分区会把同一时间段的数据随机分散到所有分区，范围查询时需要全扫。

**延伸**：实际系统中可以用两级分区：先按 `event_time` 做一级 Range 分区（减少时间范围查询扫描量），再对每个分区内按 `user_id` 做 Hash 子分区（加速用户点查询）。

</details>

**10.** 解释 Spark 的 Lazy Evaluation（懒惰求值）机制：什么是 Transformation？什么是 Action？这种机制有什么好处？

<details><summary>答案</summary>

**Transformation（转换）**：对 DataFrame/RDD 的懒惰操作，调用时不立即执行，只记录计算计划（DAG）。例：`filter()`、`select()`、`groupBy()`、`map()`、`join()`。

**Action（动作）**：触发实际计算并返回结果的操作。例：`count()`、`show()`、`collect()`、`write()`、`save()`。

**好处**：
1. **全局优化**：Spark 的 Catalyst Optimizer 在执行前看到完整的计算计划，可以做谓词下推（把 filter 尽早执行）、列裁剪等优化
2. **减少冗余计算**：合并多个连续 Transformation 为一次 Pass，避免生成不必要的中间结果
3. **按需计算**：只有真正需要结果时才执行（如调试时只看 schema 不需要跑全量数据）

</details>

**11.** 解释 HDFS 的读取流程（Client 如何从 HDFS 读取一个文件），以及 NameNode 的单点失效（Single Point of Failure）问题是什么、如何缓解。

<details><summary>答案</summary>

**HDFS 读取流程**：
1. Client 向 **NameNode** 发起读取请求（文件路径）
2. NameNode 返回该文件的**块列表及每块的 DataNode 位置**
3. Client 缓存这些信息，然后直接联系**最近的 DataNode**（按网络拓扑选最近副本）读取数据块
4. DataNode 将数据块发送给 Client

**NameNode 单点失效（SPOF）问题**：
NameNode 存储所有元数据，若宕机则整个 HDFS 不可用（DataNode 上的数据还在，但无法定位）。

**缓解方式**：
- **Secondary NameNode**（早期）：定期合并编辑日志，减少 NameNode 重启时间，但不是热备（仍有停机时间）
- **NameNode HA**（高可用，现代方案）：配置两个 NameNode（Active + Standby），通过 ZooKeeper 做故障检测和自动切换（Failover），Active 宕机后 Standby 秒级接管

</details>
