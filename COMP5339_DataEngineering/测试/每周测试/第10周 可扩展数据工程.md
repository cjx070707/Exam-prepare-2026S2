**Question 1.** A retail analytics team runs the following query daily on a 10TB sales database:
```sql
SELECT region, product_category, SUM(revenue)
FROM sales
GROUP BY region, product_category;
```
The table has 50 columns, but the query only accesses 3. The DBA is evaluating switching from a row-oriented store to a **column-oriented store**. Which statement best describes the expected outcome?

- [ ] A. Column store will be slower — grouping and aggregation are only efficient in row stores
- [ ] B. Column store will use more disk space — storing each column separately creates overhead
- [x] C. Column store will be significantly faster — it reads only the 3 relevant columns and achieves higher compression on same-type data
- [ ] D. Performance will be identical — modern row stores have indexes that automatically skip irrelevant columns

> [!note]- Answer
> **C. Column Store（列式存储）在 OLAP 查询中的显著优势。**
>
> **Row Store**：数据按行连续存储，读取任意字段需要扫描整行（含 47 个无关列），I/O 浪费严重。
>
> **Column Store**：数据按列连续存储，此查询只需读取3列的磁盘块——I/O 减少约 94%（3/50）。额外优势：同一列数据类型相同（如 `revenue` 全是 float），可用 RLE、字典编码等压缩率极高（可达 10:1）。
>
> 代价：写入需更新多个列文件，OLTP 写密集型场景（如逐行 INSERT）Column Store 性能较差。典型产品：Redshift、BigQuery（列式）；PostgreSQL（行式）。

**Question 2.** A social media platform stores 500 billion user activity events. The engineering team needs to support two query patterns:
- **Query P**: `WHERE user_id = 'u_8821934'` (fetch all events for a specific user)
- **Query Q**: `WHERE event_date BETWEEN '2024-01-01' AND '2024-03-31'` (quarterly analytics)

They can only choose **one** partition key. Which statement best describes the trade-off?

- [ ] A. Partition by `user_id` (Range) — range partitioning handles both query types equally
- [ ] B. Partition by `user_id` (Hash) optimises Query P but forces a full scan for Query Q; partition by `event_date` (Range) optimises Query Q but requires scanning all date partitions for Query P — neither single choice optimises both
- [ ] C. Partition by `user_id` (Hash) — hash partitioning handles all query types equally well
- [ ] D. No partitioning is needed — modern databases automatically route queries to the right data

> [!note]- Answer
> **B. Hash vs Range Partitioning 的本质权衡。**
>
> - **Hash Partitioning on `user_id`**：哈希函数将同一 user 的数据路由到固定分区，Query P 只访问1个分区（O(1) 定位）。但 Query Q 的时间范围跨越所有 user，必须扫描全部分区（全表扫描）。
>
> - **Range Partitioning on `event_date`**：按时间范围分区，Query Q 只扫描 Q1 2024 对应的分区。但 Query P 需要扫描所有时间分区才能找到某个 user 的全部记录。
>
> **最优解（延伸）**：两级分区——先按 `event_date` Range 分区（优化 Query Q），再在每个分区内按 `user_id` Hash 子分区（优化 Query P）。

**Question 3.** A data engineer writes the following PySpark code:
```python
df = spark.read.parquet("s3://warehouse/orders/")
filtered = df.filter(df.status == "completed")
grouped = filtered.groupBy("customer_id").agg({"amount": "sum"})
print(grouped)
grouped.write.parquet("s3://output/customer_totals/")
```
Which line(s) actually trigger computation on the cluster, and what does `print(grouped)` output?

- [ ] A. All four lines trigger computation — each Spark operation runs immediately when called
- [ ] B. Only `grouped.write.parquet(...)` triggers computation; `print(grouped)` outputs only the schema/plan description, not actual data
- [ ] C. `filtered = df.filter(...)` triggers computation because filtering must scan data immediately
- [x] D. Both `print(grouped)` and `grouped.write.parquet(...)` trigger computation and scan the data

> [!note]- Answer
> **B. Spark Lazy Evaluation（懒惰求值）：Transformation vs Action。**
>
> - **Transformation（懒惰）**：`filter()`、`groupBy()`、`agg()` — 仅记录计算计划（DAG），不执行任何实际计算，不读取 S3 数据。
> - **Action（触发执行）**：`write.parquet(...)` — 触发 Spark 执行整个 DAG，从 S3 读取数据，执行过滤、聚合，写出结果。
>
> `print(grouped)` 打印的是 Spark DataFrame 的**字符串描述**（schema 信息），不触发任何计算，不显示实际数据行。要查看数据需调用 Action：`grouped.show()`。
>
> **Lazy Evaluation 的价值**：Catalyst Optimizer 在执行前看到完整 DAG，可做谓词下推（将 filter 推到数据读取层，只读必要的 Parquet row groups）等全局优化。

**Question 4.** A global fintech company replicates its PostgreSQL primary to 3 read replicas across different regions. The Asia-Pacific replica has a 200ms network round-trip. A consultant points out: *"With synchronous replication, every write commit globally will take at least 200ms."* Which statement correctly describes the trade-off?

- [ ] A. Synchronous replication is always preferred in fintech — consistency always outweighs latency
- [ ] B. Asynchronous replication is always preferred — 200ms latency makes synchronous replication unusable
- [x] C. Synchronous replication guarantees strong consistency (no data loss on failover) at the cost of write latency; Asynchronous replication has lower latency but risks losing recently committed data if the primary fails before replication completes
- [ ] D. Both approaches have identical data loss risk — replication method only affects latency

> [!note]- Answer
> **C. Synchronous vs Asynchronous Replication 的核心权衡。**
>
> | | Synchronous（同步） | Asynchronous（异步） |
> |--|--|--|
> | **一致性** | 强一致（所有副本确认后才 commit） | 最终一致（Primary 先 commit，副本异步跟进） |
> | **写延迟** | 高（受最慢副本影响，如 +200ms） | 低（只等 Primary 本地写入） |
> | **数据丢失风险** | 无（Primary 宕机时所有副本已是最新） | 有（Primary 宕机时未同步的事务丢失） |
>
> 实际方案：许多系统采用**混合策略**——至少1个本地副本同步复制（保证同数据中心容灾），跨地区副本异步复制（避免跨洲延迟）。PostgreSQL 支持 `synchronous_standby_names` 精细控制。

**Question 5.** A financial analytics startup currently runs all queries on a single PostgreSQL server with an 80TB dataset. The CTO is evaluating two scaling options:
- **Option A**: Replace the existing server with a single 256-core, 8TB-RAM bare-metal machine
- **Option B**: Deploy 50 commodity servers (16 cores, 128GB RAM each) in a **Shared-Nothing** cluster

Which statement best describes the trade-off?

- [ ] A. Option A (Scale-Up) is always preferred for financial systems — data must reside on a single node for ACID compliance
- [ ] B. Option A (Scale-Up) provides linear performance gains — doubling CPU cores always doubles query throughput
- [ ] C. Option A (Scale-Up) hits a hardware ceiling and creates a Single Point of Failure; Option B (Scale-Out) is horizontally scalable and fault-tolerant, but introduces distributed coordination complexity (network overhead, distributed transactions)
- [ ] D. Option B (Scale-Out) requires a shared storage layer — all 50 nodes must access the same disk array, creating an I/O bottleneck

> [!note]- Answer
> **C. Scale-Up vs Scale-Out（纵向扩展 vs 横向扩展）的核心权衡。**
>
> - **A 错误**：Scale-Up 与 ACID 合规无关——分布式数据库（如 CockroachDB）同样支持分布式 ACID 事务。
> - **B 错误**：Scale-Up 存在收益递减（Diminishing Returns）——受内存总线、I/O 带宽等硬件瓶颈限制，翻倍 CPU 不等于翻倍吞吐。
> - **C 正确**：
>   - **Scale-Up**：简单（无分布式复杂性），但单机硬件有上限（如内存最大 24TB）且整机是 SPOF。
>   - **Scale-Out（Shared-Nothing）**：每节点独立存储和计算，水平线性扩展，任何节点失效不影响整体。代价：需要分布式查询协调（网络传输、分布式事务、数据分区策略）。
> - **D 错误**：描述的是 **Shared-Disk** 架构（如旧式 Oracle RAC），Shared-Nothing 架构中每个节点有独立本地存储，**没有**共享磁盘阵列。

**Scenario 1:** A data engineer implements a **Word Count** job on a 500GB text corpus stored in HDFS using **MapReduce**. The corpus contains millions of lines, each with multiple words.

**Question 6:** **(6 Marks)**

**(a)** Describe the **Map phase** and **Reduce phase** for this Word Count job. What does each Mapper output, and what does each Reducer receive and produce? **(3 Marks)**

**(b)** Between Map and Reduce, the framework automatically runs a **Shuffle and Sort** phase. Explain what it does and why it is necessary. **(2 Marks)**

**(c)** The same job is rewritten in **Apache Spark** with the intermediate RDD cached. The engineer notices the second run completes 15× faster. Why? **(1 Mark)**

> [!note]- Answer
> **(a) Map 和 Reduce 阶段**
>
> **Map Phase（映射）**：
> - 输入：每行文本，例如 `"data engineering is fun data"`
> - 每个 Mapper 对每个词输出一个 `(key, value)` 对：
>   ```
>   ("data", 1), ("engineering", 1), ("is", 1), ("fun", 1), ("data", 1)
>   ```
> - Mapper 独立并行处理，数据本地化（在 DataNode 上直接运行）
>
> **Reduce Phase（归约）**：
> - 输入：同一 key 的所有 value 列表，例如 `("data", [1, 1, 1, ...])`
> - Reducer 对每个 key 的 value 列表求和：`("data", 42)`
> - 输出：每个词的最终词频
>
> **(b) Shuffle and Sort**
>
> **Shuffle**：将所有 Mapper 的输出按 key 分组，相同 key 的所有 `(k, v)` 对通过哈希路由到同一 Reducer（`hash(key) % num_reducers`）。
>
> **Sort**：每个 Reducer 收到的数据按 key 排序，确保同一 key 的所有 value 连续出现。
>
> **必要性**：MapReduce 的 Reducer 需要处理同一 key 的所有 value——没有 Shuffle，各 Mapper 的 `("data", 1)` 就无法汇集到同一个地方求和。
>
> **(c) Spark in-memory caching**
>
> Spark 的 RDD `cache()` 将第一次计算结果存入**内存**。第二次运行时直接从内存读取，跳过了从 HDFS 读取 500GB + 全部计算的过程。MapReduce 每次都从 HDFS 读取原始数据（磁盘 I/O），无内存缓存机制。

**Scenario 2:** A startup's HDFS cluster has 1 NameNode and 20 DataNodes. Each file is stored in **64MB** blocks (HDFS default) with Replication Factor = 3. A 128MB file is stored as 2 blocks.

**Question 7:** **(6 Marks)**

**(a)** Describe step-by-step how a client reads this 128MB file from HDFS. Who does the client communicate with, and in what order? **(2 Marks)**

**(b)** The operations engineer warns: *"If the NameNode crashes, the entire cluster becomes unavailable even though all 20 DataNodes are still running."* Why is this true, and name one production solution to mitigate this Single Point of Failure. **(2 Marks)**

> 📌 **新增**

**(c)** During a network partition, the HDFS cluster halts all write operations rather than allowing DataNodes to continue accepting writes independently. The team's philosophy is: *"We would rather be unavailable than serve inconsistent data."* Which two properties of the **CAP theorem** does HDFS prioritise? Which property does it sacrifice, and how does this contrast with a system like Apache Cassandra? **(2 Marks)**

> [!note]- Answer
> **(a) HDFS 读取流程（128MB 文件 = 2 个 64MB 块）**
>
> 1. **Client → NameNode**：发送读取请求（文件路径）
> 2. **NameNode → Client**：返回该文件的块列表（Block 1, Block 2）及每块的 DataNode 副本位置
> 3. **Client 缓存**：客户端缓存块位置信息
> 4. **Client → DataNode（Block 1）**：根据网络拓扑选择最近副本所在 DataNode，建立 TCP 连接读取 Block 1
> 5. **Client → DataNode（Block 2）**：同理读取 Block 2（可能是不同的 DataNode）；两个块合并还原完整文件；整个过程 NameNode **不参与**数据传输
>
> **(b) NameNode SPOF 及解决方案**
>
> **为什么 NameNode 宕机导致整个集群不可用**：NameNode 是唯一存储**文件系统命名空间和块位置映射**（文件路径 → DataNode 地址）的节点。DataNode 上数据还在，但没有 NameNode，客户端无法知道任何文件存在哪些 DataNode，数据无法访问。
>
> **生产解决方案：NameNode High Availability（高可用）**
> - 部署 **Active NameNode + Standby NameNode** 一对
> - 两者通过共享的 **JournalNode 集群**同步 edit log
> - **ZooKeeper** 持续监控 Active NameNode 心跳，Active 宕机后自动触发 Failover，Standby 秒级接管，无需人工干预
>
> **(c) CAP 定理 — HDFS 的选择**
>
> **CAP 定理**：分布式系统在网络分区（P）发生时，只能从一致性（C）和可用性（A）中选择其一。
>
> - **HDFS 选择 CP（Consistency + Partition Tolerance）**：网络分区时停止写入，确保所有节点数据一致，避免"脑裂"导致不同 DataNode 保存不同版本的块。代价：牺牲 **Availability**——集群在分区期间拒绝服务（宁可不可用，也不服务错误数据）。
>
> - **Apache Cassandra 选择 AP（Availability + Partition Tolerance）**：网络分区时各节点继续接受读写（仍然可用），但不同节点可能暂时保存不同版本的数据，通过最终一致性（Eventual Consistency）和读修复（Read Repair）逐步同步。代价：牺牲 **Consistency**——客户端可能读到过期数据（Stale Read）。
>
> | 系统 | CAP 选择 | 分区时行为 | 牺牲 |
> |------|----------|-----------|------|
> | HDFS | **CP** | 停止写入，保证一致 | Availability |
> | Cassandra | **AP** | 继续服务，数据最终一致 | Consistency |

**Question 8.** A user updates their profile photo on a social media platform. Five seconds later, they refresh their profile page and see the old photo. The engineering team uses **asynchronous (lazy) replication** with read requests load-balanced across all replicas. What is the most likely cause?

- [ ] A. The NameNode failed to propagate the write to all DataNodes
- [ ] B. The user's read request was routed to a replica that had not yet received the update — a "Read Your Own Write" consistency violation caused by lazy replication lag
- [ ] C. The primary copy rejected the write because of a conflict with another leader
- [ ] D. The platform uses CP semantics, which blocks reads until all replicas are synchronised

> [!note]- Answer
> **B. "Read Your Own Write" — Lazy Replication 的典型一致性问题。**
>
> - **Lazy（异步）Replication**：Primary 接受写入后立即返回成功；副本在后台异步同步，存在短暂延迟。
> - **"Read Your Own Write" 违反**：用户写入 Primary 后，若读请求被路由到还未同步的副本，会读到旧值——即看不到自己刚写入的内容。
>
> - **A 错误**：NameNode/DataNode 是 HDFS 概念，与社交平台的 Profile 复制无关。
> - **C 错误**：Primary Copy 架构没有多 Leader，不存在 Leader 冲突。
> - **D 错误**：CP 语义下读操作仍然可以正常响应（停写不停读）；此场景是读到旧数据，非系统不可用。
>
> **解决方案**：写入后短期内将该用户的读请求路由回 Primary（Sticky Session），或在客户端本地缓存最新写入值。
