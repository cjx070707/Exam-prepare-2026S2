## Multiple Choice Questions

**1.** A retail company uses one system to process real-time point-of-sale transactions (thousands of writes per second), and a separate system for data analysts to run monthly sales trend reports. Which architectural pattern do these two systems represent, respectively?

- A. Both are OLAP, just with different data volumes
- B. The first is OLAP, the second is OLTP
- C. The first is OLTP, the second is OLAP
- D. The first is OLTP, the second is also OLTP — the only difference is concurrency support

> [!tip]- Answer
> **答案：C**
> OLTP（Online Transaction Processing）针对高频、低延迟的点查询与写入，强调 ACID 事务，每次操作影响少量行。OLAP（Online Analytical Processing）针对跨大量历史记录的聚合扫描（如 SUM、GROUP BY），读多写少，查询通常扫描数百万行。两者分开部署的根本原因是：分析查询若跑在 OLTP 系统上，会严重拖慢交易响应。

**2.** In a Star Schema, which of the following statements about Fact Tables and Dimension Tables is correct?

- A. The Fact Table stores descriptive attributes such as product names and customer cities
- B. Dimension Tables store aggregatable measures such as sales amount and purchase quantity
- C. The Fact Table stores measures and foreign keys (FKs) pointing to each Dimension Table
- D. A Star Schema only allows one Dimension Table; multiple dimensions must be merged into a single table

> [!tip]- Answer
> **答案：C**
> Fact Table 是 Star Schema 的核心，存储可被聚合的数值（sales_amount、quantity）以及指向各维度的外键（time_id、product_id、market_id）。Dimension Table 存储描述性属性（产品类别、客户所在地区、时间的年月季度），用于 GROUP BY 和过滤，不含度量指标。"Star"命名来自形状：中心 Fact 表，四周 Dimension 表辐射排列。

**3.** What is the fundamental difference between ETL and ELT?

- A. ETL uses relational databases; ELT uses NoSQL
- B. ETL transforms data in an external tool before loading; ELT loads raw data into the target system first, then transforms it inside that system
- C. ELT skips the Extract step and transforms data directly inside the source system
- D. ETL is only suitable for batch processing; ELT is only suitable for real-time stream processing

> [!tip]- Answer
> **答案：B**
> ETL（Extract → Transform → Load）：在专用 ETL 工具（如 Informatica）中完成转换，载入的是干净数据。ELT（Extract → Load → Transform）：先将原始数据载入数据湖或云数仓，再利用目标系统的计算能力（如 BigQuery SQL、Spark）做转换。ELT 更适合现代云数仓，因为云平台算力弹性且成本低，且保留了原始数据供后续重新处理。

**4.** Which of the following best describes the pain points of traditional ETL that have driven the industry toward ELT?

- A. ETL cannot process data from relational databases, so ELT is needed as a replacement
- B. ETL transformation logic is scattered across BI tools, hard to version-control, and excludes analysts from the data preparation process
- C. ETL has lower performance than ELT because ETL lacks parallel computing capability
- D. ETL does not support schema-on-read, so it cannot handle unstructured data

> [!tip]- Answer
> **答案：B**
> 传统 ETL 三大痛点：①转换逻辑散落在各 BI 工具（如 Tableau、Excel）中，难以统一管理和版本控制；②业务逻辑过重压在 BI 层，维护成本高；③分析师被排除在外，数据准备完全依赖工程师。ELT 将转换下沉到数仓内部（用 SQL/dbt），分析师可直接参与转换逻辑，且代码进入版本控制。

**5.** What is the difference between Schema-on-write and Schema-on-read?

- A. Schema-on-write validates and enforces the schema at write time; Schema-on-read applies the schema only at query time, with data stored in its raw format
- B. Schema-on-read means data has no schema at all — it can be stored in any format without any structural definition
- C. Schema-on-write is used for data lakes; Schema-on-read is used for data warehouses
- D. The only difference is performance: Schema-on-write is slower, Schema-on-read is faster

> [!tip]- Answer
> **答案：A**
> Schema-on-write（数据仓库）：写入前 schema 已定义，数据必须符合格式才能存入，查询时无需额外解析，性能好但灵活性低。Schema-on-read（数据湖）：数据以原始格式存储（CSV、JSON、Parquet），读取时才应用 schema 解析，灵活但查询有解析开销，数据质量风险较高。注意：Schema-on-read 不是"无 schema"，schema 仍然存在，只是延迟到查询时应用。

**6.** Which of the following correctly matches an OLAP operation name with its description?

- A. Roll Up: fix a single value on one dimension to reduce a multi-dimensional cube to a lower-dimensional subset
- B. Drill Down: aggregate data at a coarser granularity, e.g., from "month" up to "year"
- C. Slice: select a single value along one dimension, reducing an N-dimensional dataset to (N-1) dimensions
- D. Dice: select a range along a single dimension, keeping the number of dimensions unchanged

> [!tip]- Answer
> **答案：C**
> OLAP 五种操作：**Roll Up** = 向上聚合（城市→省份→国家，粒度变粗）；**Drill Down** = 向下细化（年→季度→月，粒度变细）；**Slice** = 锁定 1 个维度的单个值，降维（如只看"2024年"）；**Dice** = 锁定多个维度的范围，不降维（如"2024年 Q1-Q2、北区和南区"）；**Pivot** = 旋转维度视角，改变行列布局。A 描述的是 Slice，B 描述的是 Roll Up，D 描述的是 Dice（但描述不准确，Dice 是多维范围过滤）。

**7.** Which of the following best describes the greatest risk associated with a Data Lake?

- A. Data Lakes are too expensive to store large volumes of raw data
- B. Data Lakes lack transaction support, making data loss during writes likely
- C. A Data Lake can degrade into a Data Swamp: data accumulates without metadata or governance, and becomes impossible for anyone to find or use
- D. Data Lakes do not support unstructured data — they can only store CSV and JSON

> [!tip]- Answer
> **答案：C**
> Data Swamp（数据沼泽）是 Data Lake 最常见的失败形态：数据只进不出（write-only），缺乏数据目录（Data Catalog）、无元数据标注、无访问权限管理、无质量检查，最终数据"存在但无人能用"。应对方法：建立 Data Catalog、强制元数据注册、实施 Data Governance、定期数据质量审计。

**8.** A company has three separate OLTP databases — an order system, an inventory system, and a CRM system — that operate independently with no data sharing. Analysts cannot perform cross-system analysis of customer purchasing behaviour. What is the technical term for this problem?

- A. Schema Mismatch
- B. Data Silo
- C. Data Swamp
- D. Missing OLAP Layer

> [!tip]- Answer
> **答案：B**
> Data Silo（数据孤岛）指多个系统各自独立存储数据、互不共享，导致无法进行跨系统的整合分析。这是建设数据仓库的核心动机之一：通过 ETL/ELT 将多个 OLTP 系统的数据集中到数仓，打破孤岛，支持全局分析视图。

## Short Answer Questions

**9.** Compare Data Lake and Data Warehouse across four dimensions: data types, schema strategy, use cases, and primary risks.

> [!tip]- Answer
> | Dimension | Data Warehouse | Data Lake |
> |-----------|----------------|-----------|
> | Data Types | Primarily structured data | Structured + semi-structured + unstructured (images, logs, JSON, etc.) |
> | Schema Strategy | Schema-on-write: schema enforced before writing | Schema-on-read: schema applied at query time |
> | Use Cases | Fixed reports, BI analytics, known query patterns | Raw data archiving, data science exploration, unknown future use |
> | Primary Risk | Low flexibility; schema changes are costly | Data Swamp: lack of governance makes data unusable |
>
> **Data Lakehouse** 是两者的折中：在 Data Lake 的存储层上加入 Metadata 管理与事务支持（Delta Lake、Apache Iceberg），获得 Lake 的灵活性与 Warehouse 的可管理性。

---

**10.** Explain the design motivation behind the Data Lakehouse, and describe how it addresses the core deficiencies of the Data Lake.

> [!tip]- Answer
> **动机**：Data Lake 存储灵活、成本低，但缺乏 schema 强制、事务支持和数据治理，容易退化为 Data Swamp；Data Warehouse 查询性能好、治理完善，但灵活性低、扩展成本高。两者各有短板。
>
> **Lakehouse 的做法**：在 Data Lake 的对象存储（S3、ADLS）之上，叠加一层 Metadata 与 Governance 机制：
> - **Delta Lake / Apache Iceberg**：提供 ACID 事务、schema 强制、Time Travel（时间旅行）、数据版本管理
> - **统一存储**：原始数据和处理后数据共用同一存储层，避免数据复制
> - **结果**：保留 Lake 的低成本和多格式支持，获得 Warehouse 级别的可靠性和查询性能
>
> **核心缺陷的解决**：Data Swamp 的根源是"无元数据、无治理"，Lakehouse 通过强制 schema 注册和事务日志直接解决这一问题。

**11.** Explain why a DBMS (Database Management System) offers advantages over storing data directly in a file system. List at least three aspects.

> [!tip]- Answer
> DBMS 相比文件方式的核心优势：
>
> 1. **统一共享访问（Concurrent Access）**：多个应用和用户可并发访问同一数据，DBMS 通过并发控制（锁、事务隔离）防止数据不一致；文件方式难以协调多用户同时读写。
>
> 2. **完整性约束（Integrity Constraints）**：DBMS 强制数据规则（Primary Key 唯一且非 NULL、Foreign Key 引用完整性、CHECK 约束），文件存储无内置机制阻止错误数据写入。
>
> 3. **安全与权限管理（Access Control）**：DBMS 提供细粒度的访问控制（用户/角色级别的 SELECT、INSERT 权限），文件系统权限粒度粗，难以做到"只能读某几列"。
>
> 4. **标准化查询接口（SQL）**：SQL 提供统一的声明式查询语言，文件存储需要每个应用自己编写解析逻辑，重复且易出错。
>
> **Primary Key 的约束**：唯一（每行值不同）且不允许为 NULL，保证每条记录可被精确定位，是关系型数据库完整性的基础。
