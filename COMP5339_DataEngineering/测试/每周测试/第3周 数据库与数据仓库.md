# 第3周测试 — 数据库、数据仓库与数据湖

---

## 选择题

**1.** 一家零售公司使用一套系统处理收银台的实时交易（每秒数千次写入），另一套系统供数据分析师每月跑销售趋势报表。这两套系统分别对应哪种架构？

- A. 两套都是 OLAP，只是数据量不同
- B. 前者 OLAP，后者 OLTP
- C. 前者 OLTP，后者 OLAP
- D. 前者 OLTP，后者也是 OLTP，区别在于是否支持并发

> [!tip]- 答案
> **答案：C**
> 解析：OLTP（Online Transaction Processing）针对高频、低延迟的点查询与写入，强调 ACID 事务，每次操作影响少量行。OLAP（Online Analytical Processing）针对跨大量历史记录的聚合扫描（如 SUM、GROUP BY），读多写少，查询通常扫描数百万行。两者分开部署的根本原因是：分析查询若跑在 OLTP 系统上，会严重拖慢交易响应。

**2.** 在 Star Schema 中，以下关于 Fact Table 与 Dimension Table 的描述，哪项正确？

- A. Fact Table 存储产品名称、客户城市等描述性属性
- B. Dimension Table 存储销售金额、购买数量等可聚合的度量指标
- C. Fact Table 存储度量指标（Measures）和指向各 Dimension Table 的外键（FK）
- D. Star Schema 只允许一个 Dimension Table，多个维度须合并存储

> [!tip]- 答案
> **答案：C**
> 解析：Fact Table 是 Star Schema 的核心，存储可被聚合的数值（sales_amount、quantity）以及指向各维度的外键（time_id、product_id、market_id）。Dimension Table 存储描述性属性（产品类别、客户所在地区、时间的年月季度），用于 GROUP BY 和过滤，不含度量指标。"Star"命名来自形状：中心 Fact 表，四周 Dimension 表辐射排列。

**3.** ETL 与 ELT 的根本区别在于？

- A. ETL 使用关系型数据库，ELT 使用 NoSQL
- B. ETL 在外部工具中先转换再载入；ELT 先将原始数据载入目标系统，转换在目标系统内完成
- C. ELT 跳过了 Extract 步骤，直接在源系统内转换
- D. ETL 只适用于批处理，ELT 只适用于实时流处理

> [!tip]- 答案
> **答案：B**
> 解析：ETL（Extract → Transform → Load）：在专用 ETL 工具（如 Informatica）中完成转换，载入的是干净数据。ELT（Extract → Load → Transform）：先将原始数据载入数据湖或云数仓，再利用目标系统的计算能力（如 BigQuery SQL、Spark）做转换。ELT 更适合现代云数仓，因为云平台算力弹性且成本低，且保留了原始数据供后续重新处理。

**4.** 传统 ETL 模式存在哪些痛点，促使行业转向 ELT？以下哪项描述最准确？

- A. ETL 无法处理关系型数据库的数据，因此需要 ELT 替代
- B. ETL 转换逻辑分散在 BI 工具中、版本控制困难、分析师被排除在数据准备流程之外
- C. ETL 的性能比 ELT 低，因为 ETL 缺乏并行计算能力
- D. ETL 不支持 schema-on-read，因此无法处理非结构化数据

> [!tip]- 答案
> **答案：B**
> 解析：传统 ETL 三大痛点：①转换逻辑散落在各 BI 工具（如 Tableau、Excel）中，难以统一管理和版本控制；②业务逻辑过重压在 BI 层，维护成本高；③分析师被排除在外，数据准备完全依赖工程师。ELT 将转换下沉到数仓内部（用 SQL/dbt），分析师可直接参与转换逻辑，且代码进入版本控制。

**5.** Schema-on-write 与 Schema-on-read 的区别是？

- A. Schema-on-write 在写入时验证并强制 schema；Schema-on-read 在读取时才解析 schema，数据以原始格式存储
- B. Schema-on-read 表示数据没有 schema，可以存储任意格式而无需任何结构定义
- C. Schema-on-write 用于数据湖，Schema-on-read 用于数据仓库
- D. 两者区别仅在于性能：Schema-on-write 更慢，Schema-on-read 更快

> [!tip]- 答案
> **答案：A**
> 解析：Schema-on-write（数据仓库）：写入前 schema 已定义，数据必须符合格式才能存入，查询时无需额外解析，性能好但灵活性低。Schema-on-read（数据湖）：数据以原始格式存储（CSV、JSON、Parquet），读取时才应用 schema 解析，灵活但查询有解析开销，数据质量风险较高。注意：Schema-on-read 不是"无 schema"，schema 仍然存在，只是延迟到查询时应用。

**6.** 以下哪个 OLAP 操作描述与名称对应正确？

- A. Roll Up：在某一维度上固定一个特定值，将多维数据集缩减为更低维的子集
- B. Drill Down：将数据按更粗的粒度聚合，例如从"月"汇总到"年"
- C. Slice：沿一个维度选取单个值，将 N 维数据集缩减为 (N-1) 维
- D. Dice：沿单个维度选取一个范围，保持维度数量不变

> [!tip]- 答案
> **答案：C**
> 解析：OLAP 五种操作：**Roll Up** = 向上聚合（城市→省份→国家，粒度变粗）；**Drill Down** = 向下细化（年→季度→月，粒度变细）；**Slice** = 锁定 1 个维度的单个值，降维（如只看"2024年"）；**Dice** = 锁定多个维度的范围，不降维（如"2024年 Q1-Q2、北区和南区"）；**Pivot** = 旋转维度视角，改变行列布局。A 描述的是 Slice，B 描述的是 Roll Up，D 描述的是 Dice（但描述不准确，Dice 是多维范围过滤）。

**7.** 以下关于 Data Lake 最大风险的描述，哪项最准确？

- A. Data Lake 存储成本太高，不适合大规模原始数据
- B. Data Lake 缺乏事务支持，导致数据写入时容易丢失
- C. Data Lake 容易退化为 Data Swamp：数据无元数据、无治理，堆积后无人能找到或使用
- D. Data Lake 不支持非结构化数据，只能存储 CSV 和 JSON

> [!tip]- 答案
> **答案：C**
> 解析：Data Swamp（数据沼泽）是 Data Lake 最常见的失败形态：数据只进不出（write-only），缺乏数据目录（Data Catalog）、无元数据标注、无访问权限管理、无质量检查，最终数据"存在但无人能用"。应对方法：建立 Data Catalog、强制元数据注册、实施 Data Governance、定期数据质量审计。

**8.** 一家企业有订单系统、库存系统、CRM 系统三套独立的 OLTP 数据库，各系统互不相通，数据分析师无法做跨系统的客户购买行为分析。这个问题的术语是？

- A. Schema 不匹配（Schema Mismatch）
- B. Data Silo（数据孤岛）
- C. Data Swamp（数据沼泽）
- D. OLAP 缺失（Missing OLAP Layer）

> [!tip]- 答案
> **答案：B**
> 解析：Data Silo（数据孤岛）指多个系统各自独立存储数据、互不共享，导致无法进行跨系统的整合分析。这是建设数据仓库的核心动机之一：通过 ETL/ELT 将多个 OLTP 系统的数据集中到数仓，打破孤岛，支持全局分析视图。

## 简答题

**9.** 比较 Data Lake 与 Data Warehouse，从数据类型、Schema 策略、适用场景和主要风险四个维度说明各自的特点。

> [!tip]- 答案
> | 维度 | Data Warehouse | Data Lake |
> |------|----------------|-----------|
> | 数据类型 | 结构化数据为主 | 结构化 + 半结构化 + 非结构化（图片、日志、JSON 等）|
> | Schema 策略 | Schema-on-write：写入前强制 schema | Schema-on-read：读取时才应用 schema |
> | 适用场景 | 固定报表、BI 分析、已知查询模式 | 原始数据归档、数据科学探索、未来用途不确定 |
> | 主要风险 | 灵活性低，schema 变更成本高 | Data Swamp：无治理导致数据无法被使用 |
>
> **Data Lakehouse** 是两者的折中：在 Data Lake 的存储层上加入 Metadata 管理与事务支持（Delta Lake、Apache Iceberg），获得 Lake 的灵活性与 Warehouse 的可管理性。

---

**10.** 解释 Data Lakehouse 的设计动机，并说明它如何解决 Data Lake 的核心缺陷。

> [!tip]- 答案
> **动机**：Data Lake 存储灵活、成本低，但缺乏 schema 强制、事务支持和数据治理，容易退化为 Data Swamp；Data Warehouse 查询性能好、治理完善，但灵活性低、扩展成本高。两者各有短板。
>
> **Lakehouse 的做法**：在 Data Lake 的对象存储（S3、ADLS）之上，叠加一层 Metadata 与 Governance 机制：
> - **Delta Lake / Apache Iceberg**：提供 ACID 事务、schema 强制、时间旅行（Time Travel）、数据版本管理
> - **统一存储**：原始数据和处理后数据共用同一存储层，避免数据复制
> - **结果**：保留 Lake 的低成本和多格式支持，获得 Warehouse 级别的可靠性和查询性能
>
> **核心缺陷的解决**：Data Swamp 的根源是"无元数据、无治理"，Lakehouse 通过强制 schema 注册和事务日志直接解决这一问题。

**11.** 解释为什么 DBMS（数据库管理系统）相比直接用文件系统存储数据具有优势。列举至少三个方面。

> [!tip]- 答案
> DBMS 相比文件方式的核心优势：
>
> 1. **统一共享访问**：多个应用和用户可并发访问同一数据，DBMS 通过并发控制（锁、事务隔离）防止数据不一致；文件方式难以协调多用户同时读写。
>
> 2. **完整性约束**：DBMS 强制数据规则（Primary Key 唯一且非 NULL、Foreign Key 引用完整性、CHECK 约束），文件存储无内置机制阻止错误数据写入。
>
> 3. **安全与权限管理**：DBMS 提供细粒度的访问控制（用户/角色级别的 SELECT、INSERT 权限），文件系统权限粒度粗，难以做到"只能读某几列"。
>
> 4. **标准化查询接口**：SQL 提供统一的声明式查询语言，文件存储需要每个应用自己编写解析逻辑，重复且易出错。
>
> **Primary Key 的约束**：唯一（每行值不同）且不允许为 NULL，保证每条记录可被精确定位，是关系型数据库完整性的基础。
