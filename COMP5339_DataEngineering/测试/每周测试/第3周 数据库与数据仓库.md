## 选择题

**1.** OLTP 和 OLAP 的主要区别是？

- A. OLTP 用于历史分析，OLAP 用于事务处理
- B. OLTP 针对高频点查询和写入，OLAP 针对聚合扫描和历史分析
- C. OLTP 数据量比 OLAP 更大
- D. 两者没有本质区别，只是规模不同

> [!tip]- 答案
>
> **答案：B**
>
> 解析：OLTP（Online Transaction Processing）优化高频、低延迟的增删改查（INSERT/UPDATE/DELETE），需要 ACID 事务保证。OLAP（Online Analytical Processing）优化跨大量历史数据的聚合扫描（SUM、COUNT、GROUP BY），读多写少，对延迟容忍度高。
>

**2.** Star Schema 中，Fact Table 的作用是？

- A. 描述业务维度（如时间、地点）
- B. 存储可度量的业务事件，包含各维度的外键和度量指标
- C. 存储原始未处理的数据
- D. 记录 ETL 操作日志

> [!tip]- 答案
>
> **答案：B**
>
> 解析：Fact Table 存储业务事件（如一笔销售），包含度量指标（sales_amount、quantity）和指向各 Dimension Table 的外键（time_id、product_id、market_id）。Dimension Table 存储描述性属性（如时间的年月日、产品的名称类别等）。
>

**3.** Data Lake 相比 Data Warehouse 的核心优势是？

- A. 查询速度更快，适合复杂聚合
- B. 数据质量更高，有严格的质量控制
- C. 支持 schema-on-read，可存储任意格式的原始数据
- D. 维护成本更低，不需要专人管理

> [!tip]- 答案
>
> **答案：C**
>
> 解析：Data Lake 的核心优势是 schema-on-read：数据以原始格式存储（CSV、JSON、图片等），读取时才应用 schema 解析。这让它能接受未来用途不确定的异构数据。Data Warehouse 是 schema-on-write，写入前必须符合预定义的 schema，查询性能更好但灵活性低。
>

**4.** ELT 与 ETL 的主要区别是？

- A. ELT 先转换再载入，ETL 先载入再转换
- B. ETL 先转换再载入，ELT 先载入再转换（在目标系统内转换）
- C. 两者顺序相同，只是工具不同
- D. ELT 不需要 Extract 步骤

> [!tip]- 答案
>
> **答案：B**
>
> 解析：**ETL**：Extract → Transform（在外部工具中转换）→ Load（载入仓库）。**ELT**：Extract → Load（先载入数据湖/仓库）→ Transform（利用目标系统的计算能力转换）。ELT 更适合现代云数仓（如 BigQuery、Snowflake），因为云平台算力便宜，可在仓库内做大规模转换。
>

**5.** 以下哪种情况最适合使用 Data Lake 而非 Data Warehouse？

- A. 需要对历史销售数据做固定格式的月度报表
- B. 需要存储混合了文本、图片和传感器数据，未来用途不确定
- C. 业务分析师需要快速做复杂聚合查询
- D. 需要强一致性的事务处理

> [!tip]- 答案
>
> **答案：B**
>
> 解析：Data Lake 适合：数据格式多样（结构化+非结构化）、用途暂不确定、需要长期保留原始数据。A 和 C 更适合 OLAP / Data Warehouse（结构化、固定 schema、高性能聚合）。D 属于 OLTP 的需求。
>

**6.** 以下 SQL 语句存在什么安全问题？

```python
cursor.execute("SELECT * FROM Sales WHERE region = " + region)
```

- A. 查询语法错误，无法执行
- B. SQL 注入漏洞，攻击者可以通过构造恶意 region 值篡改或泄露数据
- C. 性能问题，应该用存储过程
- D. 没有问题，字符串拼接是标准做法

> [!tip]- 答案
>
> **答案：B**
>
> 解析：字符串拼接构造 SQL 是 SQL Injection 的根源。攻击者可传入如 `"'; DROP TABLE Sales; --"` 破坏数据库。正确做法是使用参数化查询：
> ```python
> cursor.execute("SELECT * FROM Sales WHERE region = %s", (region,))
> ```
> 参数化查询会将用户输入作为数据而非 SQL 代码处理。
>

**7.** 什么是 "Data Swamp"（数据沼泽）？

- A. 数据量过大导致查询变慢的数据湖
- B. 数据湖退化后的状态：数据无组织堆积、无元数据、无治理，数据无法被找到或使用
- C. 数据仓库中存储了过期数据的状态
- D. 跨多个数据湖复制数据导致的冗余问题

> [!tip]- 答案
>
> **答案：B**
>
> 解析：Data Swamp 是 Data Lake 的反面教材：缺乏数据目录（Data Catalog）、无元数据标注、无质量控制、无访问权限管理，导致数据"堆在里面但无人能用"。避免方法：建立 Data Catalog、强制元数据注册、定期质量检查、实施数据治理（Data Governance）。
>

**8.** 在 Star Schema 中，以下哪个字段应该放在 Fact Table 而不是 Dimension Table？

- A. 产品类别名称
- B. 销售金额（sales_amount）
- C. 客户所在城市
- D. 时间的年份

> [!tip]- 答案
>
> **答案：B**
>
> 解析：Fact Table 存储**度量指标**（Measures）——可被聚合的数值，如 sales_amount、quantity、revenue。Dimension Table 存储**描述性属性**（Attributes）——用于过滤和分组，如产品类别、城市、年份。
>

## 简答题

**9.** 解释 schema-on-write 和 schema-on-read 的区别，分别对应哪种数据存储系统？

> [!tip]- 答案
>
> - **Schema-on-write**：写入数据前必须定义好表结构，数据必须符合 schema 才能存入；写入时做格式验证，读取性能好，但灵活性低。对应：**Data Warehouse**（如 Snowflake、BigQuery）、关系型数据库。
> - **Schema-on-read**：数据以原始格式存储，读取时才应用 schema 解析；可存储任意格式，灵活但查询时有解析开销，数据质量风险更高。对应：**Data Lake**（如 HDFS + Parquet、Amazon S3）。
>
> **考试要点**：schema-on-read 不是"没有 schema"，而是 schema 在查询时才应用。
>

**10.** 画出一个零售场景的 Star Schema 结构，包含 Fact Table（Sales）和至少3个 Dimension Table，列出每张表的关键字段。

> [!tip]- 答案
>
> ```
> Fact Table: Sales
> ─────────────────────────────────────────
> sale_id (PK)
> time_id   (FK → Time)
> product_id (FK → Product)
> market_id  (FK → Market)
> sales_amount
> quantity
>
> Dimension: Time          Dimension: Product       Dimension: Market
> ────────────────         ────────────────          ────────────────
> time_id (PK)             product_id (PK)           market_id (PK)
> date                     name                      city
> week                     category                  state
> month                    brand                     region
> quarter                  unit_price                country
> year
> ```
>
> **要点**：Fact Table 中只存 FK 和度量指标，所有描述性字段放 Dimension Table。Star Schema 的命名来自形状：中间 Fact 表 + 四周 Dimension 表辐射状排列。
>

**11.** 你是一家电商公司的数据工程师，需要设计以下两个系统：
- 用户实时订单系统（高频写入，每秒数千条记录）
- 月度销售趋势分析（扫描数亿条历史记录）

分别应该使用 OLTP 还是 OLAP 架构？说明理由。

> [!tip]- 答案
>
> - **实时订单系统 → OLTP**：高频点查询和写入，需要低延迟和 ACID 事务保证（防止订单重复、数据不一致）。典型工具：PostgreSQL、MySQL。
> - **月度分析报表 → OLAP**：需要扫描海量历史数据做聚合（SUM、GROUP BY），读多写少，对延迟容忍度高，查询模式固定。典型工具：Snowflake、BigQuery、Redshift（列式存储更高效）。
>
> **延伸**：实际生产中，OLTP 数据通过 ETL/ELT 定期同步到 OLAP 系统，两者分开部署，避免分析查询拖慢交易系统。
>
