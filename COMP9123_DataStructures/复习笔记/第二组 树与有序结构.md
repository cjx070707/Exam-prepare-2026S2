## Part 1 — 树的基础

### 树的定义与术语 `★★`

> [!tip] 公司组织架构图中，CEO 在最上面，每个经理管几个下属——如果你要找某个员工的"级别"和他到 CEO 的距离，你怎么描述？

**树的定义**：一种表示层级关系（hierarchical structure）的抽象模型，由节点（nodes）和父子关系（parent-child relation）组成。

**形式化定义**：
- 如果树 T 非空，有一个特殊的 **root** 节点（没有 parent）
- 除 root 外，每个节点有且只有一个 parent
- 父子关系没有环（following parent always leads to root）

#### 节点分类

| 类型 | 定义 | 示例 |
|------|------|------|
| Root | 没有 parent 的节点 | A |
| Internal node | 至少有一个 child 的节点 | A, B, C, F |
| External / Leaf node | 没有 child 的节点 | E, I, J, K, G, H, D |

#### 关系术语

| 术语 | 定义 |
|------|------|
| Ancestors | parent, grandparent, great-grandparent ... |
| Descendants | child, grandchild, great-grandchild ... |
| Siblings | 共享同一 parent 的节点 |
| Lowest Common Ancestor (LCA) | x 和 y 的共同祖先中最深的那个 |

#### 度量术语

| 术语 | 定义 | 示例 |
|------|------|------|
| Depth（深度） | 从 root 到该节点的祖先个数（不含自身） | depth(root) = 0 |
| Level（层） | 相同深度的所有节点 | level 2 = {E, F, G, H} |
| Height（高度） | 树中节点的最大深度 | 叶子的 height = 0 |

#### 子结构

| 术语 | 定义 |
|------|------|
| Subtree | 以某节点为 root 的子树（包括其所有 descendants） |
| Edge | 一对有父子关系的节点 (u, v) |
| Path | 连续节点序列，相邻两节点间有边 |

**树的事实**：
- 祖先/后代关系是传递的
- 每个节点都是 root 的后代
- 任意两个节点至少有一个共同祖先（root）
- 存在节点对，互不为祖先/后代关系

**有序树（Ordered Tree）**：子节点之间有规定的顺序，图中通常以从左到右表示。

<span style="color: #9b59b6">💡 **白话理解**：树就像一个家族族谱——root 是老祖宗，每个人只有一个父亲（parent），可以有多个孩子（children）。Depth 就是"第几代"，Height 就是"这个家族最多有几代人"。</span>

### Tree ADT `★`

Tree ADT 定义的操作（使用 Position 作为节点抽象）：

| 类别 | 方法 |
|------|------|
| 通用方法 | `size()`, `isEmpty()`, `iterator()`, `positions()` |
| 访问方法 | `root()`, `parent(p)`, `children(p)`, `numChildren(p)` |
| 查询方法 | `isInternal(p)`, `isExternal(p)`, `isRoot(p)` |

Node 对象典型属性：`value`, `children`（子节点集合）, `parent`（可选）

```
def is_external(v):
    return v.children.is_empty()

def is_root(v):
    return v.parent = null
```

<span style="color: #9b59b6">💡 **白话理解**：Tree ADT 就是告诉你"一棵树能做什么"——查根、查父母、查孩子、判断是不是叶子。具体怎么存、用链表还是数组，那是 Data Structure 层面的事。</span>

<span style="color: #e67e22">*我们知道了树的定义和术语，但怎么系统地访问树中的每个节点？需要一套遍历策略。*</span>

### 树遍历 Traversals `★★★`

> [!tip] 一个文件系统需要计算每个文件夹的总大小——你必须先知道子文件夹的大小才能计算父文件夹。这是哪种遍历？

遍历（Traversal）= 按某种系统方式访问树中所有节点。树比线性结构复杂，存在多种自然的遍历方式。

#### 前序遍历 Preorder

**规则**：先访问节点自身，再遍历所有子树（有序树按规定顺序）。

```
def pre_order(v):
    visit(v)
    for each child w of v:
        pre_order(w)
```

**访问顺序示意**（树的节点值）：

```
        12
      / |  \
    14  13  20
   /  \    / \
  18  23  34  31
 / \
16  11
```

Preorder: 12, 14, 18, 16, 11, 23, 31, 13, 34, 31, 20

**应用**：公司战略决策（自上而下）、聊天机器人交互

#### 后序遍历 Postorder

**规则**：先遍历所有子树，再访问节点自身。

```
def post_order(v):
    for each child w of v:
        post_order(w)
    visit(v)
```

同一棵树的 Postorder: 18, 16, 11, 23, 31, 14, 34, 31, 13, 20, 12

**应用**：文件系统备份（先备份子文件再备份父文件夹）、计算文件夹大小、解算术表达式

#### 中序遍历 Inorder（仅适用于二叉树）

**规则**：先遍历左子树，再访问节点自身，再遍历右子树。

```
def in_order(v):
    if v.left ≠ null then
        in_order(v.left)
    visit(v)
    if v.right ≠ null then
        in_order(v.right)
```

**关键性质**：对 BST 做中序遍历，访问顺序恰好是 key 的升序排列。

**应用**：打印算术表达式（中缀表示）

```
def print_expr(v):
    if v.left ≠ null then
        print("(")
        print_expr(v.left)
    print(v.element)
    if v.right ≠ null then
        print_expr(v.right)
        print(")")
```

对表达式树 `((2 × (a − 1)) + (3 × b))` 输出中缀表达式。

<span style="color: #9b59b6">💡 **白话理解**：三种遍历的区别只在于"什么时候处理自己"。Preorder = 先处理自己再处理孩子（老板先决策，下属再执行）。Postorder = 先处理孩子再处理自己（先算子文件夹大小，再加总）。Inorder = 左-自己-右（BST 中恰好得到排序结果）。</span>

<span style="color: #e67e22">*通用树的每个节点可以有任意多个孩子。如果限制为最多两个孩子，就得到二叉树——更多性质和算法变得可用。*</span>

### 二叉树 Binary Tree `★★`

> [!tip] 一个"是/否"的诊断问卷——每个问题只有两个分支——这种结构适合用什么树？

**定义**：有序树，每个内部节点最多两个孩子（left child 和 right child）。

- **Proper（满）二叉树**：每个内部节点恰好有两个孩子
- Left subtree / Right subtree：以 left child / right child 为根的子树

**Binary Tree 扩展的 ADT 方法**：
- `leftChild(p)`, `rightChild(p)`, `sibling(p)`
- 当不存在时返回 null

**Node 对象属性**：`value`, `left`, `right`, `parent`（可选）

```
def is_external(v):
    return v.left = null and v.right = null
```

**应用**：
- **算术表达式树**：内部节点 = 运算符，外部节点 = 操作数
- **决策树**：内部节点 = 是/否问题，外部节点 = 决策结果

<span style="color: #9b59b6">💡 **白话理解**：二叉树就像一个"只能选左或右"的迷宫。限制了分支数量后，很多算法变得更简单、更高效——比如二叉搜索就是利用"左小右大"每次排除一半。</span>

### Euler Tour 遍历 `★`

Euler Tour 是二叉树的通用遍历框架：沿着树的边"走一圈"，保持树在左手边，每个节点被访问三次：
- 从**左侧**经过（对应 preorder）
- 从**下方**经过（对应 inorder）
- 从**右侧**经过（对应 postorder）

示例（树节点值 6, 10, 2, 9, 5, 1, 7, 3, 4）：
- Preorder（第一次访问）：6, 10, 2, 9, 5, 1, 7, 3, 4
- Inorder（第二次访问）：2, 10, 5, 9, 1, 6, 3, 7, 4
- Postorder（第三次访问）：2, 5, 1, 9, 10, 3, 4, 7, 6

<span style="color: #9b59b6">💡 **白话理解**：Euler Tour 就像在公园沿着小路走一圈——每棵树你都会从左边路过（preorder）、从底下路过（inorder）、从右边路过（postorder）。三种遍历其实是同一次散步的三个不同"打卡时机"。</span>

### 树的实现与递归 `★★`

#### Linked Structure 实现

**二叉树**：每个 Node 存储 Element + Parent + Left child + Right child

**通用树**：每个 Node 存储 Element + Parent + Sequence of children

#### 递归算法示例

**计算深度**（沿 parent 向上走）：
```
def depth(v):
    if v.parent = null then
        return 0
    else
        return depth(v.parent) + 1
```

**计算高度**（沿 children 向下走）：
```
def height(v):
    if v.isExternal() then
        return 0
    else
        h ← 0
        for each child w of v:
            h ← max(h, height(w))
        return h + 1
```

#### 递归算法复杂度分析

| 递归模式 | Worst Case | 条件 |
|----------|------------|------|
| 对所有 children 递归 | O(n) | 每次调用工作量 O(1)，每个节点访问一次 |
| 对至多一个 child 递归 | O(h) | 每层最多一次调用，h 为树高 |

<span style="color: #9b59b6">💡 **白话理解**：树天然适合递归——"处理这棵树"可以分解为"处理根 + 处理每棵子树"。复杂度取决于你是"全员参与"（O(n)）还是"每层只选一个方向"（O(h)）。</span>

<span style="color: #e67e22">*通用的树和二叉树提供了层级组织数据的框架。但如果我们想高效地搜索数据呢？需要在树上加一个有序性约束——这就是 BST。*</span>

## Part 2 — 二叉搜索树

> [!info] 问题链
> 有序数据怎么高效搜索？→ BST 利用 key(left) < key(node) < key(right) 实现 O(h) 搜索 → 但 h 可能退化到 n → 需要自平衡 → AVL 树用旋转保证 h = O(log n)。

### BST 定义与搜索 `★★★`

> [!tip] 你有 1000 万条按价格排序的二手车记录，用户搜索 $15,000 的车——你怎么做到不用遍历全部记录？

**BST 性质**：对于树中任意节点 v，左子树中所有节点 u 和右子树中所有节点 w，满足：

$$key(u) < key(v) < key(w)$$

**关键推论**：BST 的 inorder traversal 按 key 升序访问所有节点。

**搜索算法**：从 root 开始，比较 key，决定走左还是走右：

```
def search(k, v):
    if v.isExternal() then
        return v                    # 未找到
    if k = key(v) then
        return v                    # 找到
    else if k < key(v) then
        return search(k, v.left)    # 去左子树
    else
        return search(k, v.right)   # 去右子树
```

**复杂度**：O(h)，其中 h 是树高
- **Best case**：h ≤ log₂ n（平衡时）
- **Worst case**：h = n - 1（退化为链表）

<span style="color: #9b59b6">💡 **白话理解**：BST 搜索就像在一本已排好序的字典里查单词——翻到中间看一下，太大就往前翻，太小就往后翻。每一步排除一半可能性。但如果这本"字典"变成了一条直线（退化情况），就跟从头翻到尾一样慢了。</span>

<span style="color: #e67e22">*能搜索了，怎么往 BST 里添加新元素？*</span>

---

### BST 插入 `★★★`

> [!tip] 二手车平台新入库一台 $15,000 的车，怎么把它放到 BST 的正确位置？

**算法**：先用 search(k) 找到 key k 应该在的位置：
1. 如果 k 已存在 → 替换对应的 value
2. 如果 k 不存在 → search 到达 external node w → 用 (k, o) 替换 w 为 internal node

**示例**：在含 {1,2,4,6,8,9} 的 BST 中插入 5：
```
搜索 5：6 → 2 → 4 → (external node)
将 external node 替换为存储 5 的 internal node
```

**复杂度**：O(h)（搜索 + 常数时间创建节点）

<span style="color: #9b59b6">💡 **白话理解**：插入就是"先找位置，再坐下"。搜索到一个空位（external node），把新元素放进去就行。</span>

<span style="color: #e67e22">*插入比较简单，但删除就复杂了——删掉一个有两个孩子的节点后，怎么保持 BST 性质？*</span>

---

### BST 删除 `★★★`

> [!tip] 一台车被卖出，要从 BST 中移除它。如果这台车的节点恰好有两个子树（更贵的和更便宜的车都挂在下面），怎么处理？

#### Case 1：待删节点 w 有至少一个 external child

步骤：
1. 设 z 为 w 的 external child
2. 删除 w 和 z
3. 把 w 的另一个 child 提升到 w 的位置

```
    6              6
   / \            / \
  2   9    →     2   9
 / \            /
1   4          1
   /
  5  ← 删除 4（z 是 4 的右 external child）
     结果：5 提升到 4 的位置
```

#### Case 2：待删节点 w 有两个 internal children

步骤：
1. 找到 w 的**中序后继 y**（右子树中最小的 internal node，即右子树中一直往左走到底）
2. 将 y 的 entry 复制到 w
3. 删除 y（y 的左 child 一定是 external，所以用 Case 1 处理）

```
    3               5
   / \             / \
  1   8    →      1   8
     / \             / \
    5   9           6   9
     \
      6
删除 3：inorder successor = 5，复制 5 到根，删除原来的 5
```

**删除算法伪代码**：
```
def remove(k):
    w ← search(k, root)
    if w.isExternal() then
        return null
    else if w has at least one external child z then
        remove z
        promote the other child of w to take w's place
        remove w
    else
        y ← immediate inorder successor of w
        replace contents of w with entry from y
        remove y (using Case 1)
```

**复杂度**：O(h)（搜索 + 找后继 + 常数时间重组）

<span style="color: #9b59b6">💡 **白话理解**：删除分两种情况。简单情况：节点只有一个"真正的孩子"，直接让孩子顶上来。复杂情况：节点有两个孩子，找右子树中最小的（inorder successor）来替代自己——这个替代者一定满足"比左子树大、比右子树其他节点小"的要求。</span>

### 范围查询 Range Query `★★`

> [!tip] 二手车网站上用户筛选价格 $10K–$15K 的车——BST 怎么高效找到所有落在这个范围内的节点？

**定义**：给定 k1 和 k2，找出所有满足 k1 ≤ key ≤ k2 的节点。

**算法**（受限的 inorder traversal）：
```
def range(v, k1, k2):
    if v is external then
        return
    if key(v) > k2 then
        range(v.left, k1, k2)       # 太大，只搜左子树
    else if key(v) < k1 then
        range(v.right, k1, k2)      # 太小，只搜右子树
    else
        range(v.left, k1, k2)       # key 在范围内
        output.add(v)               # 输出当前节点
        range(v.right, k1, k2)      # 继续搜两边
```

**性能分析**：
- Boundary nodes（在 k1 或 k2 的搜索路径上）≤ 2 × tree height
- Inside nodes（key 在范围内但不在搜索路径上）≤ |output|
- 每个访问节点 O(1) 工作
- **总复杂度：O(|output| + h)**

<span style="color: #9b59b6">💡 **白话理解**：范围查询就像在排好序的书架上找价格在 $10–$15 之间的书。BST 的结构让你能快速跳过不相关的区域——太贵的只看左边，太便宜的只看右边，刚好的左右都看。</span>

<span style="color: #e67e22">*BST 的所有操作都是 O(h)。但 h 最坏是 n-1（退化链表）。我们能不能保证 h 始终是 O(log n)？*</span>

### AVL 树 `★★★`

> [!tip] 如果按升序 1, 2, 3, 4, 5 依次插入 BST，你会得到什么形状？搜索 5 需要多少步？怎么避免这种退化？

#### AVL 定义

AVL 树是一种**rank-balanced BST**，rank = 子树高度。

**平衡约束**：每个内部节点的两个 children 的 rank（高度）差至多为 1。

**关键事实**：存储 n 个 key 的 AVL 树高度为 **O(log n)**。

#### AVL 插入

1. 正常 BST 插入 key k
2. 新节点 w 可能导致某些祖先节点违反 AVL 平衡性
3. 找到最低的不平衡祖先 **z**，设 **y** 为 z 的较高 child（w 的祖先），**x** 为 y 的 child（w 的祖先）
4. 对 x, y, z 做 **trinode restructure**（旋转）

**维护高度属性**：
- 创建新节点后设高度为 1，更新祖先高度
- 旋转后更新参与节点及其祖先高度
- 维护高度只需 O(h) 额外工作

#### Trinode Restructure（旋转）

设 a, b, c 为 x, y, z 按 inorder 排列的结果。目标：让 b 成为三者中最顶部的节点。

**Single Rotation（单旋转）**：x, y, z 呈"直线"排列时

```
Case: Left-Left (z-y-x 从右到左直线)
    z               y
   / \            /   \
  y   T3   →    x      z
 / \           / \    / \
x   T2        T0 T1  T2 T3
/ \
T0  T1

Case: Right-Right (z-y-x 从左到右直线)
  z                 y
 / \              /   \
T0   y    →     z      x
    / \        / \    / \
   T1  x      T0 T1  T2 T3
      / \
     T2  T3
```

**Double Rotation（双旋转）**：x, y, z 呈"折线"排列时

```
Case: Left-Right (z 右, y 左, x 右)
  z                x
 / \             /   \
y   T3   →     y      z
 \            / \    / \
  x          T0 T1  T2 T3
 / \
T1  T2

Case: Right-Left (z 左, y 右, x 左)
  z                x
 / \             /   \
T0   y   →     z      y
    /         / \    / \
   x         T0 T1  T2 T3
  / \
 T1  T2
```

**旋转复杂度**：O(1)（只更新 O(1) 个指针）

#### AVL 删除

1. 正常 BST 删除（通过 search + promote/replace），设 w 为被删节点的 parent
2. 找最低不平衡祖先 z；y = z 的**较高 child**（注意：y 不是 w 的祖先，而是另一侧）；x = y 的较高 child
3. 对 x, y, z 做 trinode restructure
4. **与插入不同**：旋转后可能导致更高层祖先也失衡 → 必须继续向上检查直到 root

#### AVL 性能总结

| 操作 | 复杂度 |
|------|--------|
| Space | O(n) |
| Height | O(log n) |
| Search | O(log n) |
| Insert | O(log n) |
| Remove | O(log n) |

<span style="color: #9b59b6">💡 **白话理解**：AVL 树的核心思想是"不允许任何节点的左右子树高度差超过 1"。每次插入或删除后检查平衡，一旦失衡就通过旋转修复。代价是每次操作多做 O(log n) 的检查和 O(1) 的旋转，但换来了所有操作稳定 O(log n) 的保证——不再怕退化。插入只需一次旋转修复，删除可能需要多次（沿路径向上传播）。</span>

<span style="color: #e67e22">*有了高效的树结构，我们可以用它来实现更抽象的数据容器——Map（键值对存储）和 Priority Queue（按优先级取出）。*</span>

## Part 3 — Map 与优先队列

> [!info] 问题链
> 如何按 key 存取数据？→ Map ADT → 用什么数据结构实现？→ Unsorted list O(n) vs AVL tree O(log n) → 如果只需要取最小/最大？→ Priority Queue ADT → 用 Heap 实现 O(log n)。

### Map ADT `★★★`

> [!tip] 一个学生管理系统需要按学号查成绩、更新成绩、删除退课学生——你需要什么样的数据接口？

**Map**：存储 (key, value) 对的集合，key 唯一。

| 操作 | 描述 |
|------|------|
| `get(k)` | 返回 key k 对应的 value（不存在返回 null） |
| `put(k, v)` | 插入 (k, v)；若 k 已存在，替换 value 并返回旧 value |
| `remove(k)` | 删除 key k 的条目 |
| `size()` | 条目数量 |
| `isEmpty()` | 是否为空 |
| `entrySet()` | 返回所有 (key, value) 对 |
| `keySet()` | 返回所有 key |
| `values()` | 返回所有 value |

**操作追踪示例**：

| 操作 | 返回 | Map 状态 |
|------|------|----------|
| isEmpty() | true | Ø |
| put(5,A) | null | (5,A) |
| put(7,B) | null | (5,A),(7,B) |
| put(2,C) | null | (5,A),(7,B),(2,C) |
| put(8,D) | null | (5,A),(7,B),(2,C),(8,D) |
| put(2,E) | C | (5,A),(7,B),(2,E),(8,D) |
| get(7) | B | (5,A),(7,B),(2,E),(8,D) |
| get(4) | null | (5,A),(7,B),(2,E),(8,D) |
| size() | 4 | (5,A),(7,B),(2,E),(8,D) |
| remove(5) | A | (7,B),(2,E),(8,D) |
| remove(2) | E | (7,B),(8,D) |
| get(2) | null | (7,B),(8,D) |

<span style="color: #9b59b6">💡 **白话理解**：Map 就是一本通讯录——name 是 key，phone 是 value。`put` = 添加/更新联系人，`get` = 查号码，`remove` = 删联系人。关键：每个 name 只能对应一个 phone（key 唯一）。</span>

### Sorted Map ADT `★★`

在 Map 基础上，要求 key 有全序关系，额外支持：

| 操作 | 描述 |
|------|------|
| `firstEntry()` | 最小 key 的 entry |
| `lastEntry()` | 最大 key 的 entry |
| `ceilingEntry(k)` | key ≥ k 的最小 entry |
| `floorEntry(k)` | key ≤ k 的最大 entry |
| `lowerEntry(k)` | key < k 的最大 entry |
| `higherEntry(k)` | key > k 的最小 entry |
| `subMap(k1, k2)` | 所有 k1 ≤ key < k2 的 entry |

#### Map 实现方式对比

| 实现 | get / put | 适用场景 |
|------|-----------|----------|
| **Unsorted List** | O(n) | Map 很小，或只在尾部 put、很少 get（如 system log） |
| **AVL Tree (sorted)** | O(log n) | key 可比较，需要高效搜索 |

<span style="color: #9b59b6">💡 **白话理解**：Sorted Map 比普通 Map 多了"范围操作"——不只是精确查找，还能找"最近的"、"最小的"、"在某个范围内的"。用 AVL 树实现时，这些操作都是 O(log n)。</span>

<span style="color: #e67e22">*Map 需要按任意 key 存取。但有些场景只关心"最小的"——比如急诊室总是先处理最严重的病人。这就是优先队列。*</span>

### Priority Queue ADT `★★★`

> [!tip] 医院急诊室来了 6 个病人，按伤情严重程度排优先级。你用什么数据结构让护士总能 O(log n) 找到最紧急的病人？

**Priority Queue**：存储 (key, value) 对的集合，只能移除 key 最小的元素。

| 操作 | 描述 |
|------|------|
| `insert(k, v)` | 插入 (k, v) |
| `remove_min()` | 移除并返回 key 最小的 item |
| `min()` | 返回 key 最小的 item（不移除） |
| `size()` | 元素数量 |
| `is_empty()` | 是否为空 |

也可以有 max 版本（`remove_max`），但不能同时使用 min 和 max。

**操作追踪示例**：

| 操作 | 返回 | PQ 状态 |
|------|------|---------|
| insert(5,A) | — | {(5,A)} |
| insert(9,C) | — | {(5,A),(9,C)} |
| insert(3,B) | — | {(3,B),(5,A),(9,C)} |
| min() | (3,B) | {(3,B),(5,A),(9,C)} |
| remove_min() | (3,B) | {(5,A),(9,C)} |
| insert(7,D) | — | {(5,A),(7,D),(9,C)} |
| remove_min() | (5,A) | {(7,D),(9,C)} |
| remove_min() | (7,D) | {(9,C)} |
| remove_min() | (9,C) | {} |
| is_empty() | true | {} |

**应用：股票撮合引擎**
- 买单用 max-PQ（最高出价优先）
- 卖单用 min-PQ（最低要价优先）
- 当最高出价 ≥ 最低要价时，成交

```
while True:
    bid ← buy_orders.remove_max()
    ask ← sell_orders.remove_min()
    if bid.price ≥ ask.price then
        carry out trade(bid, ask)
    else
        buy_orders.insert(bid)
        sell_orders.insert(ask)
```

<span style="color: #9b59b6">💡 **白话理解**：Priority Queue 就像急诊室的分诊台——不管你什么时候来的，总是最严重的（key 最小的）先看病。与普通队列（FIFO）的区别：普通队列按到达顺序，PQ 按优先级。</span>

---

### 基于序列的优先队列与排序 `★★`

#### 两种序列实现

| 实现 | insert | remove_min / min | 原理 |
|------|--------|------------------|------|
| **Unsorted list** | O(1) | O(n) | 插入直接放末尾，取最小时遍历全部 |
| **Sorted list** | O(n) | O(1) | 插入时找正确位置，最小的总在头部 |

#### PQ-Sort 框架

用 PQ 排序：先全部 insert，再逐个 remove_min：

```
def priority_queue_sorting(A):
    pq ← new priority queue
    n ← size(A)
    for i in [0:n]:
        pq.insert(A[i])
    for i in [0:n]:
        A[i] = pq.remove_min()
```

两种序列实现的 PQ-Sort 都是 O(n²)。

#### Selection Sort（对应 unsorted list 实现）

- 第一阶段：n 次 insert → O(n)
- 第二阶段：n 次 remove_min → O(n²)（每次扫描未排序部分找最小）
- **In-place**：A[0:i] 已排序，A[i:n] 是未排序的 PQ

```
def selection_sort(A):
    n ← size(A)
    for i in [0:n]:
        s ← i
        for j in [i:n]:
            if A[j] < A[s] then
                s ← j
        A[i], A[s] ← A[s], A[i]
```

**Loop invariant**：A[0:i] 已排序，且 A[i:n] 中所有元素 ≥ A[i-1]

#### Insertion Sort（对应 sorted list 实现）

- 第一阶段：n 次 insert → O(n²)（每次找正确位置插入）
- 第二阶段：n 次 remove_min → O(n)
- **In-place**：A[0:i] 是已排序的 PQ，A[i:n] 待插入

```
def insertion_sort(A):
    n ← size(A)
    for i in [1:n]:
        x ← A[i]
        j ← i
        while j > 0 and x < A[j-1]:
            A[j] ← A[j-1]
            j ← j - 1
        A[j] ← x
```

<span style="color: #9b59b6">💡 **白话理解**：Selection Sort = 每轮从剩余元素中"选"最小的放到正确位置。Insertion Sort = 每轮把新元素"插"到已排序部分的正确位置。两者都是 O(n²)，只是"花时间的阶段"不同。</span>

<span style="color: #e67e22">*O(n²) 太慢了。有没有更好的 PQ 实现？Heap 能让 insert 和 remove_min 都变成 O(log n)。*</span>

### 堆 Heap `★★★`

> [!tip] 医院的病人管理系统要求：(1) 新病人入院 O(log n) 登记优先级；(2) 随时 O(log n) 叫到最紧急的病人。什么数据结构同时满足这两点？

#### Min-Heap 定义

Heap 是一棵存储 (key, value) 的二叉树，满足两个性质：

1. **Heap-Order**：对于任意节点 m ≠ root，key(m) ≥ key(parent(m))（即父节点的 key ≤ 子节点的 key）
2. **Complete Binary Tree**：
   - 设高度为 h，每一层 i < h 都是满的（有 2ⁱ 个节点）
   - 最底层 h 的节点从左到右紧密排列（左对齐）

**Last node**：最大深度的最右节点

#### Heap 插入 + Upheap

1. 在"下一个可用位置"（last node 之后）创建新节点
2. **Upheap**：如果新节点 key < parent key，交换两者，继续向上比较

```
def up_heap(z):
    while z ≠ root and key(parent(z)) > key(z):
        swap key of z and parent(z)
        z ← parent(z)
```

**找到插入位置的方法**：
- 从 last node 出发
- 向上走直到遇到左 child 或 root
- 如果到 root → 开新层
- 否则去 sibling（parent 的右 child）
- 再向下走左 child 直到叶子

**复杂度**：O(log n)（树高 = log n）

#### Heap 删除 + Downheap

1. 将 root 的 key 替换为 last node 的 key
2. 删除 last node
3. **Downheap**：如果 root 的 key > 某个 child 的 key，与较小的 child 交换，继续向下

```
def down_heap(z):
    while z has child with key(child) < key(z):
        x ← child of z with smallest key
        swap keys of x and z
        z ← x
```

**找到新 last node 的方法**：
- 从（旧的）last node 出发
- 向上走直到遇到右 child 或 root
- 如果到 root → 关闭一层
- 否则去 sibling（parent 的左 child）
- 再向下走右 child 直到叶子

**复杂度**：O(log n)

<span style="color: #9b59b6">💡 **白话理解**：Heap 的两个规则——(1) 老板的优先级永远 ≤ 下属（heap-order），(2) 办公楼每层从左到右坐满了才坐下一层（complete binary tree）。插入 = 新员工坐到下一个空位，然后跟上级比较，比上级优先级高就换位，一路换到合适位置（upheap）。删除最小 = 老板走了，把最后入职的人临时放到老板位置，再根据优先级一路往下换到合适位置（downheap）。</span>

---

### 堆排序 Heap-Sort `★★`

用 Heap 实现 PQ-Sort：
- n 次 insert：每次 O(log n) → 总共 O(n log n)
- n 次 remove_min：每次 O(log n) → 总共 O(n log n)
- **总复杂度：O(n log n)**

**优化**：
- 可以 in-place 实现（数组的一部分做输出，一部分做 heap）
- 建堆（heapify）可以在 O(n) 时间内完成，但 n 次 remove_min 仍是 O(n log n)

**额外操作**（给定指向元素 e 的指针）：
- `remove(e)`：删除任意元素
- `replace_key(e, k)`：更新元素 key
- `replace_value(e, v)`：更新元素 value

<span style="color: #9b59b6">💡 **白话理解**：Heap-Sort 比 Selection Sort/Insertion Sort（O(n²)）快，因为 Heap 让 insert 和 remove_min 都是 O(log n) 而非 O(n)。总时间从 O(n²) 降到 O(n log n)。</span>

---

### 堆的数组实现 `★★`

利用 complete binary tree 的紧密结构，可以用数组（无需指针）表示 Heap：

| 概念 | 数组下标 |
|------|----------|
| Root | 0 |
| Last node | n - 1 |
| 节点 i 的 left child | **2i + 1** |
| 节点 i 的 right child | **2i + 2** |
| 节点 i 的 parent | **⌊(i-1)/2⌋** |

**示例**：
```
Heap:       2
          /   \
         5     6
        / \
       9   7

Array: [2, 5, 6, 9, 7]
Index:  0  1  2  3  4
```

**优势**：无需存储 parent/child 指针，节省空间；数组连续内存，缓存友好。

**Entries & Comparators**：
- Entry 对象封装 (key, value) 关联
- Comparator 函数 `compare(a, b)` 返回 < 0 / 0 / > 0
- 注意：不要假设 compare 只返回 -1, 0, 1

<span style="color: #9b59b6">💡 **白话理解**：因为 Heap 是 complete binary tree，每层都填满了（最后一层从左到右），所以节点位置完全可预测——用简单的数学公式（2i+1, 2i+2）就能找到孩子和父母，不需要额外的指针。</span>

## 核心对比速查

### BST vs AVL Tree

| 维度 | BST | AVL Tree |
|------|-----|----------|
| 平衡保证 | 无 | 左右子树高度差 ≤ 1 |
| Search | O(h)，worst O(n) | O(log n) |
| Insert | O(h)，worst O(n) | O(log n)（含一次旋转） |
| Remove | O(h)，worst O(n) | O(log n)（可能多次旋转） |
| Height | O(n) worst | O(log n) guaranteed |
| 实现复杂度 | 简单 | 需要维护高度、做旋转 |

### Map 实现对比

| 实现 | get | put | remove | 适用场景 |
|------|-----|-----|--------|----------|
| Unsorted List | O(n) | O(n) | O(n) | 小规模、只追加很少查询 |
| AVL Tree (Sorted Map) | O(log n) | O(log n) | O(log n) | 大规模、key 可排序 |

### Priority Queue 实现对比

| 实现 | insert | remove_min | min | 排序算法 | 排序复杂度 |
|------|--------|------------|-----|----------|------------|
| Unsorted List | O(1) | O(n) | O(n) | Selection Sort | O(n²) |
| Sorted List | O(n) | O(1) | O(1) | Insertion Sort | O(n²) |
| **Heap** | **O(log n)** | **O(log n)** | **O(1)** | **Heap-Sort** | **O(n log n)** |

### 三种遍历对比

| 遍历 | 访问顺序 | 典型应用 |
|------|----------|----------|
| Preorder | 根 → 左 → 右 | 复制树、战略决策（自上而下） |
| Inorder | 左 → 根 → 右 | BST 升序输出、算术表达式中缀打印 |
| Postorder | 左 → 右 → 根 | 计算子树大小、文件夹备份（自下而上） |
