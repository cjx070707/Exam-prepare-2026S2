## Part 1 — 基础概念与数组

### ADT vs 数据结构 `★★★`

> [!tip] 面试官问你"用什么数据结构"，你说"用 List"——这是 ADT 还是数据结构？

- **Abstract Data Type (ADT)**：从**用户角度**定义的规范——有哪些操作、行为是什么，不关心怎么实现
- **Data Structure**：从**实现者角度**定义的具体表示——数据怎么在内存中组织

<span style="color: #9b59b6">💡 **白话理解**：ADT 是菜单（告诉你能点什么），Data Structure 是厨房（告诉你怎么做）。同一份菜单可以有不同的厨房实现。</span>

类比 Computational Problem vs Algorithm：
- Computational Problem 定义输入输出（做什么）
- Algorithm 定义步骤（怎么做）
- Correctness & Complexity Analysis 证明正确性 + 分析资源消耗

<span style="color: #e67e22">*数据存进去了，但我们怎么知道一个算法"够快"？需要一个与硬件无关的衡量方式。*</span>

---

### Big-O 复杂度分析 `★★★`

> [!tip] 同一个任务，算法 A 在你电脑上跑得快，算法 B 在服务器上跑得快——谁更好？

**为什么不用执行时间？**
- 依赖硬件（CPU、RAM）
- 依赖实现（语言、编译器优化）
- 受外部因素干扰（后台进程）
- 无法预测大规模输入的表现

**Big-O 记法**：描述算法运行时间/空间随输入规模 n 增长的**上界**（worst case）。
- 忽略常数项和低阶项：O(2n + 4n²) → O(n²)
- 关注**最高增长率的那一项**

#### 常见复杂度对比

| 复杂度 | 名称 | 例子 | 类比 |
|--------|------|------|------|
| O(1) | 常数 | 数组按下标访问 arr[i] | 按书架编号找书 |
| O(log n) | 对数 | 二分查找 | 在字典里查单词 |
| O(n) | 线性 | 遍历数组 | 逐页翻书 |
| O(n log n) | 线性对数 | Merge Sort | 高效排序 |
| O(n²) | 二次 | 嵌套循环比较 | 锦标赛两两对决 |
| O(2ⁿ) | 指数 | 暴力枚举子集 | 暴力破解密码 |

#### 时间复杂度分析方法

| 代码结构 | 复杂度 |
|----------|--------|
| 单条赋值/比较/读写 | O(1) |
| if-else（取较大分支） | max(block1, block2) |
| 单层 for 循环，循环体 O(1) | O(n) |
| 单层 for 循环，循环体 O(n) | O(n²) |
| 嵌套 for 循环 n×m，循环体 O(1) | O(nm) |
| 二分查找（每次减半） | O(log n) |
| 三层嵌套 n×n×n | O(n³) |

#### 空间复杂度

- 衡量算法需要的**额外内存**随 n 的增长
- 线性查找：时间 O(n)，空间 O(1)（只需一个变量记录当前位置）
- 二分查找（迭代）：时间 O(log n)，空间 O(1)

**Big-O vs Omega**：
- Big-O `O()`：上界（worst case）
- Omega `Ω()`：下界（best case）

<span style="color: #9b59b6">💡 **白话理解**：Big-O 就像说"这件事最坏不会超过 X 时间"。它不关心最好的情况有多快，只关心最差能差到哪里。面试和考试几乎只考 Big-O。</span>

<span style="color: #e67e22">*有了衡量工具，我们来看第一个数据结构——数组。它的 get/set 很快，但有什么代价？*</span>

---

### 数组 `★★`

> [!tip] 你需要存 1000 个学生成绩，但一开始不知道会有多少学生——数组适合吗？

- **定义**：固定大小、连续内存的元素集合
- **声明需要**：数据类型 + 名称 + 大小
- **核心操作**：
  - `get(i)` / `set(i, e)`：O(1)（直接通过下标定位内存地址）

**优缺点**：

| 优点 | 缺点 |
|------|------|
| O(1) 随机访问 | 大小固定，无法动态扩展 |
| 支持简单循环遍历 | 需要连续内存块 |
| | 传统数组只存同一类型数据 |

<span style="color: #9b59b6">💡 **白话理解**：数组像一排固定数量的储物柜——开柜子很快（O(1)），但柜子数量定了就不能改，中间要插一个得让后面的人全部挪位。</span>

## Part 2 — 链表

<span style="color: #e67e22">*数组的固定大小和 O(n) 插入/删除成了瓶颈。如果数据量不确定、频繁增删，我们需要一种不要求连续内存的结构。*</span>

### List ADT `★★`

一个 index-based list 支持的操作：

| 操作 | 描述 |
|------|------|
| `size()` | 返回元素数量 |
| `isEmpty()` | 是否为空 |
| `get(i)` | 返回下标 i 的元素 |
| `set(i, e)` | 替换下标 i 的元素，返回被替换的 |
| `add(i, e)` | 在下标 i 插入元素，后续元素右移 |
| `remove(i)` | 删除下标 i 的元素，后续元素左移 |

### Array-based List 的局限 `★★`

用数组实现 List ADT 时：

| 操作 | 复杂度 | 原因 |
|------|--------|------|
| get(i) / set(i, e) | O(1) | 直接访问 A[i] |
| add(i, e) | **O(n)** worst | 需要右移 n-i 个元素 |
| remove(i) | **O(n)** worst | 需要左移 n-i-1 个元素 |

- 空间：用了 O(N)，但实际只存 n 个元素（N ≥ n）→ 浪费空间

<span style="color: #e67e22">*插入删除都要挪元素，代价太高。链表用指针把节点串起来，不要求连续内存，插入只需要改几个指针。*</span>

---

### 单链表 Singly Linked List `★★★`

> [!tip] 如果你只有一个 head 指针指向链表头，要删除倒数第二个节点，你怎么找到它前面那个节点？

**结构**：每个 Node 存 `data` + `next` 指针，最后一个 node 的 next 指向 None。通过 `head` 引用第一个节点。

```
A → B → C → D → ∅
↑
head
```

**优势（相比数组）**：
- 动态大小：不需要预定义容量
- 灵活内存：节点散布在内存各处，不需要连续块
- 无浪费空间：用多少分配多少

#### 核心操作与复杂度

| 操作 | 步骤概要 | 时间复杂度 |
|------|----------|------------|
| **遍历** | current = head，沿 next 走到 None | O(n) |
| **头部插入** | 新节点 → head → 旧头 | O(1) |
| **尾部插入** | 遍历到最后一个节点，接上新节点 | O(n) |
| **指定位置插入** | 遍历到位置 i-1，修改指针 | O(n) |
| **头部删除** | head = head.next | O(1) |
| **尾部删除** | 遍历到倒数第二个节点，断开 next | O(n) |
| **指定位置删除** | 遍历到位置 i-1，跳过目标节点 | O(n) |

#### 指定位置插入的关键步骤

```
1. new_node.next = current.next    // 新节点接上后续
2. current.next = new_node         // 前驱指向新节点
```

⚠️ **顺序不能反**——先接后续再改前驱，否则会丢失后续节点的引用。

<span style="color: #9b59b6">💡 **白话理解**：单链表像一列火车——每节车厢只知道"下一节是谁"。要在中间插入一节车厢，你得先走到那个位置（O(n)），然后把挂钩重新接一下（O(1)）。但你没法往回走。</span>

<span style="color: #e67e22">*单链表只能往前走，删除尾部还得从头遍历找倒数第二个节点。如果每个节点同时记住前驱呢？*</span>

---

### 双链表 Doubly Linked List `★★★`

> [!tip] 浏览器的"前进"和"后退"功能需要双向遍历——单链表够用吗？

**结构**：每个 Node 存 `data` + `prev` + `next`。

```
∅ ← A ⇄ B ⇄ C ⇄ D → ∅
     ↑
    head
```

#### 操作复杂度（与单链表对比）

| 操作 | 单链表 | 双链表 | 差异原因 |
|------|--------|--------|----------|
| 头部插入 | O(1) | O(1) | 都只改头部指针 |
| 尾部插入（无 tail 指针） | O(n) | O(n) | 都需要遍历到尾部 |
| 头部删除 | O(1) | O(1) | 都只改 head |
| 尾部删除（无 tail 指针） | O(n) | O(n) | 都需找到倒数第二个 |
| 给定节点引用的删除 | O(n) | **O(1)** | 双链表有 prev，直接改前后指针 |

#### 指定位置插入的关键步骤（4 步）

```
1. new_node.next = current.next       // 新节点接后续
2. current.next.prev = new_node       // 后续节点的 prev 指向新节点
3. current.next = new_node            // 前驱的 next 指向新节点
4. new_node.prev = current            // 新节点的 prev 指向前驱
```

#### 指定位置删除的关键步骤

```
1. current.prev.next = current.next   // 前驱跳过 current
2. current.next.prev = current.prev   // 后续回指前驱
3. current.prev = None                // 断开 current
4. current.next = None
```

<span style="color: #9b59b6">💡 **白话理解**：双链表像双向铁轨——每站都知道上一站和下一站。多了一倍的指针开销，但换来了"已知节点的 O(1) 删除"。考试常考的对比点：单链表 vs 双链表的 tradeoff 就是 **空间（多一个 prev 指针）换时间（给定节点的删除从 O(n) 变 O(1)）**。</span>

---

### 循环链表 Circular Linked List `★`

尾节点的 next 指向 head（而非 None），形成环形结构。

```
A → B → C → D → (回到 A)
↑                  |
head ←─────────────┘
```

- 适用场景：循环调度（如 iPhone 闹钟的时间选择器——12 点之后回到 1 点）
- 可以是单向循环或双向循环
- 遍历终止条件变为 `current.next != head`（而非 `!= None`）

<span style="color: #9b59b6">💡 **白话理解**：普通链表像一条路走到头就没了；循环链表像一条环形跑道，永远不会"走到尽头"。</span>

## Part 3 — 受限线性结构

<span style="color: #e67e22">*链表/数组是通用的——可以在任意位置增删。但很多实际问题只需要在"一端"或"两端"操作。用一个更受限的 ADT 来约束操作，能让代码意图更清晰、实现更简单。*</span>

### 栈 Stack `★★★`

> [!tip] 你在写递归函数，每次调用 f(n) 会调用 f(n-1)——系统怎么记住每层的局部变量和返回地址？

**原则**：Last-In-First-Out (LIFO)

**操作**：

| 操作 | 描述 | 复杂度 |
|------|------|--------|
| `push(e)` | 在栈顶插入元素 | O(1) |
| `pop()` | 移除并返回栈顶元素 | O(1) |
| `top()` | 返回栈顶元素（不移除） | O(1) |
| `size()` | 元素数量 | O(1) |
| `isEmpty()` | 是否为空 | O(1) |

**操作追踪示例**：

| 操作 | 栈状态 | 返回值 |
|------|--------|--------|
| push(5) | [5] | - |
| push(3) | [5, 3] | - |
| size() | [5, 3] | 2 |
| pop() | [5] | 3 |
| isEmpty() | [5] | False |
| pop() | [] | 5 |
| isEmpty() | [] | True |

**应用**：
- **方法调用栈**：每次函数调用 push 一个 frame（局部变量 + 返回地址 + PC），函数结束 pop。这就是递归能工作的原因。
- **括号匹配**（详见下节）
- **撤销操作**（浏览器历史、编辑器 undo）
- 作为其他算法的辅助结构

<span style="color: #9b59b6">💡 **白话理解**：栈就像一摞盘子——只能从最上面拿、往最上面放。递归函数的调用链天然就是一个栈：最后被调用的函数最先返回。</span>

---

### 栈应用：括号匹配 `★★`

> [!tip] 给你一个表达式 `{[x + y) ] - z}`，你能快速判断它的括号匹配吗？

**算法**：
1. 初始化空栈
2. 遍历每个字符：
   - 遇到开括号 `(` `{` `[` → push
   - 遇到闭括号 `)` `}` `]` → 如果栈空或栈顶不匹配 → 返回 false；否则 pop
3. 遍历结束后：栈空 → balanced，栈非空 → unbalanced

**示例 1**：`( w * [ x + y ] / z )` → ✅ balanced
- `(` push → `[` push → `]` 匹配 pop → `)` 匹配 pop → 栈空

**示例 2**：`{ [ x + y ) ] - z }` → ❌ unbalanced
- `{` push → `[` push → `)` 与栈顶 `[` 不匹配 → false

<span style="color: #e67e22">*栈解决了"后进先出"场景。但如果场景是排队——先来的先服务呢？*</span>

---

### 队列 Queue `★★★`

> [!tip] 打印机收到 10 个打印任务，应该先执行哪个？如果用栈会怎样？

**原则**：First-In-First-Out (FIFO)

**操作**：

| 操作 | 描述 | 复杂度 |
|------|------|--------|
| `enqueue(e)` | 在队尾插入元素 | O(1) |
| `dequeue()` | 移除并返回队头元素 | O(1) |
| `first()` | 返回队头元素（不移除） | O(1) |
| `size()` | 元素数量 | O(1) |
| `isEmpty()` | 是否为空 | O(1) |

**操作追踪示例**：

| 操作 | 队列状态 | 返回值 |
|------|----------|--------|
| enqueue(5) | (5) | - |
| enqueue(3) | (5, 3) | - |
| dequeue() | (3) | 5 |
| enqueue(7) | (3, 7) | - |
| dequeue() | (7) | 3 |
| first() | (7) | 7 |
| dequeue() | () | 7 |
| isEmpty() | () | True |

**应用**：
- 缓冲流数据（视频/音频 buffering）
- 等候队列（售票窗口）
- 共享资源访问（打印机队列）
- 多任务调度（Multiprogramming）
- 作为其他算法的辅助结构（如 BFS）

<span style="color: #9b59b6">💡 **白话理解**：队列就像排队买奶茶——先排的先买。栈和队列的核心区别只有一个：**从哪一端取出**。栈从放入端取（LIFO），队列从另一端取（FIFO）。</span>

## 核心对比速查

### 数组 vs 链表

| 维度 | 数组 | 单链表 | 双链表 |
|------|------|--------|--------|
| 内存布局 | 连续 | 散布 | 散布 |
| 大小 | 固定 | 动态 | 动态 |
| get(i) | O(1) | O(n) | O(n) |
| set(i, e) | O(1) | O(n) | O(n) |
| add(i, e) | O(n) | O(n) 遍历 + O(1) 插入 | O(n) 遍历 + O(1) 插入 |
| remove(i) | O(n) | O(n) | O(n) |
| 给定节点删除 | N/A | O(n) 需找前驱 | **O(1)** 有 prev |
| 空间开销 | O(N) 可能浪费 | O(n) 每节点 +1 指针 | O(n) 每节点 +2 指针 |

### 栈 vs 队列

| 维度 | 栈 | 队列 |
|------|-----|------|
| 原则 | LIFO | FIFO |
| 插入 | push（栈顶）| enqueue（队尾）|
| 取出 | pop（栈顶）| dequeue（队头）|
| 查看 | top() | first() |
| 经典应用 | 递归/括号匹配/undo | 排队/BFS/缓冲 |
