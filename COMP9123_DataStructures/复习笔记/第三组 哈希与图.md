## Part 1 — 哈希表

### Map ADT 回顾 `★★`

> [!tip] 网络路由器每秒要处理成千上万个数据包，对每个目标地址需要 O(1) 查找——用链表实现的 Map 行吗？

Map 是一个存储 key-value 对的可搜索集合，每个 key 最多一个 entry。

**核心操作**：

| 操作 | 描述 |
|------|------|
| `get(k)` | 返回 key k 对应的 value，不存在则返回 null |
| `put(k, v)` | 插入 (k, v)；若 k 已存在则更新，返回旧 value |
| `remove(k)` | 删除 key k 的 entry，返回其 value |
| `size()` | 返回 entry 数量 |
| `is_empty()` | 是否为空 |

扩展操作：`entries()`, `keys()`, `values()` 返回可迭代集合。

**List-based Map 的性能**：用无序链表实现，put（key 不存在时）O(1)，但 get / put（需检查 key）/ remove 都是 **O(n)**——必须遍历整个链表。

**直接地址法**：如果 key 范围是 [0, N-1]，用大小为 N 的数组，key 直接做下标 → get/put/remove 全是 O(1)。但 N ≫ n 时空间浪费严重（如学号 9 位数 → 数组需要 10⁹ 个位置），且无法处理字符串等非整数 key。

<span style="color: #9b59b6">💡 **白话理解**：List-based Map 像在一堆没分类的文件里翻找，O(n) 太慢。直接地址法像给每个可能的 key 都留一格，O(1) 但空间爆炸。哈希表是折中方案：用一个函数把 key"压缩"到较小范围。</span>

<span style="color: #e67e22">*直接地址法太浪费空间、无法处理非整数 key。如果有一个函数能把任意 key 映射到小范围的下标呢？*</span>

---

### 哈希函数与哈希表 `★★★`

> [!tip] 你要设计一个学号查询系统，学号是 9 位数但只有 5000 个学生——怎么设计一个 O(1) 查询的数据结构？

**哈希函数** h 将 key 映射到固定范围 [0, N-1] 的整数：
- h 是**数学函数**（同一个 key 永远返回同一个值）
- h 要高效可计算
- h(x) 称为 key x 的**哈希值**

**哈希表**由以下组成：
- 哈希函数 h: K → [0, N-1]
- 大小为 N 的数组（称为 table）
- 理想情况下，entry (x, o) 存储在 A[h(x)]

**例子**：学号系统，N = 10000，h(x) = 学号的后四位数字。

**Load Factor（负载因子）** λ = n/N：
- n = 已存储的 entry 数量
- N = 数组大小
- λ 衡量哈希表的"拥挤程度"
- 实践中应保持 λ 不太高：Java HashSet 用 chaining，λ < 0.75；Python dict 用 open addressing，λ < 0.66
- 当 λ 达到阈值时，创建更大的表（如 2 倍大小），重新哈希所有元素

<span style="color: #9b59b6">💡 **白话理解**：哈希函数像一个"地址翻译器"——给它一个 key，它告诉你去数组的哪个格子找。负载因子就像停车场使用率：太满了找车位就慢（碰撞多），需要扩建（rehash）。</span>

<span style="color: #e67e22">*哈希函数怎么设计才能让 key 均匀分布？这直接决定了碰撞的频率。*</span>

---

### 哈希码与压缩函数 `★★`

哈希函数通常是两步的组合：
1. **Hash Code** h₁: keys → integers（把 key 转成整数）
2. **Compression Function** h₂: integers → [0, N-1]（把整数压缩到数组范围内）

$$h(x) = h_2(h_1(x))$$

**常见 Hash Code 方法**：

| 方法 | 公式 | 特点 |
|------|------|------|
| 分量求和 | h(k) = Σ xᵢ | 排列不变性问题：`"mate"` `"team"` 同码 |
| 分量求和 mod p | h(k) = (Σ xᵢ) mod p | 同上问题 |

**常见 Compression Function**：

| 方法 | 公式 | 说明 |
|------|------|------|
| Modular Division | h(k) = k mod N | N 选素数；若 key 均匀分布在 [0, M]，碰撞概率 1/N |
| Random Linear | h(k) = ((ak + b) mod p) mod N | a, b 从 [1, p-1] 随机选，a ≠ 0，p > M 为素数 |

**Universal Hash Family**：
- 一族哈希函数 H，从中随机选一个 h，满足 Pr[h(i) = h(j)] ≤ 1/N
- 随机线性哈希函数就是 2-universal 的
- 给定 key k 在 n 个 key 的集合中，期望碰撞数 ≤ n/N

<span style="color: #9b59b6">💡 **白话理解**：Hash Code 是"把任意东西变成数字"，Compression Function 是"把数字缩到数组范围内"。分量求和的问题是字母顺序不同的单词可能得到相同的码——就像"abc"和"cba"权重一样。随机线性哈希函数通过引入随机参数 a, b 来打散这种规律。</span>

<span style="color: #e67e22">*无论哈希函数多好，碰撞都无法完全避免。当两个 key 映射到同一个位置时，怎么办？*</span>

### 碰撞处理 — Separate Chaining `★★★`

> [!tip] 如果 100 个 key 全部哈希到同一个桶，Separate Chaining 的 get 操作退化成什么复杂度？

**核心思想**：数组每个 cell 指向一个**链表**，所有映射到同一位置的 entry 存在同一链表中。

```
def get(k):
    return A[h(k)].get(k)

def put(k, v):
    return A[h(k)].put(k, v)

def remove(k):
    return A[h(k)].remove(k)
```

**性能分析**：

| 情况 | get / put / remove |
|------|-------------------|
| 期望（均匀哈希） | **O(1 + λ)** |
| 最坏（全部碰撞到一条链） | **O(n)** |

- E(X) = n/N = λ，即每个桶期望存 λ 个 entry
- 当 λ 为常数时，操作期望 O(1)

**优缺点**：
- ✅ 简单直观
- ✅ 删除容易（链表删除）
- ❌ 需要额外内存存链表指针

<span style="color: #9b59b6">💡 **白话理解**：Separate Chaining 就像在每个停车位上盖了一栋小楼——碰撞了就往上叠。只要每栋楼不太高（λ 不大），找车还是很快。但如果所有车都停到同一栋楼（最坏情况），那就变成了在一栋 n 层楼里找车——O(n)。</span>

<span style="color: #e67e22">*Chaining 需要额外内存存链表。能不能把碰撞的元素也放在数组本身里？*</span>

---

### 碰撞处理 — Linear Probing `★★★`

> [!tip] 用 Linear Probing，先后插入 18, 41, 22, 44, 59, 32, 31, 63（h(x) = x mod 13），写出最终数组状态。

**Open Addressing**：碰撞的元素放到数组的**其他空位**中，不用外部链表。

**Linear Probing**：碰撞时，依次检查下一个位置（循环），直到找到空位。

**例子**：h(x) = x mod 13，依次插入 18, 41, 22, 44, 59, 32, 31, 63

| key | h(x) | 实际位置 | 说明 |
|-----|------|----------|------|
| 18 | 5 | 5 | 空位，直接放 |
| 41 | 2 | 2 | 空位 |
| 22 | 9 | 9 | 空位 |
| 44 | 5 | 6 | 5 被占（18），探测到 6 |
| 59 | 7 | 7 | 空位 |
| 32 | 6 | 8 | 6 被占（44），7 被占（59），探测到 8 |
| 31 | 5 | 10 | 5~9 全被占，探测到 10 |
| 63 | 11 | 11 | 空位 |

最终数组：`[_, _, 41, _, _, 18, 44, 59, 32, 22, 31, 63, _]`

**get(k) 操作**：
1. 从 h(k) 开始
2. 依次探测，直到：找到 key k / 找到空位（key 不存在）/ 探测了 N 次

**DEFUNCT 标记**：
- remove(k) 不能直接清空 cell（会中断探测链），改为放置特殊标记 DEFUNCT
- get(k) 遇到 DEFUNCT 继续探测
- put(k, v) 记录第一个 DEFUNCT/空位的下标 j，若最终没找到 k 则在 j 处插入

**性能分析**：

| 情况 | get / put / remove |
|------|-------------------|
| 期望 | **O(1/(1-λ))** |
| 最坏 | **O(n)** |

- λ < 1 为常数时，期望 O(1)
- 碰撞导致"聚集"（clustering）：连续被占的 cell 越来越长

<span style="color: #9b59b6">💡 **白话理解**：Linear Probing 像停车场——你的车位被占了，就往后一格一格找空位。问题是"滚雪球效应"：一旦某片区域连续满了，新来的车更容易堆在这片后面，让拥堵区越来越长。DEFUNCT 标记像"此车位暂时保留"——不能直接清空，否则后面的车就以为前面没车了。</span>

<span style="color: #e67e22">*Linear Probing 的 get/remove 最坏 O(n)。有没有方法在最坏情况下也能 O(1) 查找？*</span>

### 碰撞处理 — Cuckoo Hashing `★★★`

> [!tip] Cuckoo Hashing 中，如果 put 操作产生了驱逐循环（eviction cycle），应该怎么处理？

**目标**：最坏情况 O(1) 的 get 和 remove。

**核心思想**：Two Choices
- 使用**两个**哈希表 T₁, T₂，各自大小为 N
- 使用**两个**哈希函数 h₁, h₂
- key k 只可能存在两个位置：T₁[h₁(k)] 或 T₂[h₂(k)]

**get 和 remove — O(1) worst case**：

```
def get(k):
    if T1[h1(k)] ≠ null and T1[h1(k)].key = k then
        return T1[h1(k)].value
    if T2[h2(k)] ≠ null and T2[h2(k)].key = k then
        return T2[h2(k)].value
    return null

def remove(k):
    temp ← null
    if T1[h1(k)] ≠ null and T1[h1(k)].key = k then
        temp ← T1[h1(k)].value; T1[h1(k)] ← null
    if T2[h2(k)] ≠ null and T2[h2(k)].key = k then
        temp ← T2[h2(k)].value; T2[h2(k)] ← null
    return temp
```

**put — Expected O(1)**：

```
def put(k, v):
    if T1[h1(k)].key = k then T1[h1(k)] ← (k,v); return
    if T2[h2(k)].key = k then T2[h2(k)] ← (k,v); return
    # 驱逐序列
    i ← 1
    repeat
        if Ti[hi(k)] = null then
            Ti[hi(k)] ← (k,v); return
        temp ← Ti[hi(k)]
        Ti[hi(k)] ← (k,v)
        (k,v) ← temp
        i ← 1 if i=2 else 2
    until a cycle occurs
    rehash elements
```

**驱逐序列（Eviction Sequence）**：
- 新 key 插入时，若位置被占，就"踢走"原有 entry
- 被踢走的 entry 去它在另一个表的备用位置
- 如此反复，直到找到空位或检测到**驱逐循环**
- 循环检测：计数器（驱逐次数超过阈值）或标记法

**性能总结**：

| 操作 | 时间复杂度 |
|------|-----------|
| get | **O(1) worst case** |
| remove | **O(1) worst case** |
| put（n 次） | **O(n) expected**（需 N > 2n） |

**实践**：Cuckoo Hashing 比 Linear Probing 慢 20-30%，但因 get/remove 的最坏情况保证和更优雅的删除处理而常被使用。

<span style="color: #9b59b6">💡 **白话理解**：Cuckoo Hashing 得名于杜鹃鸟——杜鹃把蛋下在别人巢里，踢走原来的蛋。每个 key 有两个"候选巢"（两个表），查找时只看两个位置就够了（O(1)）。插入时如果候选位置都有人，就踢走一个，让被踢走的去自己的另一个候选位置——可能引发连锁反应。如果踢来踢去形成循环，就换一组哈希函数重新来（rehash）。</span>

### Set 与 MultiSet `★`

**Set ADT**：无序、无重复的元素集合，核心操作：

| 操作 | 描述 |
|------|------|
| `add(e)` | 添加元素（如不存在） |
| `remove(e)` | 删除元素（如存在） |
| `contains(e)` | 判断是否存在 |
| `iterator()` | 遍历所有元素 |

集合运算：union (S ∪ T), intersection (S ∩ T), subtraction (S − T)，对应 `addAll`, `retainAll`, `removeAll`。

**实现**：用 Map 存 key，忽略 value → 用 HashMap 实现时主要操作期望 O(1)。

**MultiSet（Bag）**：允许重复的 Set。用 Map 实现，key 是元素，value 是出现次数。`count(e)` 返回 e 的出现次数，`remove(e)` 只移除一次。

<span style="color: #9b59b6">💡 **白话理解**：Set 就是"不关心 value 的 Map"——只要知道 key 在不在。MultiSet 更像超市购物袋，同一样东西可以放多个。</span>

## Part 2 — 图的基础

> [!info] 问题链
> 之前的数据结构都是"线性"的（列表、栈、队列）或"层级"的（树）。但现实中的关系更复杂——航线网络、社交网络、网页链接——一个节点可以和任意多个节点相连 → 图 → 怎么在内存中表示图？→ 怎么系统地遍历所有节点和边？→ DFS / BFS。

<span style="color: #e67e22">*哈希表解决了"快速查找"的问题。但很多数据之间有复杂的"关系"——不是简单的 key-value 对，而是网络状的连接。图就是描述这种关系的数据结构。*</span>

---

### 图的定义与术语 `★★★`

> [!tip] 社交网络中，"关注"关系应该用有向图还是无向图建模？"好友"关系呢？

**图** G = (V, E)：
- V = 顶点集合（vertices/nodes）
- E = 边集合（edges），每条边连接两个顶点

**边的类型**：

| 类型 | 定义 | 例子 |
|------|------|------|
| 有向边 (Directed) | 有序对 (u, v)，u 为 tail/origin，v 为 head/destination | 航班、关注关系 |
| 无向边 (Undirected) | 无序对 {u, v} | 双向公路、好友关系 |

**无向图术语**：

| 术语 | 定义 |
|------|------|
| 端点 (Endpoints) | 边 e 连接的两个顶点 |
| 关联 (Incident) | 边 e 关联于其两个端点 |
| 相邻 (Adjacent) | 两个顶点由一条边连接 |
| 度 (Degree) | 顶点上的边数 |
| 平行边 (Parallel edges) | 共享相同端点的多条边 |
| 自环 (Self-loop) | 只有一个端点的边 |
| 简单图 (Simple graph) | 无平行边、无自环 |

**有向图术语**：

| 术语 | 定义 |
|------|------|
| 出度 (Out-degree) | 从该顶点出发的边数 |
| 入度 (In-degree) | 指向该顶点的边数 |
| 反平行边 (Anti-parallel) | (u, v) 和 (v, u) 同时存在；简单有向图允许反平行边 |

**路径与环**：
- **路径** (Path)：一系列顶点，相邻顶点由边连接
- **简单路径** (Simple path)：所有顶点互不相同
- **环** (Cycle)：起点和终点相同的路径
- **简单环** (Simple cycle)：除起止点外所有顶点互不相同
- **无环图** (Acyclic graph)：没有环的图

**连通性**：
- **连通图** (Connected)：任意两个顶点间存在路径
- **连通分量** (Connected component)：最大连通子图

**子图**：
- S = (U, F) 是 G = (V, E) 的子图，若 U ⊆ V 且 F ⊆ E
- **顶点诱导子图** G[U]：U 的顶点及 U 之间的所有边
- **边诱导子图** G[F]：F 的边及其端点

**树与森林（图论定义）**：
- **无根树** (Unrooted tree)：连通且无环的图
- **森林** (Forest)：无环图（其连通分量都是树）
- **事实**：n 个顶点的树恰有 **n-1** 条边
- **生成树** (Spanning tree)：图 G 的连通子图，包含 G 的所有顶点，且是一棵树
- **生成森林** (Spanning forest)：图 G 的生成子图，且是一个森林

<span style="color: #9b59b6">💡 **白话理解**：图就是"点和线"——点是事物，线是关系。有向图像单行道（微博关注），无向图像双行道（微信好友）。路径是"从 A 走到 B 的一条路线"，环是"走了一圈回到原点"。连通分量像"朋友圈"——圈内任意两人可以间接认识，不同圈之间没有联系。</span>

<span style="color: #e67e22">*定义了图的各种概念后，我们需要知道图的一些基本数学性质——边数、度数有什么约束？*</span>

---

### 图的性质 `★★`

| 性质 | 公式 |
|------|------|
| 度数和定理 | Σᵥ deg(v) = **2m**（每条边贡献两个度） |
| 简单无向图的边数上界 | m ≤ **n(n-1)/2** |

记号约定：n = |V|（顶点数），m = |E|（边数），Δ = 最大度。

<span style="color: #9b59b6">💡 **白话理解**：度数和 = 2m 就像"一次握手需要两只手"——每条边有两个端点，所以所有顶点的度加起来等于边数的两倍。简单图的边数上界 n(n-1)/2 就是"完全图"的边数——每对顶点之间都有一条边。</span>

<span style="color: #e67e22">*了解了图的概念，我们需要在计算机里表示它。不同表示方法在空间和操作效率上差异很大。*</span>

---

### 图的表示 `★★★`

> [!tip] 社交网络有 10 亿用户但每人平均只有 200 个好友（稀疏图），你应该用邻接矩阵还是邻接表？为什么？

**三种表示法**：

**1. Edge List**：
- 维护一个顶点序列 + 一个边序列
- 每条边存储两个端点的引用

**2. Adjacency List**：
- 维护一个顶点序列
- 每个顶点额外维护一个**关联边序列** I(v)
- 边对象保存在所连两个端点的关联序列中的引用

**3. Adjacency Matrix**：
- 顶点用 0 ~ n-1 编号
- n × n 的二维数组，A[i][j] 存储边引用（相邻）或 null（不相邻）

**性能对比**（n 顶点，m 条边，无平行边/自环）：

| 操作 | Edge List | Adjacency List | Adjacency Matrix |
|------|-----------|----------------|------------------|
| **空间** | O(n + m) | O(n + m) | **O(n²)** |
| incidentEdges(v) | O(m) | **O(deg(v))** | O(n) |
| getEdge(u, v) | O(m) | O(min(deg(u), deg(v))) | **O(1)** |
| insertVertex(x) | O(1) | O(1) | **O(n²)** |
| insertEdge(u, v, x) | O(1) | O(1) | O(1) |
| removeVertex(v) | O(m) | O(deg(v)) | **O(n²)** |
| removeEdge(e) | O(1) | O(1) | O(1) |

**选择指南**：
- **稀疏图** (m ≈ n)：Adjacency List 最优（空间 O(n + m)，遍历邻居 O(deg(v))）
- **稠密图** (m ≈ n²)：Adjacency Matrix 适用（getEdge O(1) 优势明显）
- Edge List 通常不如另两种，但实现最简单

<span style="color: #9b59b6">💡 **白话理解**：Edge List 像一本通讯录只按边排列——查某人的朋友要翻遍整本。Adjacency List 像每人自带一个朋友列表——找朋友只看自己的列表。Adjacency Matrix 像一张巨大的关系表格——任意两人之间有没有关系一格就能查到，但 n 很大时表格太大了。</span>

<span style="color: #e67e22">*有了图的表示，下一步是遍历：如何系统地访问图中的所有顶点和边？*</span>

### 深度优先搜索 DFS `★★★`

> [!tip] 在迷宫中找出口，你的策略是"每到一个岔路就选一条走到底，走不通了再回来换路"——这是 DFS 还是 BFS？

**策略**：尽可能沿未访问的边深入，遇到死胡同则**回溯**（backtrack）。

**伪代码**：

```
def DFS(G):
    for u in G.vertices() do
        visited[u] ← False
        parent[u] ← None
    for u in G.vertices() do
        if not visited[u] then
            DFS_visit(u)
    return parent

def DFS_visit(u):
    visited[u] ← True
    for v in G.incident(u) do
        if not visited[v] then
            parent[v] ← u
            DFS_visit(v)
```

**边的分类**：
- **DFS edge（树边）**：用于发现新顶点的边
- **Back edge（回边）**：连向已访问祖先的边

**性能**：
- DFS 主函数初始化：O(n)
- DFS_visit 对每个顶点 u 花费 O(deg(u))（不含递归）
- 总时间：**O(n + m)**（假设 Adjacency List 表示）

**关键性质**：
- DFS_visit(v) 访问 v 所在连通分量 Cᵥ 的所有顶点
- {(u, parent[u]) : u ∈ Cᵥ} 构成 Cᵥ 的一棵**生成树**
- {(u, parent[u]) : u ∈ V} 构成 G 的一个**生成森林**

**应用（均可在 O(n+m) 内完成）**：
- 查找两点间路径
- 检测环
- 测试连通性
- 计算连通分量
- 计算生成树
- 进阶：测试双连通性、寻找割边、割点

<span style="color: #9b59b6">💡 **白话理解**：DFS 就像走迷宫的"一条路走到黑"策略——每次都优先往深处走，走不通了再原路返回换一条。它用递归（或显式栈）记住"来时的路"。回边（back edge）就是"发现这条路通向之前走过的地方"。</span>

<span style="color: #e67e22">*DFS 是"一条路走到底"。如果我们想按"离起点的距离"逐层探索呢？*</span>

---

### 广度优先搜索 BFS `★★★`

> [!tip] 要找从你家到目的地的"最少换乘次数"路线，你应该用 DFS 还是 BFS？为什么？

**策略**：从起点 s 出发，先访问距离 s 为 k 的所有顶点，再访问距离为 k+1 的：
- L₀ = {s}
- L₁ = 距 s 一跳的顶点
- L₂ = 距 s 两跳但不在 L₀, L₁ 中的顶点
- …

**伪代码**：

```
def BFS(G, s):
    for u in G.vertices() do
        seen[u] ← False
        parent[u] ← None
    seen[s] ← True
    layers ← []
    current ← [s]
    next ← []
    while not current.is_empty() do
        layers.append(current)
        for u in current do
            for v in G.incident(u) do
                if not seen[v] then
                    next.append(v)
                    seen[v] ← True
                    parent[v] ← u
        current ← next
        next ← []
    return layers, parent
```

**边的分类**：
- **BFS edge（树边）**：用于发现新顶点的边
- **Cross edge（交叉边）**：连接同层或相邻层已访问顶点的边

**性能**：
- 初始化：O(n)
- 主循环：O(Σ deg(u)) = O(m)
- **总时间：O(n + m)**（Adjacency List）
- **Adjacency Matrix 表示下：O(n²)**

**关键性质**：
- BFS(G, s) 访问 s 所在连通分量 Cₛ 的所有顶点
- {(u, parent[u]) : u ∈ Cₛ} 构成 Cₛ 的一棵生成树 Tₛ
- 对每个 v ∈ Lᵢ，Tₛ 中从 s 到 v 的路径恰有 i 条边
- **对每个 v ∈ Lᵢ，G 中从 s 到 v 的任意路径至少有 i 条边**（→ BFS 找到的是最短路径，按边数计）

**应用（均可在 O(n+m) 内完成）**：
- 找**最短路径**（无权图中，按边数计）
- 检测环
- 测试连通性
- 计算生成树
- 测试二部图（bipartite）

<span style="color: #9b59b6">💡 **白话理解**：BFS 像石头丢进水里产生的波纹——先到近处、再到远处，一圈圈向外扩。BFS 天然能找到"最少步数"的路径，因为它总是先探索完所有 k 步能到的点，才去看 k+1 步的。</span>

---

### DFS vs BFS 对比 `★★`

| 维度 | DFS | BFS |
|------|-----|-----|
| 策略 | 深入优先，回溯 | 逐层扩展 |
| 数据结构 | 栈（递归调用栈） | 队列（current/next 层） |
| 非树边类型 | Back edge（指向祖先） | Cross edge（同层或相邻层） |
| 时间复杂度 | O(n + m) | O(n + m) |
| 生成森林/树 | ✅ | ✅ |
| 连通分量 | ✅ | ✅ |
| 检测环 | ✅ | ✅ |
| 最短路径（无权） | ❌ | ✅ |
| 割边 | ✅ | ❌ |
| 二部图测试 | ❌ | ✅ |

<span style="color: #9b59b6">💡 **白话理解**：DFS 和 BFS 的时间复杂度一样，都是 O(n+m)。区别在于"探索顺序"：DFS 像探险家一条路走到底，适合找连通性和割边；BFS 像广播一圈圈扩散，天然能找最短路径和判断二部图。考试常考：选 BFS 还是 DFS 要看你需要什么性质。</span>

### Cut Edges 割边 `★★`

> [!tip] 一个网络中，如果删掉某条连接导致网络断开，这条连接的重要性有多高？

**定义**：在连通图 G = (V, E) 中，边 (u, v) 是**割边**（cut edge / bridge），当且仅当删除 (u, v) 后图不再连通。

**算法**：

| 方法 | 时间复杂度 |
|------|-----------|
| 暴力：对每条边，删除后用 DFS 检查连通性 | O(m²) |
| 改进：只测试 DFS 树中的边 | O(nm) |
| 最优：基于 DFS 的 level + down_and_up | **O(n + m)** |

**O(n+m) 算法关键**：
1. 计算 DFS 树
2. 对每个 u 计算 level[u]（DFS 树中的深度）
3. 对每个 v 计算 down_and_up[v]：沿 DFS 树边向下再走一条回边能到达的**最高层级**（最小 level 值）
4. **DFS 树边 (u, v)（u = parent[v]）是割边 ⟺ down_and_up[v] > level[u]**

<span style="color: #9b59b6">💡 **白话理解**：down_and_up[v] 表示"从 v 的子树出发，不经过边 (parent[v], v)，能到达的最高祖先"。如果 v 的子树通过回边能"够得着" v 的父亲或更高层，说明即使删掉 (parent[v], v) 还有其他路径连通——不是割边。如果够不着，删了就断了——是割边。</span>

## Part 3 — 图算法

> [!info] 问题链
> BFS 能找无权图的最短路径 → 但现实中的边有"权重"（距离、费用）→ 加权图上的最短路径需要 Dijkstra 算法 → 另一个经典问题：用最小总权重连接所有顶点 → 最小生成树（Prim / Kruskal）。

<span style="color: #e67e22">*BFS 找到的是"最少跳数"的路径。但如果每条边有不同的距离/费用，"跳数最少"不一定"总距离最短"。*</span>

---

### 加权图与最短路径 `★★`

**加权图**：每条边关联一个数值（权重），表示距离、费用等。

**最短路径问题**：给定加权图 G 和两个顶点 u, v，找到 u 到 v 的**总权重最小**的路径。

**最短路径性质**：
- **子路径最优性**：最短路径的任何子路径本身也是最短路径
- **最短路径树**：从起点 s 到所有其他顶点的最短路径构成一棵树

---

### Dijkstra 算法 `★★★`

> [!tip] 如果图中有负权边，Dijkstra 还能正确工作吗？为什么？

**输入**：连通无向图 G = (V, E)，非负权重 w: E → R⁺，起点 s

**输出**：从 s 到所有 v ∈ V 的最短距离 D[v]，以及最短路径树 parent[]

**高层思路**：
- 维护距离估计 D[v] ≥ distw(s, v)
- 维护已确定最短距离的集合 S
- 每次从 V \ S 中选 D 值最小的顶点 u 加入 S
- 对 u 的邻居进行**边松弛**（edge relaxation）

**Edge Relaxation**：
$$D[z] \leftarrow \min\{D[z],\ D[u] + w(u, z)\}$$

**伪代码**：

```
def Dijkstra(G, w, s):
    for v in V do
        D[v] ← ∞
        parent[v] ← ∅
    D[s] ← 0
    Q ← new priority queue for { (v, D[v]) : v in V }
    while Q is not empty do
        u ← Q.remove_min()
        for z in G.neighbors(u) do
            if D[u] + w[u,z] < D[z] then
                D[z] ← D[u] + w[u,z]
                Q.update_priority(z, D[z])
                parent[z] ← u
    return D, parent
```

**Priority Queue 操作次数**：
- insert: n 次
- remove_min: n 次
- decrease_key: 最多 m 次

**复杂度分析**：

| PQ 实现 | 总时间 |
|---------|--------|
| Binary Heap | **O(m log n)** |
| Fibonacci Heap | **O(m + n log n)** |

非 PQ 部分的工作量为 O(m)（遍历所有边的邻居）。

**重要限制**：
- ⚠️ **Dijkstra 不能处理负权边**：因为贪心选择 D 值最小的顶点时，负权边可能使已确定的距离不再最优
- ✅ **Dijkstra 适用于有向图**：证明中只需使用有向路径

<span style="color: #9b59b6">💡 **白话理解**：Dijkstra 像一滴水从起点 s 开始向外"渗透"——总是先渗透到距离最近的干燥点，然后从那里继续渗透。edge relaxation 就是"发现了更短的路就更新距离"。它之所以不能处理负权边，是因为"已经渗透过的地方"可能因为负权边变得不再是最短的——但算法不会回头检查。</span>

<span style="color: #e67e22">*Dijkstra 解决了"从一个点到其他所有点的最短路径"。另一个经典问题是：如何用最小总代价把所有点连起来？*</span>

---

### 最小生成树 MST `★★`

**定义**：给定连通图 G = (V, E) 及边权重 cₑ，MST 是 E 的子集 T，使得 T 是生成树且 Σ_{e∈T} cₑ 最小。

**应用**：网络设计（电话线、电缆、公路）、NP-hard 问题的近似算法（TSP、Steiner tree）等。

**MST 的两个核心性质**（假设所有边权不同）：

| 性质 | 内容 |
|------|------|
| **Cut Property** | 对任意顶点子集 S，横跨 S 和 V\S 的最小权边 e 一定在 MST 中 |
| **Cycle Property** | 对任意环 C，C 中权重最大的边 f 一定不在 MST 中 |

**Cycle-Cut 交集引理**：一个环和一个割集（cutset）的交集边数一定是偶数。

<span style="color: #9b59b6">💡 **白话理解**：Cut Property 说"把图分成两半，连接两半的最便宜的那条路一定要选"。Cycle Property 说"任何一个环里最贵的那条路一定不选——因为去掉它环还是通的"。这两个性质是 Prim 和 Kruskal 正确性的基础。</span>

---

### Prim 算法 `★★★`

> [!tip] Prim 和 Dijkstra 的伪代码非常相似——它们的关键区别在哪里？

**思路**：从任意顶点开始，不断选择"连接已选集合 S 和未选集合 V\S 的最小权边"，将对应新顶点加入 S。

**高层伪代码**：

```
def prim(G, c):
    u ← arbitrary vertex in V
    S ← { u }
    T ← ∅
    while |S| < |V| do
        (u, v) ← min cost edge s.t. u in S and v not in S
        add (u, v) to T
        add v to S
    return T
```

**使用 PQ 的实现**：

```
def prim(G, c):
    for v in V do
        d[v] ← ∞; parent[v] ← ∅
    u ← arbitrary vertex in V
    d[u] ← 0
    Q ← new PQ with items { (v, d[v]) for v in V }
    S ← ∅
    while Q is not empty do
        u ← delete min from Q
        add u to S
        for (u, v) incident to u do
            if v ∉ S and c(u,v) < d[v] then
                parent[v] ← u
                decrease priority d[v] to c(u,v)
    return parent
```

**关键区别（Prim vs Dijkstra）**：
- Dijkstra：d[v] = 从 s 到 v 的**路径总权重**（D[u] + w(u,z)）
- Prim：d[v] = v 到集合 S 的**最近单条边权重**（c(u,v)）

**复杂度**（与 Dijkstra 相同）：

| PQ 实现 | 总时间 |
|---------|--------|
| Binary Heap | **O(m log n)** |
| Fibonacci Heap | **O(m + n log n)** |

<span style="color: #9b59b6">💡 **白话理解**：Prim 像在地图上建铁路网——从一个城市开始，每次都把最近的一个新城市连进来。和 Dijkstra 几乎一样的代码，区别只在"近"的定义：Dijkstra 关心"从起点到这个点的总距离"，Prim 关心"从已建铁路网到这个点的单段距离"。</span>

<span style="color: #e67e22">*Prim 是"从一个点向外生长"。另一种思路是"把所有边从便宜到贵排序，能选就选"——这就是 Kruskal。*</span>

---

### Kruskal 算法 `★★★`

> [!tip] Kruskal 算法中，怎么高效判断"加入一条边是否会形成环"？

**思路**：按边权从小到大排序，依次考虑每条边：
- 如果加入该边**不形成环** → 选入 MST（Cut Property）
- 如果加入该边**形成环** → 丢弃（Cycle Property）

**伪代码**：

```
def Kruskal(G, c):
    sort E in increasing c-value
    answer ← []
    comp ← make_sets(V)
    for (u, v) in E do
        if comp.find(u) ≠ comp.find(v) then
            answer.append((u, v))
            comp.union(u, v)
    return answer
```

**Union-Find 操作次数**：
- make_sets: 1 次，|A| = n
- find: **2m** 次（每条边两次）
- union: **n-1** 次（MST 恰有 n-1 条边）

**复杂度**：
- 排序：O(m log m)
- 简单 Union-Find 实现下主循环：O(n²)
- 总：O(m log m + n²)

<span style="color: #9b59b6">💡 **白话理解**：Kruskal 像"贪便宜"策略——把所有道路按造价排序，从最便宜的开始选。唯一的限制是：如果一条路会形成环（两端已经连通了），就跳过。判断"两端是否已经连通"就是 Union-Find 的工作。</span>

---

### Union-Find ADT `★★`

用于维护一个不断演化的**集合划分**。

**操作**：

| 操作 | 描述 |
|------|------|
| `make_sets(A)` | 为 A 中每个元素创建一个单元素集合 |
| `find(a)` | 返回元素 a 所在集合的标识 |
| `union(a, b)` | 合并 a 和 b 所在的两个集合 |

**简单实现（列表 + 数组）**：

| 操作 | 时间复杂度 |
|------|-----------|
| make_sets(A) | O(n) |
| find(u) | **O(1)** |
| union(u, v) | **O(n)** |

- 用数组记录每个元素所属集合
- find 是简单的数组查找
- union 需要把一个集合的所有元素的标记改为另一个集合的标识

**在 Kruskal 中的使用**：
- 初始时每个顶点是自己的集合
- find(u) ≠ find(v) → u, v 不在同一连通分量 → 加入边不会形成环
- union(u, v) → 合并两个连通分量

<span style="color: #9b59b6">💡 **白话理解**：Union-Find 像班级分组——开始每人一组，`union` 是"把两个组合成一组"，`find` 是"你是哪个组的"。Kruskal 用它来快速判断两个顶点是否已经在同一个连通分量里。</span>

---

### 词汇 — Lexicographic Tiebreaking `★`

当边权有相同值时，Prim 和 Kruskal 的结果可能不唯一。

**处理方式**：对边加微小扰动来打破平局（如给第 i 条边加 1/n²）。

**实践**：按边的索引进行字典序比较来隐式处理平局，不需要实际修改权重。

| 时间 | O(n + m) | O(n + m) |
| 最短路径（无权） | ❌ | ✅ |
| 割边检测 | ✅ | ❌ |

### Dijkstra vs Prim

| 维度 | Dijkstra | Prim |
|------|----------|------|
| 解决问题 | 单源最短路径 | 最小生成树 |
| 贪心选择 | D[v] = 从 s 到 v 的总距离 | d[v] = v 到 S 的最近边权 |
| 更新规则 | D[z] ← D[u] + w(u,z) | d[v] ← c(u,v) |
| 负权边 | ❌ 不支持 | ✅（只比较单条边权） |
| 复杂度（Heap） | O(m log n) | O(m log n) |

### Prim vs Kruskal

| 维度 | Prim | Kruskal |
|------|------|---------|
| 策略 | 从一个顶点"生长" | 全局按边权排序选边 |
| 辅助结构 | Priority Queue | Union-Find + 排序 |
| 基于性质 | Cut Property | Cut Property + Cycle Property |
| 复杂度（Heap） | O(m log n) | O(m log m) + Union-Find |
| 适用偏好 | 稠密图略优 | 稀疏图（边少，排序快） |
