## Part 1 — 贪心算法 Greedy

> [!info] 问题链
> 面对优化问题，如何逐步构建解？→ 每步选当前"最好的"（贪心）→ 但什么是"最好的"？不同策略可能给出不同结果 → 需要证明贪心策略的正确性 → 正确时效率极高，但不正确时需要其他范式。

### 贪心算法模式 `★★★`

> [!tip] 你是一个包工头，有 10 个工作可选，每个耗时 1 小时、报酬不同，你只有 7.5 小时——怎么选才能赚最多？

**定义**：贪心算法逐步构建解，每一步选择当前看来最优的选项，期望最终得到全局最优解。

**通用模板**（来自讲义）：

```python
def generic_greedy(input):
    initialize result
    determine order in which to consider input
    for each element i of the input (in above order) do
        if element i improves result then
            update result with element i
    return result
```

**关键特征**：
- 最简洁、最容易实现的算法之一
- 但正确性证明往往最难
- 即使理论上不最优，贪心也常是好的启发式（heuristic）

**证明贪心正确性的两种方法**：

| 方法 | 思路 |
|------|------|
| **Exchange argument** | 假设存在更优解，证明可以通过"交换"使其变为贪心解且不更差 |
| **Greedy stays ahead** | 证明贪心在每一步都"领先"于任何其他可行解 |

**反驳贪心正确性**——找 Counterexample（反例）：
1. 描述具体输入实例
2. 展示贪心算法的输出
3. 展示正确的最优解
4. 两者不同即证明贪心策略不正确

<span style="color: #9b59b6">💡 **白话理解**：贪心就像去自助餐——每次夹菜都拿当前看起来最好的。有时这策略能让你吃到最满意（如按性价比排序），有时会失败（先拿了一堆大虾发现肚子没空间装龙虾了）。关键在于用数学证明"每步选最好的"确实能导致"整体最好"。</span>

<span style="color: #e67e22">*贪心模式有了，来看第一个经典问题：如何往背包里装东西才能价值最大？*</span>

### 分数背包问题 Fractional Knapsack `★★★`

> [!tip] 你有一个 10ml 容量的背包，面前有 5 种液体商品（可以取部分），价值和体积不同——怎么装才能价值最大？

**问题定义**：
- 给定 n 个物品，物品 i 的价值为 bᵢ，重量为 wᵢ
- 背包容量为 W
- 可以取物品的一部分（分数），xᵢ ∈ [0, wᵢ]
- 目标：最大化 Σ(xᵢ · bᵢ/wᵢ)，约束 Σxᵢ ≤ W

**三种贪心策略**：

| 策略 | 思路 | 是否最优 |
|------|------|----------|
| 最高价值优先 | 选 bᵢ 最大的物品 | ❌ 反例可证 |
| 最小重量优先 | 选 wᵢ 最小的物品 | ❌ 反例可证 |
| 最高性价比优先 | 选 bᵢ/wᵢ 最大的物品 | ✅ 最优 |

**讲义中的示例**（5 个物品，W = 10ml）：

| 物品 | 重量 | 价值 | 性价比 bᵢ/wᵢ |
|------|------|------|--------------|
| 1 | 4ml | $12 | 3 |
| 2 | 8ml | $32 | 4 |
| 3 | 2ml | $40 | 20 |
| 4 | 6ml | $30 | 5 |
| 5 | 1ml | $50 | 50 |

- **最高价值优先**：取 5($50) + 3($40) + 7ml的2($28) = **$118**（非最优）
- **最小重量优先**：取 5($50) + 3($40) + 1($12) + 3ml的4($15) = **$117**（非最优）
- **性价比优先**：取 5($50) + 3($40) + 4($30) + 1ml的2($4) = **$124**（最优 ✅）

**算法伪代码**：

```python
def fractional_knapsack(b, w, W):
    x ← array of size |b| of zeros
    curr ← 0
    for i in descending b[i]/w[i] order do
        x[i] ← min(w[i], W - curr)
        curr ← curr + x[i]
    return x
```

**正确性证明（Exchange Argument 概要）**：
- 假设存在可行解 x 不同于贪心解
- 则存在物品 i, k 使得 xᵢ < wᵢ, xₖ > 0, 且 bᵢ/wᵢ > bₖ/wₖ
- 用 min{wᵢ - xᵢ, xₖ} 的量将 k 替换为 i，总价值更高
- 因此不存在比贪心更优的解

**时间复杂度**：
- 排序：O(n log n)
- 遍历填装：O(n)
- **总计：O(n log n)**

<span style="color: #9b59b6">💡 **白话理解**：想象你在水果摊上买水果，背包有限。按"每公斤单价"从高到低拿，先把最贵的拿满，最后一种可能只拿一部分填满背包。注意：如果不允许取部分（0-1 背包），贪心就不一定最优了——那是一个 NP-hard 问题！</span>

<span style="color: #e67e22">*背包问题解决了"选什么最值"。现在换个场景——你有很多课要排教室，怎么用最少的教室？*</span>

### 任务调度 / 区间划分 Interval Partitioning `★★`

> [!tip] 10 节课有时间重叠，你怎么安排最少的教室让所有课都能上？

**问题定义**：
- 给定 n 节课，课 i 从 sᵢ 开始、fᵢ 结束
- 目标：用最少的教室，使同一教室中没有两节课时间重叠

**关键概念 — 深度（Depth）**：
- 一组开区间的深度 = 任意时刻最多重叠的区间数
- **下界**：教室数 ≥ 深度（同时上课的课必须在不同教室）

**贪心策略**：按开始时间递增排序，依次分配到兼容的教室；若无兼容教室则新开一间。

```python
def interval_partition(S):
    sort intervals in increasing starting time order
    d ← 0
    for i in increasing starting time order do
        if lecture i is compatible with some classroom k then
            schedule lecture i in classroom 1 ≤ k ≤ d
        else
            allocate a new classroom d+1
            schedule lecture i in classroom d+1
            d ← d+1
    return d
```

**正确性证明（Greedy Stays Ahead）**：
- 设贪心分配了 d 间教室
- 第 d 间是因为课 i 与前 d-1 间都不兼容才开的
- 由于按开始时间排序，这些不兼容意味着有 d-1 节课在 sᵢ 时刻还没结束
- 加上课 i 本身，共 d 节课在 sᵢ 时刻重叠 → 深度 ≥ d
- 因此贪心所用教室数 = 深度 = 最优

**实现优化**：用优先队列维护每个教室的最后结束时间 → **O(n log n)**

<span style="color: #9b59b6">💡 **白话理解**：按上课时间排序，每来一节课就看看哪个教室空着（最早结束的那个）。如果都在用，就再开一间。贪心用的教室数恰好等于"最拥挤时刻的重叠课数"。</span>

<span style="color: #e67e22">*调度解决了空间分配问题。接下来看另一个贪心经典——如何用最少的 bit 编码一段文字？*</span>

### Huffman 编码 `★★★`

> [!tip] 你要传输 "abracadabra" 这段文字，5 个不同字符用等长编码需要 3×11 = 33 bits——有没有更短的编码方式？

**背景**：
- 文本压缩目标：将字符串 X 编码为更短的二进制串 Y
- **前缀码（Prefix Code）**：没有任何编码是另一个编码的前缀 → 可以无歧义解码
- **编码树（Encoding Tree）**：二叉树表示前缀码
  - 每个叶节点存一个字符
  - 从根到叶的路径给出编码（左 = 0，右 = 1）

**优化目标**：最小化 Σ f(c) × depth_T(c)
- f(c)：字符 c 的频率
- depth_T(c)：字符 c 在编码树 T 中的深度
- 高频字符用短编码，低频字符用长编码

**Huffman 算法**：

```python
def huffman(C, f):
    Q ← empty priority queue
    for c in C do
        T ← single-node binary tree storing c
        Q.insert(f[c], T)
    while Q.size() > 1 do
        f1, T1 ← Q.remove_min()
        f2, T2 ← Q.remove_min()
        T ← new binary tree with T1/T2 as left/right subtrees
        f ← f1 + f2
        Q.insert(f, T)
    f, T ← Q.remove_min()
    return T
```

**构建示例**（X = "abracadabra"）：

频率：a=5, b=2, r=2, c=1, d=1

| 步骤 | 优先队列状态 | 操作 |
|------|-------------|------|
| 初始 | c(1), d(1), b(2), r(2), a(5) | — |
| 1 | b(2), r(2), cd(2), a(5) | 合并 c+d → cd(2) |
| 2 | cd(2), a(5), br(4) | 合并 b+r → br(4) |
| 3 | a(5), cdbr(6) | 合并 cd+br → cdbr(6) |
| 4 | root(11) | 合并 a+cdbr → root(11) |

最终编码示例：a = 0，其余字符在更深层 → 总位数 24 bits（比等长 33 bits 省 27%）

**正确性证明（归纳法）**：
1. **基础**：|C| = 1 时，编码平凡最优
2. **归纳假设**：对 contracted instance (C', f')（合并两个最低频字符后），Huffman 构建最优树 T'
3. **关键观察**：
   - 最优编码树中，最深的兄弟叶节点可以是频率最低的两个字符（交换论证）
   - 合并这两个字符得到 contracted instance (C', f')
   - 展开 T' 回 T：Σf(c)·depth_T(c) = Σf'(c)·depth_T'(c) + f(i) + f(k)
   - 由归纳假设 T' 最优 → T 也最优

**时间复杂度**：
- 扫描计算频率：O(n)（n = |X|）
- 优先队列操作（用 heap）：|C|-1 次合并，每次 O(log |C|)
- **总计：O(n + |C| log |C|)**

<span style="color: #9b59b6">💡 **白话理解**：Huffman 编码就像给城市建公交线路——常去的地方（高频字符）安排直达线（短编码），偶尔去的地方（低频字符）可以多转几站（长编码）。建树时从频率最低的开始"两两合并"，像淘汰赛反过来——最弱的先配对。</span>

<span style="color: #e67e22">*贪心很优雅，但并非万能——小的问题变化就可能让贪心失效。*</span>

---

### 贪心算法的局限性 `★`

贪心启发式容易设计，但不总是最优。且问题的微小变化可能使贪心失效：
- **0-1 背包问题**（不能取部分）→ 贪心不最优（NP-hard）
- **有特殊需求的任务调度** → 贪心不最优
- **非二进制编码** → Huffman 不适用

<span style="color: #9b59b6">💡 **白话理解**：贪心像走迷宫时"永远选看起来最近的路"——有时恰好找到出口，有时走进死胡同。关键是用数学证明来确认贪心对当前问题是否有效。</span>

## Part 2 — 分治法 Divide and Conquer

> [!info] 问题链
> 贪心每步只看局部 → 如果把问题拆成子问题、分别解决再合并呢？→ 分治法（递归分解）→ 子问题的时间怎么分析？→ 递推关系（recurrence）→ 不同的分解方式带来不同效率 → 减少递归调用次数可以显著降低复杂度。

### 分治法模式 `★★★`

> [!tip] 你有 12 枚硬币，其中一枚是假币（重量不同），只能用天平称 3 次——怎么找出来并判断轻重？

**三步框架**：

1. **Divide（分）**：如果是基础情况，直接求解；否则将问题拆分为若干子问题
2. **Recur（递归）**：递归地解决每个子问题（"递归小精灵"处理，类似归纳假设）
3. **Conquer（合）**：将子问题的解合并为原问题的解

**典型基础情况**：子问题规模为常数（通常 0 或 1 个元素），直接计算

**时间复杂度分析 — 建立递推关系**：

设 T(n) 为运行时间，需要确定：
- Divide 步骤的代价（关于 n）
- Recur 步骤：用 T(更小的值) 表示
- Conquer 步骤的代价（关于 n）

```
T(n) = { "Recur 代价" + "Divide + Conquer 代价"    for n > 1
       { O(1)                                      for n = 1
```

<span style="color: #9b59b6">💡 **白话理解**：分治法像打一场锦标赛——把所有选手分成小组，每组先内部比赛（递归），然后冠军之间再比（合并）。关键在于"怎么分"和"怎么合"——分得好、合得快，整体就高效。</span>

<span style="color: #e67e22">*分治模式的第一个应用——在排好序的数组中快速找到目标元素。*</span>

### 二分查找 Binary Search `★★`

> [!tip] 一个排好序的数组有 100 万个元素，你怎么最快找到目标值？

**思路**：每次与中间元素比较，排除一半。

```python
def binary_search(A, left, right, x):
    if left = right then
        return "unsuccessful"
    mid = floor((left + right) / 2)
    if A[mid] < x then
        return binary_search(A, mid + 1, right, x)
    else if A[mid] > x then
        return binary_search(A, left, mid, x)
    else
        return mid
```

**复杂度分析**：
- Divide：找中间并比较 → O(1)
- Recur：左半或右半 → T(n/2)
- Conquer：返回结果 → O(1)

递推：T(n) = T(n/2) + O(1)

**Unrolling 证明**：每层 1 个子问题，每层代价 c，共 log₂n 层 → c · log₂n = **O(log n)**

<span style="color: #9b59b6">💡 **白话理解**：二分查找像查字典——先翻到中间看看目标在前半还是后半，每次排除一半。100 万个元素最多只需约 20 次比较（2²⁰ ≈ 100 万）。</span>

<span style="color: #e67e22">*二分查找只有 1 个子问题。如果拆成 2 个子问题再合并呢？这就是归并排序。*</span>

### 归并排序 Merge Sort `★★★`

> [!tip] 给你一叠乱序的试卷按学号排序，你和一个搭档怎么分工最高效？

**三步**：
1. **Divide**：将数组分成两半
2. **Recur**：递归排序每一半
3. **Conquer**：将两个已排序的半部分合并为一个有序数组

```python
def merge_sort(S):
    if |S| < 2 then
        return S
    mid ← ⌊|S|/2⌋
    left ← S[:mid]
    right ← S[mid:]
    sorted_left ← merge_sort(left)
    sorted_right ← merge_sort(right)
    return merge(sorted_left, sorted_right)
```

**Merge 算法**：

```python
def merge(L, R):
    result ← array of length (|L| + |R|)
    l, r ← 0, 0
    while l + r < |result| do
        index ← l + r
        if r ≥ |R| or (l < |L| and L[l] < R[r]) then
            result[index] ← L[l]
            l ← l + 1
        else
            result[index] ← R[r]
            r ← r + 1
    return result
```

- 两个指针分别指向两个已排序数组的当前最小元素
- 每次取较小的放入结果数组
- 需要 O(n) 次比较和 O(n) 额外空间

**复杂度分析**：
- Divide：找中间并拆分 → O(n)
- Recur：两个子问题 → 2T(n/2)
- Conquer：合并 → O(n)

递推：T(n) = 2T(n/2) + O(n)

**Unrolling 证明**：

| 层级 | 子问题数 | 每个子问题大小 | 本层总代价 |
|------|----------|---------------|-----------|
| 0 | 1 | n | cn |
| 1 | 2 | n/2 | cn |
| 2 | 4 | n/4 | cn |
| k | 2ᵏ | n/2ᵏ | cn |
| log₂n | n | 1 | cn |

每层代价 cn，共 log₂n 层 → 总计 cn · log₂n = **O(n log n)**

<span style="color: #9b59b6">💡 **白话理解**：归并排序像这样排试卷——你和搭档各拿一半分别排好，然后两人各从自己那叠最上面拿一张比较学号，小的先放，就像两条流水线合并。每一层都要扫一遍全部 n 个元素，共 log n 层，所以是 O(n log n)。</span>

<span style="color: #e67e22">*归并排序的递推怎么解？我们需要一套系统的方法来处理各种递推关系。*</span>

### 递推关系求解 Solving Recurrences `★★★`

> [!tip] 你知道 T(n) = 3T(n/2) + O(n)，怎么算出 T(n) 的增长率？

**Unrolling 方法（展开法）**：
1. 展开前几层，找出规律
2. 推导第 k 层的代价
3. 对所有层求和
4. （可选）用归纳法验证

**常见递推公式与解**（讲义原表）：

| 递推公式 | 解 | 典型算法 |
|----------|------|---------|
| T(n) = 2T(n/2) + O(n) | O(n log n) | Merge Sort |
| T(n) = 2T(n/2) + O(log n) | O(n) | — |
| T(n) = 2T(n/2) + O(1) | O(n) | — |
| T(n) = T(n/2) + O(n) | O(n) | — |
| T(n) = T(n/2) + O(1) | O(log n) | Binary Search |
| T(n) = T(n-1) + O(n) | O(n²) | — |
| T(n) = T(n-1) + O(1) | O(n) | — |

**直觉判断（"递归 vs 合并"的比赛）**：
- 合并代价高 + 子问题少 → 合并主导（如 T(n/2) + O(n) → O(n)）
- 子问题多 + 合并代价低 → 递归主导（如 2T(n/2) + O(1) → O(n)）
- 两者平衡 → 得到 log 因子（如 2T(n/2) + O(n) → O(n log n)）

**Master Theorem（主定理）**：
对 T(n) = aT(n/b) + O(nᵈ) 形式的递推，讲义提到可用主定理求解（详见教材 GT），但 unrolling 方法对本课程已足够。

<span style="color: #9b59b6">💡 **白话理解**：递推求解就像算"公司总工作量"——每层管理者把工作拆给下属（子问题），自己还要做协调（合并代价）。总工作量取决于"下属多不多"和"协调重不重"的比赛。如果每层协调工作量总和不变（如 Merge Sort 每层 cn），层数乘以每层代价就是答案。</span>

<span style="color: #e67e22">*递推分析工具到手了。来看快速排序——同样是分治，但分法不同。*</span>

### 快速排序 Quick Sort `★★`

> [!tip] 归并排序"分得简单但合得费力"；有没有一种排序"分得费力但合得简单"？

**三步**：
1. **Divide**：选一个**随机** pivot，将元素分成三组：< pivot, = pivot, > pivot
2. **Recur**：递归排序 < 和 > 两组
3. **Conquer**：拼接三组（免费，O(n) 或更少）

**复杂度分析**：
- Divide：选 pivot + 分区 → O(n)
- Recur：T(nₗ) + T(nᵣ)（取决于 pivot 的位置）
- Conquer：拼接 → O(n)

递推：E[T(n)] = E[T(nₗ) + T(nᵣ)] + O(n)

**期望时间复杂度：O(n log n)**（详细证明见教材，不可考）

- 最坏情况：pivot 每次都是最大/最小 → O(n²)
- 随机选 pivot 使最坏情况极不可能

<span style="color: #9b59b6">💡 **白话理解**：快速排序像整理扑克牌——随机抽一张做基准，比它小的放左边、比它大的放右边，然后对左右两堆分别重复。和归并排序相比：归并"先递归再合并"（merge 费力），快排"先分区再递归"（partition 费力，拼接免费）。</span>

<span style="color: #e67e22">*快排和归并都是排序。分治还能解决其他类型的问题吗？来看一个几何问题。*</span>

### 最大点集 Maxima-Set (Pareto Frontier) `★★`

> [!tip] 一组学生各有"成绩"和"体育"两个维度得分，哪些学生"不被任何人两项都超越"？

**问题定义**：
- 点 p 是最大点（maximum point），若集合中没有其他点 q 满足 q.x ≥ p.x **且** q.y ≥ p.y
- 目标：找出所有最大点

**朴素解法**：对每个点检查是否被其他点支配 → **O(n²)**

**分治解法**：

1. **预处理**：按 x 递增排序（x 相同按 y 递增），O(n log n)
2. **Divide**：将排序后的数组分成两半
3. **Recur**：递归求左半和右半的 Maxima-Set
4. **Conquer**：
   - 找右半 MS 中 y 最大的点 p
   - 左半 MS 中 y > p.y 的点属于整体 MS（因为它们 x 值都小于右半所有点，只有 y 可能被 p 支配）
   - 右半 MS 的所有点都属于整体 MS
   - 合并代价：O(n)

**关键观察**：
- 右半所有点的 x 坐标 ≥ 左半所有点的 x 坐标（排序保证）
- 右半 MS 中 y 最大的点 p 是唯一可能支配左半 MS 点的"威胁"
- 左半 MS 的点只需和 p 比较 y 坐标

**复杂度**：T(n) = 2T(n/2) + O(n) → **O(n log n)**（含预处理排序）

<span style="color: #9b59b6">💡 **白话理解**：如果没有人成绩和体育都比你高，你就是"全面发展的佼佼者"（最大点）。分治法先左右分组各自找佼佼者，再看左边的佼佼者是否能被右边最强的那个"全面碾压"——只需比 y 坐标就行，因为 x 坐标右边肯定更大。</span>

<span style="color: #e67e22">*Maxima-Set 展示了分治的灵活性。接下来看一个更巧妙的例子——同样的框架，不同的分解方式如何带来不同效率。*</span>

### 大整数乘法 Integer Multiplication (Karatsuba) `★★★`

> [!tip] 两个 1000 位的整数相乘，小学方法要做 100 万次基本运算——有更快的方法吗？

**问题**：给定两个 n 位整数 x 和 y，计算 x·y

**朴素方法**：逐位相乘再求和 → **Θ(n²)**

#### 分治 v1（4 次递归调用）

设 x = x₁·2^(n/2) + x₀，y = y₁·2^(n/2) + y₀

则 x·y = x₁y₁·2ⁿ + (x₁y₀ + x₀y₁)·2^(n/2) + x₀y₀

需要 4 次 n/2 位乘法 + O(n) 加法

```python
def multiply(x, y):
    # base case ...
    let x1, x0 such that x = x1·2^(n/2) + x0
    let y1, y0 such that y = y1·2^(n/2) + y0
    return multiply(x1, y1)·2^n +
           (multiply(x1, y0) + multiply(x0, y1))·2^(n/2) +
           multiply(x0, y0)
```

递推：T(n) = 4T(n/2) + O(n)

**Unrolling**：每层代价分别为 cn, 2cn, 4cn, ...（几何增长）→ **T(n) = O(n²)**，和朴素一样！

#### 分治 v2：Karatsuba（3 次递归调用）

**关键观察**：

(x₁ + x₀)(y₁ + y₀) = x₁y₁ + x₁y₀ + x₀y₁ + x₀y₀

所以 x₁y₀ + x₀y₁ = (x₁ + x₀)(y₁ + y₀) - x₁y₁ - x₀y₀

只需 **3 次** 递归乘法（而非 4 次）：

```python
def multiply(x, y):
    # base case ...
    let x1, x0 such that x = x1·2^(n/2) + x0
    let y1, y0 such that y = y1·2^(n/2) + y0
    first_term ← multiply(x1, y1)
    last_term ← multiply(x0, y0)
    other_term ← multiply(x1 + x0, y1 + y0)
    return first_term·2^n +
           (other_term - first_term - last_term)·2^(n/2) +
           last_term
```

递推：T(n) = 3T(n/2) + O(n)

**Unrolling**：每层代价 (3/2)ᵏ·cn（几何增长但底数 < 2），求和得 **T(n) = O(n^(log₂3)) ≈ O(n^1.585)**

#### 对比总结

| 方法 | 递归调用数 | 递推 | 复杂度 |
|------|-----------|------|--------|
| 朴素 | — | — | Θ(n²) |
| 分治 v1 | 4 | 4T(n/2) + O(n) | O(n²) |
| 分治 v2 (Karatsuba) | 3 | 3T(n/2) + O(n) | O(n^(log₂3)) ≈ O(n^1.585) |

<span style="color: #9b59b6">💡 **白话理解**：Karatsuba 的核心是"少做一次乘法"。原来需要 4 次乘法（x₁y₁, x₁y₀, x₀y₁, x₀y₀），通过一个代数技巧把中间两项合并成一次乘法。虽然多做了几次加减法（O(n)），但少了一个递归分支，积累下来复杂度从 n² 降到了 n^1.585。这告诉我们：**减少递归调用次数是分治优化的关键**。</span>

<span style="color: #e67e22">*Karatsuba 展示了减少递归调用的威力。分治法还能用来在不排序的情况下快速找到第 k 小的元素吗？*</span>

### 选择问题 Selection `★★★`

> [!tip] 给你一个乱序数组，不排序的情况下，怎么 O(n) 找到中位数？

**问题**：给定无序数组 A 和整数 k，找第 k 小的元素

**平凡解**：先排序再取第 k 个 → O(n log n)。能否 O(n)？

#### 分治思路

1. **Divide**：找中位数，按中位数将数组分成 ≤ 和 > 两半
2. **Recur**：如果 k ≤ n/2 → 在较小半查找；否则在较大半查找
3. **Conquer**：返回递归结果

如果能 O(n) 找中位数 → T(n) = T(n/2) + O(n) → O(n)

但中位数本身就是 k = n/2 的选择问题！→ 需要**近似中位数**

#### Median of 3 Medians

- 将 A 分成 ⌊n/3⌋ 组，每组 3 个元素
- 每组 brute force 找中位数
- 递归找这些中位数的中位数 x
- x 的 rank 在 [n/3, 2n/3] 之间

递推：T(n) = T(2n/3) + T(n/3) + O(n) → **O(n log n)**，不比排序好！

#### Median of 5 Medians

- 将 A 分成 ⌊n/5⌋ 组，每组 5 个元素
- 每组 brute force 找中位数
- 递归找这些中位数的中位数 x
- x 的 rank 在 [3n/10, 7n/10] 之间

递推：T(n) = T(7n/10) + T(n/5) + O(n) → **O(n)**

**为什么 5 个一组能成功？** 因为 7/10 + 1/5 = 9/10 < 1，两个递归分支的总大小严格小于 n。而 3 个一组时 2/3 + 1/3 = 1，不严格小于 n。

#### Quick Select（随机化版本）

1. **Divide**：随机选 pivot，分成三组（< pivot, = pivot, > pivot）
2. **Recur**：在正确的组中递归查找
3. **Conquer**：返回结果

**期望时间：O(n)**（详细证明不可考）

<span style="color: #9b59b6">💡 **白话理解**：选择问题的核心是"不需要完全排序也能找第 k 个"。关键在于找一个"差不多在中间"的元素来分割。如果每次都能把问题规模砍掉一个常数比例（至少砍掉 3/10），那几轮下来问题就变得很小了。Median of 5 之所以行，是因为分成 5 组保证了两个递归分支的总大小 < n（9/10 < 1）；而分成 3 组时刚好 = n，不够。</span>

## Part 3 — 随机化算法 Randomization

> [!info] 问题链
> 确定性算法有时受最坏情况困扰（如快排的 O(n²) worst case）→ 如果引入随机性呢？→ 随机化算法的行为不仅取决于输入，还取决于随机选择 → 能避免病态最坏情况、提供概率性能保证、简化实现 → 但需要证明"期望"表现好。

### 随机排列生成 Random Permutations `★★★`

> [!tip] 一个音乐播放器要"随机播放" 100 首歌——怎么保证每种播放顺序都等概率？

**问题**：给定整数 n，生成 {1, …, n} 的一个排列，使得每种排列被生成的概率相等（均匀随机）。

**应用**：
- 随机化输入以改善算法的实际性能（如 QuickSort）
- 不预知 k 的情况下按需采样
- 分配稀缺资源
- 作为更复杂随机算法的构建块

#### 第一次尝试（不正确 ❌）

```python
def permute(A):
    n ← length of array A
    for i in {0, ..., n-1} do
        j ← random number in {0, ..., n-1}
        A[i], A[j] ← A[j], A[i]
    return A
```

**为什么不正确？**

均匀随机要求每种排列被生成的**执行序列数相同**。

- 执行序列数 = n × n × ... × n = **nⁿ**（每步从 n 个位置中选，共 n 步）
- 排列数 = **n!**
- nⁿ 不能被 n! 整除 → 无法将执行序列均匀映射到排列
- 例：n=3 时，nⁿ = 27，n! = 6，27 不是 6 的倍数 → 某些排列比其他排列更可能

#### Fisher-Yates Shuffle（正确 ✅）

```python
def FisherYates(A):
    n ← length of array A
    for i in {0, ..., n-1} do
        j ← random number in {i, ..., n-1}    # 关键：从 i 开始
        A[i], A[j] ← A[j], A[i]
    return A
```

**关键区别**：j 的范围是 **{i, ..., n-1}** 而非 {0, ..., n-1}

**为什么正确？**
- 执行序列数 = n × (n-1) × (n-2) × ... × 1 = **n!**
- 排列数 = **n!**
- 每个执行序列产生不同的排列（可证明一一映射）
- 因此每种排列被生成的概率恰好为 **1/n!**

**示例**（从 ⟨1,2,3,4⟩ 生成 ⟨3,2,4,1⟩）：

| 步骤 | i | j 的范围 | 选择 j | 数组状态 |
|------|---|---------|--------|---------|
| 1 | 0 | {0,1,2,3} | 2 | ⟨3,2,1,4⟩ |
| 2 | 1 | {1,2,3} | 1 | ⟨3,2,1,4⟩ |
| 3 | 2 | {2,3} | 3 | ⟨3,2,4,1⟩ |
| 4 | 3 | {3} | 3 | ⟨3,2,4,1⟩ |

**时间复杂度**：O(n)

<span style="color: #9b59b6">💡 **白话理解**：Fisher-Yates 像这样洗牌——第 1 轮从所有 n 张牌中随机抽一张放到第 1 位，第 2 轮从剩下 n-1 张中随机抽一张放到第 2 位……每轮范围在缩小。这恰好产生 n! 种等概率的序列。错误版本的问题在于每轮都从全部 n 个位置中选，产生 nⁿ 种序列，无法均匀对应 n! 种排列。</span>

<span style="color: #e67e22">*随机排列是基础工具。接下来看一个用随机性构建的数据结构——不需要复杂旋转就能达到 O(log n) 的跳跃表。*</span>

---

### 跳跃表 Skip List `★★★`

> [!tip] 有序链表查找要 O(n)，AVL 树需要复杂的旋转平衡——有没有不需要平衡操作的 O(log n) 查找结构？

**定义**：Skip List 是一种随机化的多层链表结构，用于实现 Map ADT。

**与其他 Map 实现的对比**：
- 有序链表：查找 O(n)，简单但慢
- AVL 树：查找 O(log n)，但需要复杂的旋转平衡
- Skip List：**期望** O(log n)，不需要平衡操作

**结构**：
- S₀（底层）：包含所有元素的有序链表，加上 -∞ 和 +∞ 哨兵
- Sᵢ（第 i 层）：Sᵢ₋₁ 的随机子集，每个元素以概率 1/2 被"提升"到上一层
- 每个节点 p 有四个指针：after(p), before(p), above(p), below(p)

```
S3: -∞ ────────────────────────── +∞
S2: -∞ ──────── 14 ─────────────── +∞
S1: -∞ ── -23 ── 14 ──────────── +∞
S0: -∞ ── -23 ── 6 ── 14 ── 30 ── +∞
```

#### Search 操作

```python
def search(p, k):
    while below(p) ≠ null do
        p ← below(p)
        while key(after(p)) ≤ k do
            p ← after(p)
    return p
```

从最高层左上角开始，先向下一层，再在该层尽可能向右，重复直到底层。返回的是底层中 key ≤ k 的最右节点。

#### Insert 操作

```python
def insert(p, k):
    p ← search(p, k)
    q ← insertAfterAbove(p, null, k)
    while coin flip is heads do
        while above(p) = null do
            p ← before(p)
        p ← above(p)
        q ← insertAfterAbove(p, q, k)
```

1. 先搜索找到底层插入位置
2. 在底层插入节点
3. 抛硬币：正面 → 向上建 tower（继续插入上层副本）；反面 → 停止

#### Remove 操作

```python
def remove(p, k):
    p ← search(p, k)
    if key(p) ≠ k then return null
    repeat
        remove p
        p ← above(p)
    until above(p) = null
```

从底层找到节点，然后向上逐层删除该节点的所有副本。

#### 顶层管理

两种选择：
- **固定层数**：max{10, 3log(n)}，插入时需额外维护
- **可变层数**：插入时硬币正面就一直往上建，概率保证不超过 O(log n) 层

#### 复杂度分析

**期望高度 = O(log n)**：
- 元素出现在第 i 层的概率 = 1/2ⁱ
- 第 i 层至少有一个元素的概率 ≤ n/2ⁱ
- 高度超过 c·log n 的概率 ≤ n/2^(c·log n) → 非常小

**期望搜索时间 = O(log n)**：
- 垂直步骤：O(log n) 层
- 每层水平步骤：节点不在上一层的概率 = 1/2 → 期望每层 2 步水平移动 → 每层 O(1)
- 总计：O(log n)

**期望空间 = O(n)**：
- 第 i 层期望 n/2ⁱ 个节点
- 总节点数 = Σ n/2ⁱ ≤ 2n → O(n)

**Skip List 汇总**：

| 指标 | 期望复杂度 |
|------|-----------|
| 空间 | O(n) |
| 搜索 | O(log n) |
| 插入 | O(log n) |
| 删除 | O(log n) |

插入和删除的分析与搜索类似（搜索是主要步骤，插入/删除本身只需额外 O(层高) 操作）。

<span style="color: #9b59b6">💡 **白话理解**：Skip List 像一栋楼的电梯系统——底层（S₀）每站都停（所有元素），高层是快速通道只停大站。搜索时先坐快速电梯到大致位置，再换慢电梯精确到达。每个元素"被选为大站"的概率是抛硬币决定的——正面往上加一层，反面停止。这种随机结构在期望上和 AVL 树一样快，但完全不需要复杂的旋转操作。</span>

## 核心对比速查

### Greedy vs Divide & Conquer vs Randomization

| 维度 | 贪心 Greedy | 分治 D&C | 随机化 Randomization |
|------|------------|---------|---------------------|
| 核心思想 | 每步选局部最优 | 递归拆分 + 合并 | 引入随机性辅助决策 |
| 决策方式 | 单向不回退 | 递归分而治之 | 概率性选择 |
| 正确性保证 | 需证明（exchange arg / stays ahead） | 归纳法 | 概率保证（期望 / 高概率） |
| 典型复杂度 | O(n log n)（排序后贪心） | O(n log n)（平衡分治） | 期望 O(n) 或 O(log n) |
| 适用条件 | 最优子结构 + 贪心选择性质 | 可自然拆分为独立子问题 | 避免最坏情况 / 简化实现 |
| 经典算法 | Fractional Knapsack, Huffman | Merge Sort, Binary Search | Fisher-Yates, Skip List |
| 主要优势 | 简洁高效 | 通用性强 | 实现简单、实际表现好 |
| 主要劣势 | 不总是正确 | 递归开销、需合并步 | 只有期望保证，无最坏保证 |
