**Q1. Hash Table Operation Tracing (5 Marks)**

Consider a hash table of size N = 7 using **Linear Probing** with the hash function h(k) = k mod 7.

**(a) (3 Marks)** Insert the following keys in order: 12, 44, 13, 88, 23, 94, 11. Show the state of the hash table after all insertions. For each key, state the hash value and any probes needed.

**(b) (1 Mark)** Now perform `remove(44)`. Explain what happens to the cell where 44 was stored and why simply clearing it would be incorrect.

**(c) (1 Mark)** After the removal in (b), trace the `get(94)` operation. List each cell index examined and state the result.

> [!note]- Answer
>
> **(a)** h(k) = k mod 7:
>
> | key | h(k) | Probes | Final index |
> |-----|------|--------|-------------|
> | 12 | 5 | — | 5 |
> | 44 | 2 | — | 2 |
> | 13 | 6 | — | 6 |
> | 88 | 4 | — | 4 |
> | 23 | 2 | 2→3 | 3 |
> | 94 | 3 | 3→4→5→6→0 | 0 |
> | 11 | 4 | 4→5→6→0→1 | 1 |
>
> Final table:
> | 0 | 1 | 2 | 3 | 4 | 5 | 6 |
> |---|---|---|---|---|---|---|
> | 94 | 11 | 44 | 23 | 88 | 12 | 13 |
>
> 解释：每个 key 先计算 h(k)，若该位置被占则向右循环探测找空位。例如 94 的 h(k) = 3，但 3, 4, 5, 6 全被占，循环到 0 才找到空位。
>
> **(b)** 将 44 所在的 cell（index 2）标记为 **DEFUNCT**，而不是直接清空。
>
> 原因：如果直接清空 index 2，那么查找 key 23（h(23) = 2）时，探测到 index 2 发现是空的，就会错误地判断 23 不在表中。实际上 23 在 index 3。DEFUNCT 标记告诉 get 操作"这里曾有元素，继续探测"。
>
> **(c)** get(94): h(94) = 3
> - Index 3: key = 23 ≠ 94 → 继续
> - Index 4: key = 88 ≠ 94 → 继续
> - Index 5: key = 12 ≠ 94 → 继续
> - Index 6: key = 13 ≠ 94 → 继续
> - Index 0: key = 94 ✅ → 返回 94
>
> 注意 index 2 是 DEFUNCT（不是空），所以如果从 index 2 开始也会跳过而非停止。但 get(94) 从 h(94)=3 开始，不经过 index 2。

**Q2. Graph Traversal Tracing (5 Marks)**

Consider the following undirected graph with vertices {A, B, C, D, E, F} and edges:
A-B, A-C, B-D, B-E, C-E, C-F, D-E, E-F

When iterating over neighbors, process them in **alphabetical order**.

**(a) (2 Marks)** Perform a DFS starting from vertex A. List the vertices in the order they are first visited, and classify each edge as either a **DFS tree edge** or a **back edge**.

**(b) (2 Marks)** Perform a BFS starting from vertex A. List the vertices in each layer L₀, L₁, L₂, … and classify each edge as either a **BFS tree edge** or a **cross edge**.

**(c) (1 Mark)** Using your BFS result from (b), what is the shortest path (by number of edges) from A to F? Is this necessarily the shortest path that DFS would find? Explain briefly.

> [!note]- Answer
>
> 邻接表（字母序）：
> - A: B, C
> - B: A, D, E
> - C: A, E, F
> - D: B, E
> - E: B, C, D, F
> - F: C, E
>
> **(a) DFS from A**（字母序处理邻居）:
>
> 访问顺序：**A → B → D → E → C → F**
>
> 过程：
> - 访问 A，邻居 B, C；选 B
> - 访问 B，邻居 A(visited), D, E；选 D
> - 访问 D，邻居 B(visited), E；选 E
> - 访问 E，邻居 B(visited), C, D(visited), F；选 C
> - 访问 C，邻居 A(visited), E(visited), F；选 F
> - 访问 F，邻居 C(visited), E(visited)；回溯
>
> 边分类：
> - DFS tree edges: A-B, B-D, D-E, E-C, C-F
> - Back edges: A-C (C 已被发现时从回溯路径中遇到), B-E, E-F
>
> 实际上回边是：当 DFS_visit(u) 检查邻居 v 时 v 已 visited，则 (u,v) 是 back edge。
> - A→B 树边, A→C? 在 DFS_visit(A) 中先选 B 递归下去了，C 后来被 E 发现
> - 更正追踪：DFS_visit(A): visited A, 检查 B(未访问→树边), DFS_visit(B): visited B, 检查 A(已访问→back edge A-B 不算因为 A 是 parent)... 
>
> 让我重新严格追踪：
> - DFS_visit(A): mark A visited. 邻居按序 B, C
>   - B 未访问 → parent[B]=A, DFS_visit(B): mark B. 邻居 A, D, E
>     - A 已访问 → back edge (B,A)? 不，A=parent[B]，是树边的反向，不算额外边
>     - D 未访问 → parent[D]=B, DFS_visit(D): mark D. 邻居 B, E
>       - B 已访问 → back edge (D,B)（B 是 D 的祖先）
>       - E 未访问 → parent[E]=D, DFS_visit(E): mark E. 邻居 B, C, D, F
>         - B 已访问 → back edge (E,B)
>         - C 未访问 → parent[C]=E, DFS_visit(C): mark C. 邻居 A, E, F
>           - A 已访问 → back edge (C,A)
>           - E 已访问 → (C,E) 是树边反向（parent）
>           - F 未访问 → parent[F]=C, DFS_visit(F): mark F. 邻居 C, E
>             - C 已访问 → parent
>             - E 已访问 → back edge (F,E)
>             - return
>           - return
>         - D 已访问 → (E,D) parent
>         - F 已访问 → back edge (E,F)? F 是 E 的后代，所以这是 DFS_visit(E) 检查时 F 已被访问的时机——实际上 F 是在 E 的子树中被发现的。当 DFS_visit(E) 回到检查 F 时，F 已经被通过 C 访问了。(E,F) 是从 E 到后代 F 的边，在无向图 DFS 中不存在 forward edge，所以这仍然是 back edge（但方向理解不同）。
>         - return
>       - return
>     - E 已访问 → back edge (B,E)
>     - return
>   - C 已访问 → back edge (A,C)
>   - return
>
> 访问顺序：**A, B, D, E, C, F**
>
> DFS tree edges（parent 关系）：A-B, B-D, D-E, E-C, C-F
> Back edges：D-B（祖先）, E-B（祖先）, C-A（祖先）, F-E（祖先）
>
> **(b) BFS from A**:
>
> - L₀ = {A}
> - 处理 A：邻居 B, C → 都未见过
> - L₁ = {B, C}
> - 处理 B：邻居 A(seen), D(unseen), E(unseen) → 加入 D, E
> - 处理 C：邻居 A(seen), E(seen), F(unseen) → 加入 F
> - L₂ = {D, E, F}
> - 处理 D：邻居 B(seen), E(seen) → 无新顶点
> - 处理 E：邻居 B(seen), C(seen), D(seen), F(seen) → 无新顶点
> - 处理 F：邻居 C(seen), E(seen) → 无新顶点
>
> BFS tree edges（parent 关系）：A-B, A-C, B-D, B-E, C-F
> Cross edges：C-E（L₁ 到 L₂? 不，C∈L₁, E∈L₂，但 E 被 B 先发现了）, D-E（同层 L₂）, E-F（同层 L₂）
>
> **(c)** BFS 最短路径 A→F：A ∈ L₀, F ∈ L₂，所以最短路径长度为 **2 条边**。
> 路径：A → C → F（通过 parent 回溯：parent[F]=C, parent[C]=A）
>
> DFS 不保证找到最短路径。从 DFS 的树来看，A 到 F 的路径是 A→B→D→E→C→F（5条边），远不是最短的。BFS 天然保证按边数计的最短路径，DFS 不保证。

**Q3. Complexity Analysis of Graph Algorithms (5 Marks)**

**(a) (2 Marks)** Dijkstra's algorithm uses a priority queue. State the number of times each PQ operation (insert, remove_min, decrease_key) is called. Then derive the total time complexity when using a binary heap.

**(b) (2 Marks)** A student claims: "BFS and DFS both run in O(n + m) time, so they are equally efficient regardless of the graph representation." Is this claim correct? Explain, considering both adjacency list and adjacency matrix representations.

**(c) (1 Mark)** Kruskal's algorithm uses a simple Union-Find where `find` takes O(1) and `union` takes O(n). Given that the edge sorting takes O(m log m), derive the total time complexity of Kruskal's algorithm with this Union-Find implementation.

> [!note]- Answer
>
> **(a)** PQ 操作次数：
> - insert: **n** 次（初始化时将所有顶点加入 PQ）
> - remove_min: **n** 次（每个顶点被取出一次）
> - decrease_key: 最多 **m** 次（每条边最多触发一次松弛）
>
> Binary Heap 下各操作代价：
> - insert: O(log n)
> - remove_min: O(log n)  
> - decrease_key: O(log n)
>
> 总时间 = n · O(log n) + n · O(log n) + m · O(log n) = O((n + m) log n)
>
> 因为连通图中 m ≥ n-1，所以 n + m = O(m)，简化为 **O(m log n)**。
>
> **(b)** 该说法**不完全正确**。
>
> - Adjacency List 表示下：BFS 和 DFS 都是 O(n + m)——遍历每个顶点一次，每条边被检查常数次。
> - **Adjacency Matrix 表示下**：查找顶点的邻居需要扫描一整行（O(n)），总共 n 个顶点，所以变成 **O(n²)**（而非 O(n+m)）。
>
> 因此，O(n+m) 的复杂度前提是使用邻接表。在稀疏图（m ≪ n²）中，O(n²) 远差于 O(n+m)。图的表示方式直接影响遍历效率。
>
> **(c)** Kruskal 总时间分析：
> - 排序：O(m log m)
> - make_sets：O(n)（1 次）
> - find：O(1) × 2m = O(m)（每条边调用 2 次 find）
> - union：O(n) × (n-1) = **O(n²)**（最多 n-1 次合并）
>
> 总时间 = O(m log m) + O(n²)
>
> 因为 m ≤ n(n-1)/2，所以 log m = O(log n)，排序为 O(m log n)。
> 总复杂度为 **O(m log m + n²)**，主循环部分被 O(n²) 的 union 操作主导。

**Q4. Comparison and Tradeoff (5 Marks)**

**(a) (2 Marks)** Compare **hash tables** (using separate chaining) and **balanced BSTs** (e.g., AVL trees) for implementing a Map ADT. For each of the following scenarios, state which data structure is more appropriate and justify:
- (i) An application that requires frequent range queries (find all keys between k₁ and k₂).
- (ii) A real-time system that cannot tolerate O(n) worst-case lookup time.

**(b) (2 Marks)** Compare **adjacency list** and **adjacency matrix** representations. A social network has 1 billion users (n = 10⁹) where each user has at most 1000 friends (Δ = 1000). Calculate the space usage for each representation and state which is more practical.

**(c) (1 Mark)** In the context of collision handling, explain one advantage and one disadvantage of **Cuckoo Hashing** compared to **Linear Probing**.

> [!note]- Answer
>
> **(a)**
>
> **(i) Range queries**: **Balanced BST** 更适合。
> - BST 中的元素按 key 有序排列，中序遍历可以高效地找出 [k₁, k₂] 范围内的所有 key，时间为 O(log n + output size)。
> - Hash table 中元素无序存储，无法高效支持范围查询——需要遍历所有 n 个 entry 来筛选，O(n)。
>
> **(ii) Real-time system, no O(n) worst-case**: **Balanced BST** 更适合。
> - AVL tree 的 get 操作最坏情况为 O(log n)，有确定性保证。
> - Hash table（separate chaining）的最坏情况为 O(n)（所有 key 碰撞到同一条链）。虽然期望 O(1)，但无法保证最坏情况。对实时系统来说，确定性的 O(log n) 优于概率性的 O(1)。
>
> **(b)** n = 10⁹, Δ = 1000, m ≤ nΔ/2 = 5 × 10¹¹
>
> - **Adjacency Matrix**: O(n²) = O(10¹⁸)。存储 10¹⁸ 个 cell，即使每个 cell 1 bit 也需要 ~125 PB（petabytes）。完全不可行。
> - **Adjacency List**: O(n + m) = O(10⁹ + 5 × 10¹¹) ≈ O(5 × 10¹¹)。每条边约 16 bytes（两个指针/ID），总共约 8 TB。虽然仍然很大，但数量级上远小于矩阵表示。
>
> **结论**：Adjacency List 是唯一可行的选择。对于这种极度稀疏的图（m ≈ 500n ≪ n²），矩阵会浪费几乎所有空间存 null。
>
> **(c)**
> - **优势**：Cuckoo Hashing 的 get 和 remove 操作是 **worst-case O(1)**（只需查看两个位置），而 Linear Probing 最坏为 O(n)。同时，Cuckoo Hashing 的删除操作更简单——直接置 null，不需要 DEFUNCT 标记。
> - **劣势**：Cuckoo Hashing 需要**两个哈希表**（空间翻倍），且要求负载因子 N > 2n（即 λ < 0.5），空间利用率低于 Linear Probing。此外，插入可能触发驱逐循环，需要 rehash，实践中 Cuckoo Hashing 比 Linear Probing 慢 20-30%。

**Q5. Scenario Design: Shortest Path with Dijkstra (5 Marks)**

Consider the following weighted undirected graph representing travel times (in minutes) between locations:

```
        10
   A -------- B
   |         / |
 5 |      3/   | 8
   |     /     |
   C --/------ D
   |  /   4    |
 2 | /         | 6
   E --------- F
        7
```

Edges and weights: A-B(10), A-C(5), B-C(3), B-D(8), C-D(4), C-E(2), E-F(7), D-F(6)

**(a) (3 Marks)** Run Dijkstra's algorithm from source vertex **A**. Show a table with columns: Vertex, D-value after each iteration, and parent. Clearly indicate the order in which vertices are added to the settled set S.

**(b) (1 Mark)** State the shortest path from A to F and its total weight. Explain why Dijkstra's greedy choice (always picking the vertex with smallest D-value) is correct for this graph.

**(c) (1 Mark)** If the edge B-C had weight **−3** instead of 3, would Dijkstra still produce the correct shortest path from A to B? Briefly explain.

> [!note]- Answer
>
> **(a)** 邻接关系：
> - A: B(10), C(5)
> - B: A(10), C(3), D(8)
> - C: A(5), B(3), D(4), E(2)
> - D: B(8), C(4), F(6)
> - E: C(2), F(7)
> - F: D(6), E(7)
>
> Dijkstra from A:
>
> | 迭代 | 选中顶点 | D[A] | D[B] | D[C] | D[D] | D[E] | D[F] |
> |------|---------|------|------|------|------|------|------|
> | Init | — | 0 | ∞ | ∞ | ∞ | ∞ | ∞ |
> | 1 | **A** | 0 | 10 | 5 | ∞ | ∞ | ∞ |
> | 2 | **C** (D=5) | 0 | **8** | 5 | **9** | **7** | ∞ |
> | 3 | **E** (D=7) | 0 | 8 | 5 | 9 | 7 | **14** |
> | 4 | **B** (D=8) | 0 | 8 | 5 | 9 | 7 | 14 |
> | 5 | **D** (D=9) | 0 | 8 | 5 | 9 | 7 | **15→14 不变** |
> | 6 | **F** (D=14) | 0 | 8 | 5 | 9 | 7 | 14 |
>
> 详细过程：
> - 选 A(0)：松弛 A-B → D[B]=10, A-C → D[C]=5
> - 选 C(5)：松弛 C-B → D[B]=min(10, 5+3)=8, C-D → D[D]=9, C-E → D[E]=7
> - 选 E(7)：松弛 E-F → D[F]=14
> - 选 B(8)：松弛 B-D → D[D]=min(9, 8+8)=9（不变）
> - 选 D(9)：松弛 D-F → D[F]=min(14, 9+6)=15 > 14（不变）
> - 选 F(14)：完成
>
> Parent 数组：parent[A]=∅, parent[B]=C, parent[C]=A, parent[D]=C, parent[E]=C, parent[F]=E
>
> 顶点加入 S 的顺序：**A, C, E, B, D, F**
>
> **(b)** A 到 F 的最短路径：**A → C → E → F**，总权重 = 5 + 2 + 7 = **14**。
>
> Dijkstra 的贪心选择正确，因为该图的**所有边权都是非负的**。当我们从 PQ 中取出 D 值最小的顶点 u 时，不可能存在另一条经过 V\S 中的顶点到达 u 的更短路径——因为那条路径必须经过 V\S 中的某个顶点 y，而 D[y] ≥ D[u]（贪心选择），加上非负权边后只会更长。这就是 Dijkstra 正确性的核心论证。
>
> **(c)** **不一定正确**。如果 B-C 权重为 -3：
>
> Dijkstra 会先选 A(D=0)，松弛得 D[B]=10, D[C]=5。然后选 C(D=5)，松弛 C-B 得 D[B]=min(10, 5+(-3))=2。然后选 B(D=2)。在这个特定例子中结果碰巧是对的（A→C→B 权重 5+(-3)=2 确实是最短的）。
>
> 但 Dijkstra 对负权边**不保证正确**。问题在于：当 C 被加入 S 时 D[C]=5 被确定为最终距离。但如果存在从 B 经过负权边回到 C 的路径，可能发现更短的到 C 的距离——而 Dijkstra 不会重新检查已确定的顶点。在更复杂的图中，负权边会导致错误结果。
