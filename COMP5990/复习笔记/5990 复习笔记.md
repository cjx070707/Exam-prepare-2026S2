## 整体答题骨架（情景题通用）

> [!info] 问题链
> 情景题 = 读场景 → 抓主要问题（按框架分类）→ 点出影响与利益相关者 → 给出可执行改进建议 → 用结构化表达让阅卷人快速对齐要点

**建议写作结构（每小问通用）**：

1. **先点名框架/概念**（例如 CIA / IR vs DR / Agile vs Waterfall / Lifecycle）
2. **把场景中的“哪一句”对上框架的哪一项**（明确对应关系）
3. **写 impact（后果）**：对用户 / 员工 / 公司
4. **写 recommendation（建议）**：最好是 2 条、且“为什么能降低风险”写清楚

## Q1 Professionalism in IT Practice（9） `★★`

> [!tip] 这题常见坑：只写“态度好”不够，要把 professionalism 具体到“伦理 + 责任 + 可靠交付 + 保密/沟通”。

**a) 定义 + 为什么重要（IT consultants）**

- **定义**：在 IT 领域以 **ethical + responsible** 的方式应用技术专长，交付 **reliable solutions** 满足客户需求，并维护信任与声誉。
- **为什么重要**：建立客户信任，保护客户与组织声誉，避免因不当行为造成合规/安全/关系损失。

**b) 两个日常 accountability 行为**

- 按时交付、对错误负责（own mistakes）
- 对项目进度进行诚实沟通（honest progress communication）
- 维护客户机密（client confidentiality）

**c) 两个可落地的组织举措**

- 强制 onboarding：统一 professional standards
- peer-feedback / mentoring：在真实项目里持续强化行为标准

## Q2 IT and Organisational Value（8） `★★`

> [!tip] 题眼：IT 价值不只 automation/效率，还包括决策、创新、协作；衡量要能落到量化指标（ROI/KPI）。

**a) 两种“超越效率”的 IT 价值**

- 支撑 **data-driven decision-making / innovation**（如 predictive maintenance, analytics）
- 增强协作（integrated systems 带来的协同）

**b) 两种 IT 与业务战略对齐方法**

- **enterprise-architecture mapping**：把项目与战略目标映射起来
- **joint IT-business planning sessions**：IT 与业务共同规划，避免“为做系统而做系统”

**c) 一个量化指标**

- ROI 或与成本节省/客户满意度绑定的 KPI（productivity gain 等）

## Q3 IT Lifecycle and Project Phases（9） `★★`

**a) Lifecycle phases（目的：确保需求满足、风险受控）**

- planning → analysis → design → development → testing → implementation → operation → retirement

**b) 跳过 formal steps 的两类风险**

- requirements errors 没被发现
- technical debt → cost overruns / failures

**c) 为什么要 regular lifecycle reviews**

- 识别 obsolescence，及时更新
- 大问题发生前就处理，降低维护成本，控制性能与成本

## Q4 Project Management Approaches：Agile vs Waterfall（8） `★★`

**a) 灵活性 + stakeholder involvement 对比**

- **Agile**：短迭代 + continuous feedback，更适应变化
- **Waterfall**：线性阶段推进，变化成本高

**b) Agile 如何降低风险**

- frequent testing + client reviews → early detection，避免 late failures

**c) 转型挑战 + 解决思路**

- 挑战：团队不适应角色/工作方式变化
- 解决：training + clear communication，建立 Agile mindsets

**a) 为什么沟通关键**

## Q6 Power–Interest Matrix + SBAR（8） `★★★`

> [!tip] 这题是“框架应用题”：必须把三类部门放进矩阵，并把 SBAR 的作用写成“结构化、证据化、可决策”。

**a) Power–Interest matrix 分类与管理策略**

- IT：high power, high interest → **manage closely**
- Finance：high power, medium interest → **keep satisfied**
- HR：low power, high interest → **keep informed**

**b) SBAR 如何帮助对高层汇报**

- SBAR（Situation–Background–Assessment–Recommendation）→ concise + evidence-based
- 帮助快速呈现问题与建议，支撑快速决策

**c) 一个协作工具/方式**

- Teams channels 或 Asana boards：共享可见性 + 讨论线程

## Q7 Estimation Accuracy：PERT / Delphi（8） `★★`

**a) estimation 的作用 + 为什么此场景要准确**

- 提供 time/cost/resources 预测 → 形成 schedules/budgets 基础
- 需求在变、数据不全 → 准确估算才能平衡客户期望、资源配置与利润

**b) 两类估算不准原因 + 影响**

- unclear/changing requirements → scope creep + rework → schedule/cost overruns
- optimism bias / lack of historical data → underestimate → missed deadlines + dissatisfaction

**c) 适合不完整需求的估算方法 + 理由**

- **three-point estimation（PERT）** 或 **Delphi method**
- 用 expert judgment + probability（best/worst/most-likely 或集体共识）处理不确定性

## Q8 CIA Triad + Vulnerabilities + Secure Design（Fintech mobile banking）（12.5） `★★★`

### (a) CIA 对应问题 + impact

- **Confidentiality**：PINs stored in plain text → 数据库泄露时凭证直接暴露 → 欺诈/身份盗用风险
- **Integrity**：transaction records altered → 金融记录不可信 → 余额错误/纠纷/法律后果/信任受损
- **Availability**：high traffic crashes → 忙时不可用 → 无法转账/支付/验证 → 体验与声誉受损

### (b) 两个 vulnerability + 攻击方式

- plain text PIN storage：一旦 DB 被入侵/内鬼/凭证盗用，PIN 直接被读取
- XSS：输入未验证/未净化 → 注入恶意脚本 → 窃取 session cookies / account hijacking / phishing

### (c) 不修复就上线的后果（customers / employees / company）

- customers：financial losses、账户被盗、identity theft、信任下降
- employees：调查/修复/投诉处理负担上升，士气下降
- company：reputational damage、客户流失、罚款/诉讼、合规风险

### (d) 两个 security improvements + 为什么有效

- secure password hashing（bcrypt/PBKDF2/Argon2 + unique salts）：即使 DB 泄露也难还原凭证
- input validation + output sanitization（+ CSP）：降低 XSS 注入与执行风险

## Q9 Incident Response + Disaster Recovery（E-commerce ransomware）（12.5） `★★★`

### (a) 一个 IR + 一个 DR，并说明原因

- **Incident response**：foreign IP login + mass download payment records → 识别/调查/contain
- **Disaster recovery**：ransomware encrypt servers → 恢复系统与业务运营

### (b) 两个 failure（incident / disruption）

- delayed containment（不确定是否真实攻击而拖延）→ 攻击窗口变大、影响扩大
- poor coordination/leadership（谁负责沟通与技术恢复不清）→ 延长 downtime、加剧混乱

### (c) containment 与 restoration 的挑战

- compromised account active for hours → attacker dwell time 长
- backups also encrypted/compromised → recovery point 不可靠 → downtime 增加
- customers cannot order + social media complaints → 技术恢复 + 声誉管理双线压力

### (d) 两个 integrated improvements

- formal incident response plan（roles/escalation/comms/containment + exercises）
- secure isolated backups（offline/immutable + regularly tested）

## Q10 CIA Triad + Vulnerabilities + Secure Design（Online learning platform）（12.5） `★★★`

### (a) CIA 对应问题 + 影响

- **Confidentiality**：weak hashing w/o salting → 密码更易被破解 → 账号与隐私泄露
- **Integrity**：uploaded exam files modified → 评估材料被篡改 → 公平性/可信度受损
- **Availability**：final exam periods down → 关键时期不可用 → 学业活动受影响

### (b) 两个 vulnerability + 攻击方式

- weak password hashing：rainbow table / brute force 更快恢复口令
- SQL injection：未净化输入 → 绕过认证/读取或修改 DB/删除数据

### (c) 不修复上线的后果

- students：账号与作业/成绩等数据风险 + 关键时刻无法考试/提交
- teachers：试卷被改导致额外调查与返工
- institution：声誉受损、投诉、法律/合规后果

### (d) 两个 improvements

- strong hashing + salts（bcrypt/PBKDF2/Argon2）
- parameterized queries / prepared statements（防 SQL injection）

## Q11 Incident Response + Disaster Recovery（Healthcare cloud systems）（12.5） `★★★`

### (a) IR activity vs DR activity

- **IR**：unfamiliar admin login + external transfer of prescription records → containment + log analysis
- **DR**：malware encryption makes DB unavailable → restore critical services quickly

### (b) 两个 weaknesses（attack / outage）

- delayed action while verifying alert → attackers gain time, data exposure increases
- lack of coordination（recovery vs comms vs forensics priority conflict）→ 响应变慢、运营受损

### (c) challenges（contain / restore）

- admin privilege → lateral movement risk，难以界定全范围 compromised assets
- corrupted backups → recovery reliability 下降，downtime 增加
- clinics pressure for updates → 运营连续性压力更大

### (d) two coordinated improvements

- automated detection + rapid containment（异常登录/异常数据外传触发隔离）
- secure backups + DR testing（offline/immutable + exercises）
